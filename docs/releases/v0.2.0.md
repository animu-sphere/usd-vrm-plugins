# v0.2.0 — the multi-bundle workspace

**Tagged:** _pending_ · **Published:** _pending_ ·
[Release page](https://github.com/animu-sphere/usd-vrm-plugins/releases/tag/v0.2.0)

The first release to ship the workspace as separate bundles. Functionally this
is [v0.1.0](v0.1.0.md)'s importer with a sharper diagnostic contract;
structurally it is a different product shape, and **artifact-breaking** for
anyone consuming the v0.1.0 asset names.

## Objective

Ship the workspace split — schemas, importer, resolver, and the container
library as separate, independently versioned components — without regressing
authored output, and publish every component a consumer needs to actually open
a `.vrm`.

## What shipped

- **Workspace Phase 1 — `vrmSchema` split**: the six typed `Vrm*API` schemas
  moved into their own `usd-schema` bundle, consumed via `requires.bundles`.
- **Workspace Phase 2 — `vrmContainer` extraction**: GLB 2 validation, immutable
  views, and embedded-resource hashing as a plain shared CMake library consumed
  by both the importer and the resolver.
- **Workspace Phase 3 — resolver split**: `usdVrmPackageResolver` became its own
  bundle.
- **Workspace Phase 4 — `usdVrm` → `usdVrmFileFormat` rename**: `usdVrm` is
  retired as a bundle id and now names the aggregate product only.
- **Sharper importer diagnostics**: malformed inputs that were previously
  sanitized or rejected silently now emit stable codes (`VRM003`, `VRM111`,
  `VRM141`, `VRM151`, `VRM190`), pinned by a nine-fixture negative corpus.
- **Whole-workspace release lane**: all three bundles are built, tested,
  packaged, and published per target.

Authored stage output, the schema contract, the registered-type set, and the
exported C++ surface are unchanged from v0.1.0 — each phase was verified
byte-identical against the Phase 0 baseline.

Granular detail: [delivery history](../reports/delivery-history.md).

## Compatibility

- **Schema contract version:** 1 (unchanged)
- **OpenUSD:** cy2026 / OpenUSD 26.05 (tolerated range `>=25.05,<27.0`; only
  cy2026 verified in CI)
- **Platforms:** Windows x86_64, macOS arm64, Linux x86_64

## Artifacts

Per target: `vrmSchema-0.2.0-<target>.tar.zst`,
`usdVrmFileFormat-0.2.0-<target>.tar.zst`, and
`usdVrmPackageResolver-0.2.0-<target>.tar.zst`, each with split `-debug`
symbols and a `.manifest.json` sidecar — plus a source archive and
`SHA256SUMS`.

> **Breaking:** v0.1.0's `usdVrm-0.1.0-<target>.tar.zst` no longer exists.
> The importer is now `usdVrmFileFormat-0.2.0-<target>.tar.zst`, and the
> schemas and resolver it used to carry are separate artifacts.

> **All three must be installed together.** `usdVrmFileFormat` alone registers
> the `.vrm` file format and then fails to open a stage: the typed `Vrm*API`
> schemas live in `vrmSchema`, and OpenStrata 0.18.0 stages a dependency
> bundle's link-time half (`libvrmSchema` + its CMake package) without its USD
> registration half (`plugInfo.json`, `generatedSchema.usda`). Embedded
> textures additionally need `usdVrmPackageResolver`. Install order and the
> exact `PXR_PLUGINPATH_NAME` entries are in the
> [install guide](../guides/INSTALL.md).

## Known limitations at this release

- **No aggregate product artifact.** `usdVrmPlugins-<version>-<target>.tar.zst`
  ([WORKSPACE.md §5](../architecture/WORKSPACE.md)) does not exist; consumers
  compose three bundles by hand. Blocked on upstream `ost` work — see
  [report 23](../reports/ost/23-2026-07-18-v0.18.0-workspace-packaging-v0.19.0-asks.md).
- **The packaged importer is not independently verifiable.**
  `ost plugin test --from-package` is incompatible with `--workspace`, so the
  release lane gates on the composed clean-install smoke instead.
- Morph-weight animation not authored.
- MToon is a `UsdPreviewSurface` approximation; no renderer-specific
  realization.
- One OpenUSD version cell only (no min-vs-latest axis).
- No runtime evaluation layer (`execVrm` does not exist).
