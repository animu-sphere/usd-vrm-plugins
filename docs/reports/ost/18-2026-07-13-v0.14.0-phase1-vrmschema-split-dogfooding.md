# OST v0.14.0: Phase 1 vrmSchema split — workspace features used in anger, v0.15.0 asks revised

- Date: 2026-07-13
- `ost --version`: `ost 0.14.0`
- Context: the first real consumer run of 0.14.0's workspace toolchain. We
  executed workspace Phase 1 (the `vrmSchema` bundle split out of `usdVrm`,
  repo PR #42 stacked on #41) end to end with `ost plugin new/build/test/
  package`, `requires.bundles`, `schema.contract`, and the generated PR lanes.
  This report records what held, what broke, and how the v0.15.0 ask list
  changes as a result.
- Prior report: `17-2026-07-13-v0.14.0-verification-v0.15.0-asks.md` (same ost
  version; that report *predicted* the split would lean on these features,
  this one records what actually happened).

## TL;DR

**The workspace model held: graph validation, contract pinning, and the
usd-schema-cpp template shape carried a real one-boundary split with zero
authored-stage drift (Phase 0 baseline byte-identical, 53/53).** But the
composition story stops at validation: `requires.bundles` is *checked* and
then *ignored* — neither `ost plugin build` nor `ost plugin test` (even under
`--workspace`) puts a declared dependency into the consumer's build prefix or
session env, and the CI cell schema has no `--with` equivalent. The result: a
correctly-declared consumer bundle is red at L2–L5 everywhere except a
hand-written `--with` invocation, and our PR lanes had to demote the importer
to `up_to: 1`. **Session/build composition from `requires.bundles` is now the
single top ask.** Also new: an L5 golden EOL trap on hosted Windows
(autocrlf × multi-line USDA strings), a template consumer that doesn't
compile on MSVC (NOMINMAX), and a template package config that dies on
pxrConfig's non-idempotence (TBB::tbb re-add).

## What worked (and worked well)

- **`ost plugin test --workspace` graph validation**: after adding
  `requires.bundles: [{id: vrmSchema, version: ">=0.1,<0.2", contract: 1}]`
  to usdVrm and `schema.contract: 1` to vrmSchema, `data.graph` came back
  with 2 nodes / 1 edge / `passed: true`, and the direction rules matched the
  contract table in `docs/architecture/WORKSPACE.md` §2. Declaring the edge
  was one manifest stanza. This is the enforcement layer our plan §11 wanted.
- **Strictness pays off again**: putting composition fields into a manifest
  without the `manifest.schema: openstrata.plugin/v1alpha1` header fails
  fast with a *named* fix (`INVALID_CONFIG: bundle composition fields need
  manifest.schema: ...`, exit 3). Same for the CI cell experiment below —
  the unknown-key error lists the full valid-key set. Both errors told us
  exactly what to do; no time lost.
- **usd-schema-cpp template as the Phase 1 blueprint**: we scaffolded into a
  scratch dir and ported the shape (CMake package export via
  `<name>Config.cmake.in` + targets export, installed-header layout,
  installed-package consumer smoke, plugInfo.json.in configure flow,
  self-contained `OpenStrataPlugin.cmake` helpers) onto our existing
  generated schema code. The template's deliverables *are* the Phase 1
  checklist; we kept our committed-generated-sources flow and everything
  coexisted. `ost plugin test` on the result: L2 `schema.registration` and
  L4 `schema.apply_roundtrip` picked up the six APIs with no extra wiring.
- **`ost plugin package plugins/vrmSchema`**: first try, lean package,
  `vrmSchema-0.1.0-cy2026-windows-x86_64-py313-usd.tar.zst` — exactly the
  §5 artifact naming our contract specifies.
- **`--with` (0.14.0's session composition escape hatch)**:
  `ost plugin test plugins/usdVrm --with plugins/vrmSchema` and
  `ost plugin run ... --with ...` compose the session correctly (12 pass,
  L5 golden included; the Phase 0 baseline gate runs through it now).
- **Generated PR lanes on first push: 5/6 green**, including
  `usdvrm-pr-windows` — which proves the repo-side vrmSchema bootstrap
  (configure/build/install the sibling into a private prefix, then
  `find_package`) works on a hosted runner from a cold checkout.

## What broke

### 1. `requires.bundles` validates, but composes nothing (the big one)

Empirical sequence, all on 0.14.0:

- `ost plugin build plugins/usdVrm --dry-run`: the rendered configure command
  contains no vrmSchema prefix, package dir, or any artifact of the declared
  dependency. A consumer bundle whose CMake does
  `find_package(vrmSchema CONFIG REQUIRED)` cannot build via `ost plugin
  build` without repo-side help.
- `ost plugin test plugins/usdVrm` (single bundle): L2 discovery fails —
  `libUsdVrmFileFormat.dll` imports `libvrmSchema.dll`, which is neither on
  the loader path nor registered, so the plugin doesn't load at all. L2–L5
  all red.
- `ost plugin test --workspace`: **validates the graph first (green), then
  tests each bundle in an isolated session anyway** — usdVrm fails L2–L5
  identically. The workspace mode knows the edge exists and still doesn't
  use it.
- CI cells: we tried `with: [plugins/vrmSchema]` on a cell; strict validation
  (correctly) rejects it — valid keys are `name, lane, runner,
  runtime_artifact, runtime_remote, plugin_artifact, bundle, platform,
  profile, up_to, host_python, publish, host`. So the lanes cannot express
  what the CLI's own `--with` flag can.

Interim repo-side mitigations (both in PR #42):

- Build: usdVrm's CMake bootstraps the sibling bundle **as an installed
  package** into `<build>/vrmSchema-pkg` when `find_package` misses
  (configure→build→install at configure time; never `add_subdirectory`).
  This is exactly the kind of dual-mode glue a generator should own.
- Test lanes: usdVrm PR cells demoted to `up_to: 1`; new `vrmschema-pr-*`
  cells run the full pyramid. The importer's L2–L5 live only in local
  `--with` runs, ctest, and the baseline gate until ost composes sessions.

### 2. L5 golden vs hosted-Windows checkout: CRLF becomes *string content*

`vrmschema-pr-windows` failed L5 (`flattened output differs from the golden`)
while the identical commit passed L5 locally on Windows and on the Linux and
macOS lanes. Root cause, reproduced in a fresh local clone (`git ls-files
--eol`: `i/lf w/crlf`) and pinned with a byte diff:

- Hosted Windows runners check out with autocrlf, so the *fixture* usda gets
  CRLF in the working tree.
- Our schema-only fixture has a **multi-line (triple-quoted) layer comment**.
  Inside a triple-quoted USDA string, the `\r` is part of the parsed string
  value — so `usdcat --flatten` faithfully re-emits `...authors a\r` and the
  output genuinely differs from the LF golden. Single-line strings can't
  embed literal newlines, which is why `minimal.vrm.golden.usda` never hit
  this in months of lanes.
- Whatever EOL/doc-header normalization the L5 comparator already does (the
  committed goldens carry machine-absolute `doc = "Generated from Composed
  Stage of root layer C:\dev\..."` lines and pass on all three platforms, so
  path normalization clearly exists) does not — and arguably should not —
  reach *inside* string values.

Repo fix (verified in the CI-equivalent clone: 11 pass / 0 fail after
re-smudge): `.gitattributes` gains `*.usda text eol=lf`, covering fixtures,
goldens, and the Phase 0 baseline snapshots.

Ask-shaped residue for ost: a one-line note in the golden guidance (or an L5
`suggested_action` when the flatten/golden difference is `\r`-only inside
string literals) would turn a cross-platform head-scratcher into a 30-second
fix. The current message ("review the diff") is under-actionable because the
report JSON doesn't include the diff either.

### 3. usd-schema-cpp template consumer doesn't compile on MSVC

`tests/consumer/main.cpp` + the template's consumer CMakeLists hit
`robin_hash.h(344): C2238` — pxr headers don't survive `windows.h` min/max
macros; the consumer target sets no `NOMINMAX`. The template's *plugin*
target gets NOMINMAX via `openstrata_configure_plugin`, but the consumer
smoke doesn't go through that helper. One-line template fix.

### 4. Template package config vs pxrConfig non-idempotence

`<Name>Config.cmake.in` does an unconditional `find_dependency(pxr CONFIG)`.
When the consumer has already resolved OpenUSD (every bundle in a shared
runtime does), pxrConfig re-runs and dies on `add_library cannot create
imported target "TBB::tbb"`. Our copy now guards it with `if(NOT pxr_FOUND)`
— the same guard the template's own bundle CMakeLists already uses two lines
into its pxr resolution. Suggest the template adopt the guard.

Related trap worth a docs note: when the guarded `find_package(vrmSchema
CONFIG REQUIRED)` fails mid-configure, `vrmSchema_DIR` is already cached, so
the *next* configure finds the stale/broken package silently. We burned one
cycle on a wiped build dir before spotting it.

### 5. Small observations

- After the split, `ost plugin build plugins/usdVrm` still passes
  `-DOPENSTRATA_SCHEMA_SOURCES_FILE=<.strata>/schema-sources.cmake` even
  though usdVrm's manifest no longer has a `schema:` block (stale target
  state; harmless because our CMake no longer includes it, but a fresh reader
  would be confused). Conversely, building the *schema* bundle compiled the
  committed sources — no fragment was generated for the `usd-schema` kind on
  this run.
- `ost plugin test` (single bundle) exits 5 for an L-level failure — same
  exit code as a workspace graph violation. Distinct codes would make CI
  triage greppable.

## v0.15.0 asks (revised priority)

1. **Compose sessions and build prefixes from `requires.bundles`** (new, top).
   The declared edge should (a) put the dependency's plugInfo + lib on the
   consumer's session env in `plugin test`/`plugin run` — `--workspace` mode
   especially has no excuse, it already resolved the graph; (b) surface an
   installed-package prefix (or build the sibling) for `plugin build`; and
   (c) be expressible in CI cells (a `with:`/dependency-closure semantic) so
   generated lanes can run a consumer's full pyramid. Until then every
   workspace consumer bundle is capped at `up_to: 1` in hosted lanes.
2. Validate `requires:` strictly / give libraries a contract (carried;
   `vrmContainer` lands next phase and would be silently ignored today).
3. Golden EOL guidance / `\r`-aware L5 diagnostic + include the diff (or a
   diff artifact path) in the JSON report (new, small).
4. usd-schema-cpp template: NOMINMAX on the consumer smoke; `if(NOT
   pxr_FOUND)` guard in the package config (new, small).
5. Generator-owned reusable bootstrap/runtime-pull fragment (carried).
6. `ost ci pin bootstrap --version` helper (carried).
7. Workspace-aware `--from-package` (carried; becomes load-bearing at
   Phase 5).
8. Stale-PDB non-fatal diagnostic (carried).

  contract 1), `passed: true`, no issues.
- vrmSchema: build / test (11 pass, 0 fail incl. L5 golden) / package all
  standalone-green; installed-package consumer smoke compiles, links, and
  applies `UsdVrmHumanoidAPI` at runtime.
- usdVrm: builds standalone through the sibling bootstrap;
  `--with plugins/vrmSchema` pyramid 12 pass / 0 fail.
- Phase 0 baseline gate: byte-identical 53/53 in both the ost session and
  the plain-CMake ctest run — the split changed no authored-stage, schema,
  discovery-union, or symbol-union behavior.
- PR lanes after the `.gitattributes` fix: pending re-run at time of writing
  (5/6 were green before it; the sixth was the L5 EOL failure above).
