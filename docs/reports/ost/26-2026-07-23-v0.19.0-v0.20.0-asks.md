## Adoption result

The repository now pins `ost 0.19.0` in the CI source matrix and release
workflow. The generated source workflow was regenerated from
`openstrata.ci.yaml`.

`ost plugin test --workspace --from-package --json` passed all three packaged
bundles: 40 checks passed and 5 golden-file checks were skipped. The composed
packaged configuration remains green under 0.19.0.

`ost plugin package --workspace --product --json` produced three member archives
and one aggregate product archive. The release workflow now requests
`--product`, verifies that the product exists, and stages it with the member
bundles. No separate debug package was produced; 0.19.0 reports the explicit
`not-produced` reason for the current Windows build.

## Asks for v0.20.0

1. **Close the packaging-closure P0.** A standalone package containing a bundle
   dependency must stage both its link/runtime half and its USD registration
   half. Acceptance: the bare `usdVrmFileFormat --from-package` run passes L3/L4
   without adding `vrmSchema` as a separate package.
2. **Define activation for staged `requires.runtime_libs`.** Provide a
   consumer-facing, non-`ost` recipe or materialize the required DLL search
   layout so a plain OpenUSD/Python host can load the package.
3. **Make the aggregate product the first-class release artifact.** Document
   its version/name contract and provide a corresponding install/verify command
   that does not require manually unpacking member archives.
4. **Keep path portability and diagnostics contractual.** Preserve `/` in
   emitted manifest paths and retain actionable warnings for `--no-inject`
   plugin-path exclusions and absent debug packages.
