# OST v0.14.0: asks verified (3 delivered / 3 carried), workspace tooling arrives, v0.15.0 asks

- Date: 2026-07-13
- `ost --version`: `ost 0.14.0`
- Context: verify-forward of the six v0.14.0 asks from the prior report, plus a
  first evaluation of 0.14.0's plugin-workspace features against our new
  workspace implementation plan (schema / fileformat / resolver / exec bundle

## TL;DR

**Three of the six asks are delivered and verified (#2 strict matrix keys,
#5 native Windows `GITHUB_PATH`, #6 ABI-matched python under `plugin run`);
the other three (#1 generator-owned lane steps, #3 pin helper, #7 stale-PDB
warn) are explicitly named on ost's own roadmap as future work.** The surprise
payload of 0.14.0 is everything we *didn't* ask for: `ost plugin test
--workspace` dependency-graph validation, a compiled USD schema template with
CMake package export, a `usd-package-resolver` plugin kind, and an
`ost init --template usd-plugin-workspace` root — which lands exactly as we
begin splitting `usdVrm` into a plugin workspace. The one gap that will bite
that plan: `requires.libraries` is silently ignored by workspace validation.

## v0.14.0 asks — scorecard (3 ✅ / 3 carried)

1. **`ost ci generate` repo-specific steps / lanes** ❌ carried. The strict
   cell-key set is now `name, lane, runner, runtime_artifact, runtime_remote,
   plugin_artifact, bundle, platform, profile, up_to, host_python, publish,
   host` — no step-level hooks. The two new keys are trust/host stubs, not
   lane extensibility: `publish: never | candidate` is parsed and semantically
   checked (`publish: candidate` on a `pull_request` cell fails with "the
   pull_request lane must never publish (fork-PR safety)") but not rendered
   into the workflow yet; `host: {os, labels}` likewise parses strictly with
   no rendered effect. Release notes state it directly: "Generated trusted
   release lanes remain future work." Our `release.yml` keeps its duplicated
   pins + "mirror openstrata.ci.yaml" comment for another cycle.
2. **`ost ci validate` rejects unknown cell keys** ✅ — our exact 0.13.0 repro
   now fails correctly: `from_package: true` on a cell →
   `error[MANIFEST_INVALID]: cells[0]: unknown field 'from_package', expected
   one of ...` (exit 3), with the full valid-key list in the message. Nested
   objects are strict too (`host.wat` → unknown field, enum values
   enumerated). Release notes: "strict at every authored object."
   (Verification honesty note: our first probe false-passed because the probe
   script's regex didn't match CRLF line endings and validated an *unmodified*
   matrix — redone with the key actually inserted, rejection confirmed.)
3. **Bootstrap pin version-bump helper** ❌ carried — release notes name the
   exact shape (`ost ci pin bootstrap --version ...`) as a non-blocking
   follow-up. Fresh datapoint this session: bumping `bootstrap.ost.version`
   0.13.0 → 0.14.0 again meant hand-editing the manifest, regenerating, and
   re-mirroring the hand-authored release lane.
4. **(carried) corpus CTest smoke in generated lanes** ❌ — subsumed by #1,
   which did not land.
5. **Windows `GITHUB_PATH` native path** ✅ — the generated bootstrap now
   hard-requires `cygpath` on Windows (`::error` + exit 1 if missing), exports
   `cygpath -w` conversions of both the binary path and its directory, writes
   the native directory to `GITHUB_PATH`, records
   `{executable, exported_path}` in `.ost-ci/bootstrap.json`, and (per release
   notes) verifies hosted-Windows availability via native `ost.exe --version`.
   Exactly the ask; python-spawned `ost` on Windows runners is now safe.
6. **`plugin run -- python` ABI resolution** ✅ — verified with a
   falsification test: with Python 3.14.6 first on `PATH`
   (`python --version` → 3.14.6), `ost plugin run plugins/usdVrm -- python -c
   "import sys; print(sys.version)"` returns **3.13.14** (the runtime ABI
   match) and `import pxr` succeeds. Release notes confirm the mechanism:
   bare `python` / `python3` resolve through the runtime-aware Python
   resolver, selecting an ABI-matched concrete interpreter or failing before
   launch with an actionable precondition; explicit paths stay explicit. The
   PR #36 `setup-python` 3.13 pin on the Windows release cell is now
   redundant (kept until the next release-lane touch confirms it in CI).
7. **(candidate) stale-PDB warning in `plugin package`** ➖ roadmapped — ost's
   `docs/roadmap/current.md` carries "Packaging diagnostic — stale debug
   sidecar candidate": warn when a same-basename PDB is older than its DLL,
   non-fatal until PE/PDB identity can be compared reliably
   (`DEBUG_SYMBOL_STALE_CANDIDATE` in the release-lane design doc). Not
   implemented in 0.14.0.

## What 0.14.0 shipped that we didn't ask for — and are about to lean on

Our workspace implementation plan (bundle split: `vrmSchema` /
`usdVrmFileFormat` / `usdVrmPackageResolver` / `execVrm`, shared plain-CMake
`vrmContainer`) turns out to be co-designed with this release:

- **`ost plugin test --workspace`** discovers `plugins/*` bundle manifests,
  validates the dependency graph *before* running per-bundle verification,
  and emits `data.graph {nodes, edges, issues}` in `--json`. The enforced
  direction rules are literally our plan's §2.2 table: a schema bundle
  depends on nothing, a resolver cannot depend on a file format, a file
  format may consume schema + resolver, no cycles. Stable `WORKSPACE_*`
  issue codes (dependency missing / version mismatch / contract mismatch /
  direction forbidden / cycle), exit 5 on graph failure. Ran green on our
  current single-bundle repo (1 node `usdVrm`, 0 edges, L0 8 pass / 3 skip).
- **Manifest composition fields**: `requires.bundles: [{id, version-range,
  contract}]` on consumers; `schema.contract` on providers, explicitly
  decoupled from `plugin.version` — the exact schema-contract pinning our
  plan §4.1 specifies. Dependency entries reject unknown keys.
- **`usd-schema-cpp` template** (`ost plugin new usd-schema vrmSchema
  --template usd-schema-cpp`): compiled-schema skeleton with generated
  contract API + tokens, `plugInfo.json.in`, a copied CMake helper, a CMake
  **package export** (`VrmSchemaConfig.cmake.in`) and an installed-package
  **consumer smoke test** — i.e. the Phase 1 `find_package(vrmSchema)`
  deliverables pre-built. The 0.14.0 fix history even used `vrm-schema` as
  its casing test case.
- **`usd-package-resolver` plugin kind**: `ArPackageResolver` scaffold with
  extension dispatch, a traversal-rejecting entry-path guard, and an L2
  plug-registry registration level — the Phase 3 resolver-bundle shape.
- **`ost init --template usd-plugin-workspace`**: dual-mode workspace root
  for `ost plugin new` bundles (candidate shape for our root once the split
  starts; needs evaluation against our existing root CMake).
- **Artifact trust policy foundation** (`openstrata-artifact-policy.toml`,
  `ARTIFACT_POLICY_*`, publisher-gated `ost artifact push`, `verify
  --policy`): not needed by us today, but the v0.15.0 milestone (SBOM +
  provenance attached to artifacts) is what our §7.7-style release checksums
  would graduate into once we publish bundle artifacts under a protected
  `animu-sphere` namespace.

**The gap for our plan:** `requires.libraries` is *not* validated — ost's own
reference doc concedes unknown keys under `requires:` are currently ignored
rather than rejected. So `vrmContainer` (a plain shared library, deliberately
not a plugin bundle) is invisible to workspace validation, and a typo'd
`requires:` sub-key silently vanishes — the same failure mode ask #2 just
fixed for the CI matrix, one manifest over.

## v0.15.0 asks (priority order)

1. **(carried) Generator-owned reusable bootstrap/runtime-pull fragment.**
   Unchanged ask, sharpened: we don't need the full trusted-release-lane
   feature to land first — a generator-emitted composite action (bootstrap +
   runtime pull, pins resolved from `openstrata.ci.yaml`) that hand-authored
   workflows can `uses:` would delete the duplicated pins in `release.yml`
   today, and the future trusted lane can consume the same fragment.
2. **Validate `requires:` strictly, and give libraries a contract.** Either
   land the portable `requires.libraries` contract or, at minimum, reject
   unknown keys under `requires:` the way every other authored object now
   does. Our workspace split ships a shared non-bundle library
   (`vrmContainer`) consumed by two bundles in 0.15.0-era work; today its
   declaration would be silently dropped.
3. **(carried) `ost ci pin bootstrap --version <V>`** — ost already named the
   command; we hand-bumped the pin again this cycle (0.13.0 → 0.14.0).
4. **Workspace-aware packaged verification.** `--from-package` is
   `incompatible with --workspace`. Our plan's Phase 5 gate is per-bundle
   packaged verification plus an aggregate clean-install — a
   `--workspace --from-package` loop (and eventually an aggregate-artifact
   story) keeps that gate native instead of hand-scripted per bundle.
5. **(carried, roadmapped) stale-PDB non-fatal diagnostic** — keep the nudge;
   it silently shipped a stale symbol file once.

Not asked: `publish:` rendering into generated workflows — that's the trust
lane ost's roadmap already owns; we'll consume it when it exists.

## Net

0.14.0 delivered the two correctness asks that were biting CI (#5, #6) and the
strictness ask (#2), and deferred exactly the ergonomics asks it said it would
(#1, #3, #7 — all named on ost's roadmap). Meanwhile it quietly shipped the
workspace toolchain our bundle-split plan assumed we'd have to hand-roll:
graph-validated bundle dependencies, schema contracts as first-class manifest
data, and templates for both new bundle kinds. The split can start with ost
enforcing the architecture from day one; the only contract ost can't yet see
is the plain-library one (`vrmContainer`), which is ask #2 of this cycle.
