## Dogfood Findings

1. The existing `cy2026/usd` runtime was a mock runtime. It had to be replaced with a real `build` runtime before L2+ plugin checks could execute.
2. `plugins/usdVrm/openstrata.plugin.yaml` declared `cxx_abi: msvc143`, which fails on macOS where the runtime reports `libcxx`.
3. `plugins/usdVrm/plugin/resources/usdVrm/plugInfo.json` had a Windows `.dll` `LibraryPath`; macOS requires `.dylib`.
4. Standalone plugin CMake did not define `USDVRM_BUILD_TESTS`, so `ost plugin build` configured successfully but generated zero CTest tests. The plugin CMake now defines the option with a standalone default of ON.
5. The first plugin build failed in the sandbox because CMake `FetchContent` could not resolve `github.com` for `cgltf`. Re-running with network permission succeeded.
6. OpenUSD's bundled oneTBB 2021.9.0 needs `-DCMAKE_POLICY_VERSION_MINIMUM=3.5` when configured by CMake 4.1.
7. OpenUSD's Apple Silicon codesign step assumes full Xcode because it calls `xcodebuild -version`. This host only has Command Line Tools. I patched `/private/tmp/OpenUSD-25.05.01/build_scripts/apple_utils.py` locally to fall back to ad-hoc `codesign -s -`.
8. usdview requires Python-side UI packages. Installed `PySide6`, `PyOpenGL`, and `Jinja2` into the Python 3.13 user site. `pyside6-uic` lives in `$HOME/Library/Python/3.13/bin`, so that directory must be on `PATH` for the OpenUSD build.
9. `usdview --help` initially warned that `$HOME/.usdview` could not be created from the sandbox. Creating that directory removed the settings-directory issue for GUI checks.
10. Calling the runtime's `bin/usdview` directly without the OpenStrata environment fails because `PYTHONPATH` does not include the runtime's `lib/python`. Use `ost plugin view`, `ost plugin run`, or `eval "$(ost env cy2026 --profile usd)"` before direct `usdview` calls.
