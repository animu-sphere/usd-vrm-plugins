## TL;DR

**All five v0.13.0 asks were delivered, and they unblocked the release lane
completely.** Packaging is digest-reproducible (measured, twice, including with
our new configure-time `buildInfo.json` stamp in the bundle), the PDB is split
out by default, `--from-package` gives a native packaged-artifact verification
level, and `run --no-inject/--plugin-path` exists for bare-runtime sessions.
The remaining friction is small: `ost ci generate` cannot express repo-specific
steps (so the release lane is hand-authored YAML with duplicated pins), and
`ost ci validate` silently accepts unknown cell keys.

## v0.13.0 asks — scorecard (all delivered)

1. **Reproducible packaging** ✅ — two `plugin package` runs of an unchanged
   build now give the *same* archive digest (`sha256:87ee36e4…` twice, 30
   files). We made it a hard release gate (package twice, compare digests) and
   it held on the first try. Also survived adding a configure-time-generated
   file (`buildInfo.json`) to the bundle.
2. **PDB split** ✅ — lean main archive (15.2 MB) + sibling
   `usdVrm-…-debug.tar.zst` (0.4 MB) by default; `--with-debug` opts back in.
   `--json` grew a tidy `data.debug {archive, archive_digest, archive_size,
   files}` block. Exactly the asked shape.
3. **Bare-runtime run** ✅ — `plugin run --no-inject` +
   `--plugin-path <extracted-tree>` documented for clean-install-style
   sessions. (Not yet exercised in our lane; the extracted-bundle
   `plugin run <dir>` idiom from 0.12.0 still covers us.)
4. **Packaged-artifact smoke level** ✅ — `plugin test --from-package`
   extracts the already-built package to a clean dir and runs
   discovery/open/validate against it. Ran green locally (15 pass / 2 skip vs
   the source tree's 14/3 — it *adds* the packaged-discovery check). Wired
   into the release lane on all three OS.
5. **`extract --into` alias** ✅ — `artifact extract <DIGEST> [DEST]` now also
   accepts `--into <DEST>`.

## What worked well beyond the asks

- **`plugin publish`** re-publish of the refreshed lean package was
  frictionless; the scheduled-lane `plugin_artifact` pin swap was a one-line
  matrix edit + regenerate.
- The reproducibility guarantee made **release checksums + a configure-time
  build-metadata stamp** compose cleanly: as long as the stamp itself is
  deterministic (no timestamp), the package digest is a pure function of
  source + toolchain. This is exactly the §7.7 contract we wanted.

## v0.14.0 asks (priority order)

1. **`ost ci generate` — repo-specific steps / lanes.** The release lane
   (`release.yml`) had to be hand-authored: it needs extra steps
   (`--from-package`, a repo smoke script, package-twice digest gate, artifact
   staging) and a `tag` trigger, neither expressible in `openstrata.ci.yaml`.
   Consequence: the runtime pins + bootstrap block are now **duplicated** in a
   file the generator does not own, with a "mirror openstrata.ci.yaml" comment
   as the only drift guard. Wanted (either works):
   a. cell-level hooks (e.g. `extra_steps:` / `from_package: true`) rendered
      into the generated workflow, or
   b. a generator-owned reusable action/fragment (bootstrap + runtime pull)
      that hand-authored workflows can `uses:` so pins live in ONE place.
2. **`ost ci validate` should reject unknown cell keys.** `from_package: true`
   (not a real key) validates "structure OK" and is silently dropped by the
   renderer — the exact failure mode that would mask a typo'd real key.
3. **Version-bump helper for the bootstrap pin.** `bootstrap.ost.version`
   `0.12.0 → 0.13.0` required editing the manifest and regenerating — fine —
   but the workflow's pinned-sha256 escape hatch (`pinned=""` case in the
   generated bootstrap) is still an empty stub; consider `ost ci generate
   --pin-ost-sha256` filling it from the release's published checksums.
4. **(carried) corpus CTest smoke in generated lanes** — unchanged from the
   0.12.0 report's ask; subsumed by ask #1a here.
