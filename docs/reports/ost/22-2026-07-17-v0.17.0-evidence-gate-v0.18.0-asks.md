# OST v0.17.0: the evidence gate — `ci generate` emits a gate no producer can satisfy, and `import` silently drops the evidence that would satisfy it

- Date: 2026-07-17
- Versions exercised: `ost 0.17.0` (bootstrapped from the pinned
  `animu-sphere/open-strata` v0.17.0 release asset on GitHub-hosted
  windows-2022 / macos-15 / ubuntu-24.04, and locally on Windows).
- Prior report: `21-2026-07-15-v0.17.0-phase3-resolver-split-dogfooding.md`.
- Context: PR #44 (workspace Phase 3, the `usdVrmPackageResolver` split) adopted
  ost 0.17.0 and regenerated CI. **All nine PR cells went red immediately**, in
  10–18s, on every platform. This report is the diagnosis and the repo-side
  workaround.

## TL;DR

Adopting 0.17.0 turned CI red without a single line of repo logic changing,
and the two causes are both in `ost`:

1. **`ost ci generate` unconditionally emits `--require-sbom
   --require-provenance`** for the pinned runtime artifact. Any project whose
   runtimes were published before evidence existed is instantly and
   permanently red on regeneration. There is no matrix knob and no warning.

2. **`ArtifactStore::import` silently discards evidence** when the archive
   digest is already in the registry and the producer manifest differs — which
   it always does across ost versions. So the obvious migration ("re-export the
   runtime with a newer ost to attach evidence") is a **success-reporting
   no-op**. This is the more serious of the two: it cost a full CI round to
   detect precisely because it does not fail.

The good news: once (2) is worked around, the fix is cheap and *non-invasive*.
Re-exporting is byte-reproducible on Linux/Windows, so evidence attaches to
**the exact `runtime_artifact` digest CI already pins** — no runtime change, no
OpenUSD rebuild. That is the outcome we shipped.

The `runtime_remote` OCI digest is a different matter, and the reason this took
two attempts: attaching provenance makes the manifest digest **unpinnable across
republishes**, because provenance embeds the producing commit (§6). Identical
payload, new address, every time. We deleted the old manifests on the assumption
that a re-push would land on the pinned digests, and broke `main` for ten
minutes proving otherwise.

Two smaller findings: an artifact record's `producer` field records the ost that
**imported** it, not the one that built it (§4), and the generated cache-hit
test does not use the gate's own predicate, so a cached evidence-less record
would wedge a lane permanently (§8).

Separately, a probe into workspace packaging ahead of our Phase 5 (§11) found
per-bundle `plugin package` solid, but no product-level closure above it:
`dependencies.json` resolves `requires.libraries` and omits `requires.bundles`
entirely, `--workspace` exists only on `test`, and `--from-package` is
incompatible with it — so the multi-bundle configuration a consumer installs is
the one configuration ost cannot package or verify.

## 1. What broke

`ost ci generate github` (0.17.0) renders this into every source-CI cell:

```yaml
- name: Verify and materialize the pinned runtime SDK
  run: |
    ost artifact verify ${{ matrix.runtime_artifact }} --minimum-trust ${{ matrix.minimum_trust }} --require-sbom --require-provenance
```

Against the runtimes this repo has pinned since v0.9.0:

```text
Verify sha256:c1c54f1e8a31983ad24f2997e419b000d4ad5d030c392d592d20fd6c8ac8b227
  archive digest: OK
  files matched:  4045
  trust:          local
  policy minimum: local
  policy result:  OK
  SBOM: FAIL
  ARTIFACT_SBOM_REQUIRED: artifact has no attached SPDX SBOM
  provenance: FAIL
  ARTIFACT_PROVENANCE_REQUIRED: artifact has no attached SLSA/in-toto provenance
  result: FAIL      (exit 5)
```

Nine cells, all three OSes, failing at the same step. Nothing in the repo could
fix it: the images are on GHCR, published before ost could emit evidence.

Note what the gate actually gates. `minimum_trust: local` is the generator's
default, and `local` is the **weakest** rung of the ladder
(`Local < Unsigned < Attested < Verified < Trusted`), so the trust floor admits
everything. The only live constraint in that command is the evidence pair —
precisely the part that cannot be satisfied.

### 1.1 There is no producer that can satisfy it, locally

`ost 0.17.0` has no `artifact attest`, no `sbom` verb, and no
`--sbom`/`--provenance` flags on `runtime export` or `plugin package`. Evidence
is only ever produced as a side effect of exporting/packaging, via
`generate_evidence(dist, manifest)`, which writes two fixed-name sidecars into
the dist dir:

```text
sbom.spdx.json
provenance.intoto.jsonl
```

and **provenance is written only when the manifest carries `build` metadata**,
which comes from `github_build_metadata()` — reading `GITHUB_REPOSITORY`,
`GITHUB_SHA`, `GITHUB_WORKFLOW_REF`, `GITHUB_REF`, `GITHUB_ACTOR`,
`GITHUB_EVENT_NAME`. Outside GitHub Actions it returns `None`.

Verified locally (Windows, re-export of an already-materialized runtime):

```text
dist/  manifest.json  SHA256SUMS  *.tar.zst  sbom.spdx.json
       (no provenance.intoto.jsonl)

ost artifact verify sha256:2ac64c5e… --minimum-trust local --require-sbom --require-provenance
  SBOM: OK
  provenance: FAIL
  ARTIFACT_PROVENANCE_REQUIRED
```

So: **a local build, however long, can never satisfy the generated gate.** This
is worth stating plainly because the intuitive reaction to
`ARTIFACT_SBOM_REQUIRED` is "rebuild the runtime" — hours of OpenUSD compile on
two or three platforms — and it would not have worked.

## 2. P0 — `ci generate` emits a gate with no migration path

The flags are hardcoded in `crates/ost-ci/src/github.rs` (four render sites,
plus a test asserting `verify.matches("--require-sbom").count() == 2`). There is
no cell key, no global, and no `--no-require-evidence`.

The failure mode for any existing project is: upgrade ost → regenerate CI →
every lane red → and the natural remedy (re-export) silently does nothing
(§3). We hit exactly that sequence.

**Acceptance criteria**

- The evidence requirement is expressible in `openstrata.ci.yaml` — per cell
  and/or globally, e.g. `require_evidence: true|false`, defaulting to whatever
  ost considers right for new projects.
- `ost ci generate` **warns at generation time** when a pinned
  `runtime_artifact` in the local registry lacks the evidence the rendered gate
  will demand. The information is available at that moment; discovering it via
  a red hosted matrix is a bad trade.
- Ideally `ost ci validate` fails fast on the same condition, so the mismatch
  surfaces before CI runs at all.

## 3. P0 — `import` silently discards evidence on an existing digest

This is the headline. `ArtifactStore::import`:

```rust
let hex = record.digest_hex().to_string();
let object_dir = self.object_dir(&hex);
if object_dir.join(RECORD_FILE).as_std_path().is_file() {
    let mut existing = self.read_record(&object_dir)?;
    let stored_manifest = std::fs::read(object_dir.join(MANIFEST_FILE))?;
    if stored_manifest == manifest_bytes {          // <-- the gate
        …copy sbom/provenance into object_dir, rewrite the record…
    }
    self.index_upsert(&existing)?;
    return Ok(ImportOutcome { record: existing, already_present: true });
}
```

The evidence attach sits **inside** `stored_manifest == manifest_bytes`. A
re-export by a newer ost always changes that manifest — the `producer` string
alone guarantees it, and under Actions `generate_evidence` also inserts the
`build` object *before* the manifest is written. So the guard never matches on
the one path that needs it, the sidecars are dropped, and the caller is told
`already_present: true` with an `ok` status and no warning.

`ost runtime export` calls this (`registry.import(&dist_dir,
ArtifactSource::Published)`), so the drop happens inside the very command whose
job is to produce evidence.

### 3.1 How it presented

Republish run
[29514114428](https://github.com/animu-sphere/usd-vrm-plugins/actions/runs/29514114428),
which materialized each published runtime from its pin and re-exported it under
0.17.0:

| cell | re-exported digest | vs. the pin it was replacing | verify |
|---|---|---|---|
| linux | `sha256:f6a2047c…` | **identical** | `ARTIFACT_SBOM_REQUIRED` |
| windows | `sha256:c1c54f1e…` | **identical** | `ARTIFACT_SBOM_REQUIRED` |
| macos-arm64 | `sha256:5630132e…` | different | PASS |

The Linux/Windows exports are byte-reproducible against the published archives,
so they collided with the pulled record and took the short-circuit. The
`file_count` guard passed (3948/4045, unchanged), the export reported success —
and the artifact was exactly as evidence-less as before.

macOS passed **by accident**: ost 0.12 changed macOS packing to preserve exec
bits, so a 0.17 re-export of an older image lands on a different digest and
takes the fresh path, which does write evidence. Reproducibility, normally a
feature, is what makes the bug bite — and the platform where reproducibility is
broken is the only one that worked.

### 3.2 It breaks consumers too, and says so while reporting success

The producer side is only half of it. Once the republished, evidence-carrying
image was on GHCR, pulling it onto a machine that already had a record for that
digest **does not attach the evidence either** — same short-circuit, same
dropped sidecars. Reproduced locally against the real published image:

```text
$ ost artifact pull "oci://…openstrata-runtime-cy2026-usd@sha256:e5e38405…" \
    --expect-artifact sha256:c1c54f1e… --require-kind runtime
  skipped sbom
  skipped provenance
  passed  local_import
  import:     already-present (…\artifacts\objects\sha256\c1c54f1e…)

$ ost artifact show sha256:c1c54f1e… --json | …
sbom: None / provenance: None / trust: local

$ ost artifact verify sha256:c1c54f1e… --minimum-trust local --require-sbom --require-provenance
  SBOM: FAIL       ARTIFACT_SBOM_REQUIRED
  provenance: FAIL ARTIFACT_PROVENANCE_REQUIRED
  result: FAIL
```

Note `skipped sbom` / `skipped provenance`: **the command knows** it is
discarding evidence that the remote is offering, prints it as a routine step
result, and exits `ok`. A user reading that output has been told the truth and
still has no reason to think anything is wrong.

The practical consequence is that **evidence can never reach any machine that
already holds the digest**, and there is no `ost artifact rm` to reset it — the
only escape is `rm -rf` inside the store. That is not a hypothetical: this
repo's `usdvrm-support-windows-cy2026` cell runs on a self-hosted runner with a
persistent registry, so a lane that pins an unchanged digest is now permanently
red on that machine until someone reaches into `$OST_HOME/artifacts` by hand.
Hosted lanes survive only because their runners are fresh.

It also quietly punishes doing the right thing. Keeping the artifact digest
stable across the republish — the conservative choice, since the payload really
is identical — is precisely what triggers the collision. Had we let the digest
churn, every consumer would have picked up the evidence automatically.

### 3.3 Why the guard is wrong, not just unlucky

Evidence is bound to the **archive digest** (`verify_sbom(&…, &record.digest)`,
and provenance subject-matching on the same digest). At the short-circuit, that
digest is equal by construction — that is *why* we are on this branch. The
producer manifest's equality is irrelevant to whether the evidence describes
these bytes. The comment above the guard explains it as repairing "this exact
producer manifest … imported by a pre-sidecar-aware version", which is a
coherent goal for the *debug archive* repair it was written for; the evidence
attach appears to have been folded into the same block, where it does not
belong.

### 3.4 Workaround (shipped)

Evict the artifact registry between materializing and exporting. The runtime
store (`$OST_HOME/runtimes`) is a sibling of the artifact store
(`$OST_HOME/artifacts`), so the runtime survives and the export's import is
forced onto the fresh path:

```bash
ost artifact pull "oci://…@${OCI}" --expect-artifact "${ARTIFACT}" --require-kind runtime
ost runtime pull cy2026 --profile usd --from-artifact "${ARTIFACT}" --force
ost runtime validate cy2026 --profile usd          # export refuses an unvalidated runtime
rm -rf "$OST_HOME/artifacts"                       # <-- the workaround
ost runtime export cy2026 --profile usd --dist dist   # fresh import path -> evidence lands
```

Result — run
[29514518826](https://github.com/animu-sphere/usd-vrm-plugins/actions/runs/29514518826),
all three green:

| cell | digest | vs. pin | trust | verify |
|---|---|---|---|---|
| windows | `sha256:c1c54f1e…` | **unchanged** | `local` → `attested` | SBOM OK, provenance OK |
| linux | `sha256:f6a2047c…` | **unchanged** | `local` → `attested` | SBOM OK, provenance OK |
| macos-arm64 | `sha256:5630132e…` | changed | `local` → `attested` | SBOM OK, provenance OK |

`file_count` matched the source on all three. Because the archive bytes are
unchanged, **evidence attached to the digests CI already pins** — the migration
required no runtime change and no `runtime_artifact` re-pin on two of three
platforms.

**Acceptance criteria**

- Move the evidence attach out of the `stored_manifest == manifest_bytes`
  guard: if the incoming dist carries sidecars that verify against the stored
  archive digest, attach them, whatever the manifest says.
- Never drop caller-supplied evidence silently. If it is refused, say so with a
  code and a non-zero exit; `ok: true` + `already_present: true` while
  discarding freshly generated attestations is the worst available outcome.
- Surface it in `--json`: an `evidence_attached` / `evidence_skipped` field on
  the import outcome would have made this a 30-second diagnosis instead of a
  hosted CI round trip.

## 4. P1 — `record.producer` is the importer, not the producer

```rust
// ArtifactStore::import
let producer = format!("ost {}", env!("CARGO_PKG_VERSION"));
let mut record = ArtifactRecord::from_producer_manifest(&manifest, source, created, &producer)?;
```

The record's `producer` is stamped with the version of ost performing the
**import**. The same GHCR image therefore reads:

- `"producer": "ost 0.10.0"` on this workstation (imported 2026-07-11), and
- `"producer": "ost 0.17.0"` on a fresh runner that pulled it minutes ago.

This misled the first pass of this very investigation: the local record was read
as evidence that the images were *built* by 0.10.0. They may well have been, but
the field cannot support that claim. Under a field named `producer`, on a record
whose whole purpose is provenance, that is a trap.

**Acceptance criteria**

- Either carry the producing tool from the producer manifest (it is right
  there in the dist), or rename the field to `imported_by` / `importer` so it
  cannot be read as origin. The former is more useful: "what built this?" is a
  question consumers legitimately have, and today nothing answers it.

## 5. P2 — provenance is GitHub-Actions-only, with no escape hatch

`github_build_metadata()` is the sole source of `manifest["build"]`, and
`generate_evidence` writes provenance only if that key exists. There is no
`--build-metadata <file>`, no env-agnostic path. Consequences:

- Any non-GHA CI (GitLab, Buildkite, local reproducible builders) can produce an
  SBOM but never provenance, and therefore can never satisfy a generated gate
  that ost itself renders.
- Testing the publish path requires burning hosted CI minutes; it cannot be
  rehearsed locally.

The strictness itself is right — the docstring's "All fields are required so a
partial environment never produces provenance that looks authoritative" is a
good instinct. The ask is an *explicit* input, not a looser inference:
`ost runtime export --build-metadata build.json`, validated against the same
required-fields rule.

## 6. P2 — attaching provenance makes the OCI digest unpinnable across republishes

Not a bug, but a consequence worth stating in the docs, because it silently
inverts an assumption most people will start from.

An OCI manifest is built from `build_oci_manifest(layers, config_digest,
config_size)` and does not contain its own tag, so "re-pushing identical bytes
under a different tag reproduces the same manifest digest" *sounds* safe. It is
not, once evidence is attached: the manifest covers the evidence layers, and
`provenance.intoto.jsonl` embeds the producing commit (`github_build_metadata()`
reads `GITHUB_SHA`, `GITHUB_REF`, `GITHUB_WORKFLOW_REF`, …). Republishing from a
different `main` rewrites the provenance, hence its layer digest, hence the
manifest digest — while the archive bytes, and therefore the **artifact**
digest, stay byte-identical.

We learned this by doing it (run 29517954310). Re-pushing the same three
runtimes onto their canonical tags produced:

```text
windows  artifact c1c54f1e (unchanged)  OCI e5e38405 -> bf24d8b6
linux    artifact f6a2047c (unchanged)  OCI 73a7d359 -> 3f785e30
macos    artifact 5630132e (unchanged)  OCI 00007f4b -> 459ec9a3
```

Three identical payloads, three new addresses. `openstrata.ci.yaml` pins both
`runtime_artifact` (ost digest) and `runtime_remote.expected_oci_digest` (OCI
digest); only the first is stable. **The OCI pin must be refreshed after every
republish, including one that changes nothing.**

The two digests behave so differently that the distinction deserves to be
explicit: `runtime_artifact` is the contract (it is what `--expect-artifact`
enforces, and what stays fixed across a byte-reproducible re-export);
`runtime_remote` is a mutable address that churns whenever provenance is
regenerated. A note in the matrix schema docs would have saved us a broken
`main` — we deleted the old manifests *before* discovering the new push would
not land on the pinned digests.

**Acceptance criteria**

- Document that `expected_oci_digest` is expected to change on every republish
  while `runtime_artifact` is not, and say why (evidence layers are
  build-specific).
- Consider making `ost artifact push --json` state plainly that the OCI digest
  is provenance-dependent, or offer `ost ci repin <cell> --oci <digest>` so the
  follow-up edit is a command rather than a careful hand-edit across
  `openstrata.ci.yaml` + a hand-authored `release.yml` + regenerated workflows.

## 7. Credit where due — `artifact push`'s error is a model

Publishing the republished runtimes failed on all three cells with GHCR 403.
The report ost emits is exactly what an error should be:

```json
{
  "error": {
    "category": "precondition",
    "code": "ARTIFACT_AUTH_DENIED",
    "message": "ghcr.io denied access to 'oci://ghcr.io/animu-sphere/openstrata-runtime-cy2026-usd:26.05-linux-x86_64-attested' (403)",
    "hint": "credentials were accepted but the registry still refused the write to 'animu-sphere/openstrata-runtime-cy2026-usd': confirm the token has write:packages scope and your account may publish there (on GHCR the package must grant you write, or the org must allow package creation)"
  }
}
```

It distinguishes *authentication succeeded* from *authorization refused* — the
single most useful bit when a 403 could be either — and the hint names the
actual remedy (the GHCR package must grant the repo write access; a
`GITHUB_TOKEN` with `packages: write` is not sufficient for an org-level package
that was published out-of-band and never linked to the repo). Diagnosis took one
read, no guessing.

Contrast §3, where the *silent* success cost a full hosted round trip. The gap
between these two is the argument for the §3 asks: this codebase clearly knows
how to report a precondition well.

The self-inflicted half is worth recording too: our first push step ran
`ost artifact push … --json > push.json` under `set -e`, so the report above
went into a file the shell never got to print, and the failure surfaced as a
bare `exit code 4`. Redirecting `--json` is a foot-gun whenever the command can
fail — arguably ost should also write the error report to stderr, or at least
the `code` line, so a redirected stdout cannot hide it.

## 8. Smaller notes

- **`runtime export --json` key names.** The success payload is `data.digest`
  and `data.files`; `artifact show --json` uses `data.artifact.digest` and
  `data.artifact.file_count`. Same concepts, two shapes, so any script that
  moves between them guesses wrong once. Worth unifying, or documenting
  together.
- **`--slim` is still a trap for this repo** (carried from v0.9.0): it drops the
  top-level `resources/` tree, and `find_package(pxr)` →
  `MaterialXConfig.cmake` `set_and_check(<prefix>/resources)` then fails to
  configure. We publish full. A `runtime export --slim` that warned when the
  layout it is about to drop is referenced by the SDK's own CMake config would
  have saved that discovery.
- **`ost artifact` has no `rm`.** The §3.4 workaround is `rm -rf` on the store's
  internals, i.e. reaching past the CLI into its on-disk layout. An
  `ost artifact rm <digest>` (or `--force-reimport` on export) would let a
  project do this supported-ly. §3.2 raises the stakes: without it, a machine
  that already holds a digest has *no supported way* to ever obtain its
  evidence.
- **The generated cache-hit test does not use the gate's own predicate.**
  `ci generate` renders a cache short-circuit that decides the cached registry
  is good using a bare `ost artifact verify … --json`, then gates the job with
  `verify … --require-sbom --require-provenance`. A cached record satisfying the
  first and failing the second skips the pull and fails the gate — on every
  subsequent run, unfixably, because §3.2 means the pull could not have
  attached evidence anyway. We dodged it only because bumping `bootstrap.ost`
  to 0.17.0 rotated the cache key prefix (`ost-registry-0.17.0-…`), so every
  0.15.0-era cache was orphaned. A project that adopted the evidence gate
  *without* an ost version bump would have wedged. The cache test should assert
  what the gate asserts; we hardened our hand-authored `release.yml` this way,
  but the generated lanes are ost's to fix.
- **`--require-sbom` applies to plugin artifacts too, with no story for
  already-published ones.** `usdvrm-support-windows-cy2026` pins
  `plugin_artifact: sha256:87ee36e4…`, which has `sbom: None, provenance: None`.
  That lane is scheduled and self-hosted, so it is red on both counts (§3.2 for
  the runtime, no evidence at all for the plugin) and cannot be fixed by
  republishing runtimes alone.

## 10. Repo state after this report

**PR #44's nine PR cells are green** (run
[29516785891](https://github.com/animu-sphere/usd-vrm-plugins/actions/runs/29516785891))
— the evidence gate now passes on every hosted lane, on the same runtime
payloads, without an OpenUSD rebuild.

- The republish workflow (PRs #45–#49) was **removed** once the migration landed
  (#50): its source pins named the pre-evidence images, which no longer exist,
  and a standing ability to publish to a shared registry is not worth keeping
  for a job that should happen approximately never. `openstrata.ci.yaml`'s
  digest-provenance comment carries the two non-obvious facts a re-run would
  need (evict the store before exporting; re-pin the OCI digest every time) and
  points here.
- The package carries the canonical three tags only, each resolving to the
  evidence-carrying image. The `…-attested` tags were a migration guard and are
  gone; so are the pre-evidence manifests.
- `openstrata.ci.yaml` keeps its Windows/Linux `runtime_artifact` pins (evidence
  attached to the same digests) and takes a new macOS artifact pin plus three
  `runtime_remote` OCI digests — re-pinned twice, once after the republish and
  again after the retag (§6). `release.yml` is re-pinned to match and now
  requires the same evidence as the PR gate — it had been left verifying the
  runtime bare, i.e. the publishing lane was the weakest verifier in the repo.
- Final state verified independently of CI: each `(runtime_remote,
  runtime_artifact)` pair was pulled into a throwaway `OST_HOME` — a fresh
  registry, which is the only way to observe what a clean runner sees, given
  §3.2 — and run through the gate. All three: `trust: attested`, SBOM OK,
  provenance OK.

### Known-red, deliberately not fixed here

`usdvrm-support-windows-cy2026` (scheduled, self-hosted) is expected to fail its
next run, for two independent reasons neither of which the republish addresses:

1. its runner's persistent registry holds the pre-evidence record for
   `sha256:c1c54f1e…`, and per §3.2 no `ost artifact pull` can attach evidence
   to a digest already present — someone must `rm -rf` that runner's
   `$OST_HOME/artifacts` by hand;
2. its `plugin_artifact: sha256:87ee36e4…` has no evidence at all and would need
   republishing through a plugin package flow.

Recording it rather than papering over it: the lane is genuinely broken by the
0.17 adoption, and the fix is an ost-side question (§2's knob, or §7's
`artifact rm`) as much as a repo-side chore.

## 11. P1 — packaging stops at the bundle; the workspace product has no closure

Added the same day, from a separate probe: we went looking for per-bundle
packaging before starting workspace Phase 5. **It exists and it works** — this
is not an ask for something missing at the bundle level. The ask is that ost
models a *workspace* everywhere except the one verb that ships it.

### 11.1 What works today

`ost plugin package <BUNDLE>` packages one bundle, deterministically:

```console
$ ost plugin package plugins/usdVrmFileFormat --json
{"data":{"archive":".../usdVrmFileFormat-0.1.0-cy2026-windows-x86_64-py313-usd.tar.zst",
         "archive_digest":"sha256:b6b8ce57c48c9946bf5900561430e943fd6d11536b0522a763ddc25a2ce561eb",
         "archive_size":15138472,"files":43,"packaged":true,
         "target":"cy2026-windows-x86_64-py313-usd"},"ok":true,"schema":1,"warnings":[]}
```

The 0.16 `requires.libraries` work shows up here exactly as promised: the
extracted tree carries `vrmContainer` staged under `runtime/libraries/{bin,lib}`
(DLL, import lib, and its CMake package) plus a `dependencies.json` recording
provenance `source-workspace`. That half of the closure is solved.

### 11.2 The asymmetry: `libraries` are staged, `bundles` are not

The same extracted package has **no trace of `vrmSchema`**, the bundle the
importer declares and links against:

```console
$ ost artifact extract sha256:b6b8ce57c48c <tmp>     # 43 files
$ cat <tmp>/dependencies.json
{"libraries":[{"id":"vrmContainer","version":"0.1.0","provenance":"source-workspace",
               "runtime_directories":["runtime/libraries\\bin","runtime/libraries\\lib"],
               ...}],
 "schema":"openstrata.dependencies/v1alpha1"}
```

`dependencies.json` has a `libraries` key and **no `bundles` key at all** — not
an empty list, not an unresolved reference. The shipped
`openstrata.plugin.yaml` still declares `requires.bundles: [{id: vrmSchema,
version: ">=0.1,<0.2", contract: 1}]`, so the *intent* ships while the
*resolution* does not. A consumer who extracts this artifact holds a plugin that
cannot apply its own schemas, and nothing in the resolved dependency record says
so. The manifest is the only hint, and it is a declaration, not a closure.

This is the same class of bug as §3.2, one layer up: the artifact reports
success while omitting the thing that makes it usable.

### 11.3 There is no workspace-level packaging verb

`--workspace` exists on exactly one subcommand:

| subcommand | `--workspace` |
| --- | --- |
| `plugin test` | yes (validates the graph, then tests every discovered bundle) |
| `plugin build` / `run` / `view` / `package` / `publish` / `inspect` / `doctor` | no |

So the graph ost validates in `test --workspace` is unavailable to `package`.
Worse, `--from-package` is explicitly **"incompatible with `--workspace`"**, so
the composed workspace is the one configuration that cannot be verified from its
packaged artifacts — precisely the configuration a consumer installs.

WORKSPACE.md §5 specifies an aggregate artifact
(`usdVrmPlugins-<version>-<target>.tar.zst`) alongside the per-bundle ones.
There is no verb that emits it, so our clean-install harness hand-rolls the
closure ost already knows how to compute
(`scripts/clean_install_smoke.py`):

```python
# The importer's runtime bundle dependencies for a full textured clean-install:
DEFAULT_RUNTIME_BUNDLES = ["plugins/vrmSchema", "plugins/usdVrmPackageResolver"]
...
# The aggregate clean-install therefore composes all three shipped packages with
# `ost plugin run --no-inject --plugin-path` (there is no single-package product
# closure yet — Phase 5).
```

Three `package` + `import` + `extract` cycles, then one `plugin run --no-inject`
with three `--plugin-path` flags. Every workspace that ships more than one
bundle rewrites this script. It is also a *weaker* test than ost could give: our
list is hand-maintained, so a new `requires.bundles` edge silently fails to be
covered until someone remembers to add it here.

### 11.4 Asks for v0.18.0

1. **`ost plugin package --workspace`** — package every discovered bundle in
   dependency order, the same set `test --workspace` validates. Purely additive.
2. **Record `bundles` in `dependencies.json`**, resolved the way `libraries`
   already are (id, version, contract, provenance). Even without staging, a
   consumer could then *detect* the missing closure instead of discovering it as
   a runtime schema-application failure.
3. **An aggregate product artifact** composing the workspace into one
   installable tree (WORKSPACE.md §5). If the product model is too big for
   0.18.0, (1) + (2) already retire most of `clean_install_smoke.py`.
4. **Let `--from-package` compose `--workspace`**, so the packaged aggregate is
   verifiable by the same pyramid as the source tree. This is the acceptance
   test for (1)–(3): today it is the documented incompatibility.

Priority is P1, not P0: unlike §2/§3 nothing here is *red*, and the workaround
is a script we already maintain. But it is the last thing standing between this
repo and Phase 5, and (2) is arguably a correctness bug in the artifact contract
rather than a missing feature.

### 11.5 Smaller note found in the same probe — mixed path separators ship

Both the staged manifest and `dependencies.json` carry Windows separators inside
otherwise-POSIX relative paths:

```yaml
requires:
  runtime_libs:
  - runtime/libraries\bin      # note the mixed separators
  - runtime/libraries\lib
```

`"runtime/libraries\\bin"` appears the same way in `dependencies.json`. It is
host-shaped data written into a portable, digest-addressed artifact. It happens
to work because Windows accepts both, but it means a Windows-produced package
and a Linux-produced one differ in a field that describes the same layout, and
any consumer splitting on `/` gets a wrong component. Normalizing to `/` at
staging time would also remove a gratuitous source of cross-target digest
divergence.
