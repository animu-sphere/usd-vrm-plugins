# OST v0.18.0: cutting a real multi-bundle release — the workspace verbs hold, and the staged dependency layout has no consumer-facing contract

- Date: 2026-07-18
- Versions exercised: `ost 0.18.0` (GitHub-hosted windows-2022 / macos-15 /
  ubuntu-24.04, and locally on Windows).
- Prior report: `23-2026-07-18-v0.18.0-workspace-packaging-v0.19.0-asks.md`
  (same day — report 23 is the probe, this is the release that used it).
- Context: **v0.2.0 shipped** on 2026-07-18, this repo's first multi-bundle
  release: three bundles × three OS, published from `ost plugin package
  --workspace`. This is the report of doing it for real.

## TL;DR

The 0.18.0 workspace verbs are good enough to drive a real release lane, and
they did. Two things are worth sending upstream from the attempt:

1. **The new P1.** `requires.runtime_libs` staging has **no activation contract
   outside `ost`**. ost stages a bundle's dependency libraries into
   `runtime/libraries/{lib,bin}` and knows to activate them at `plugin run`
   time. Nothing in the artifact tells a consumer who is *not* running under
   `ost` how to do the same — and the failure mode blames the wrong component.
   This is the same "ships resolved, isn't usable" shape as report 23 §2, one
   layer further out: there it was the schema registration half, here it is the
   library search path.
2. Report 23's §4 findings held up in production CI: the evidence-gate matrix
   knob and the fixed cache-hit predicate ran green across nine PR cells and
   three release cells, on the first try, with no repo-side workaround.

## 1. What worked, in production

The release lane now runs, per OS cell:

```console
$ ost plugin build <bundle> --target cy2026 --profile usd    # x3, dependency order
$ ost plugin test --workspace --target cy2026 --profile usd --up-to 5 --json
$ ost plugin package --workspace --target cy2026 --profile usd --json
```

Observations from three green release runs plus a dry run:

- **`--workspace` composes cleanly with `--target` / `--profile` / `--up-to`.**
  No flag conflicts, no per-bundle special-casing.
- **`.data.packages[]` is a good CI contract.** Staging release assets is a
  loop over `name` / `archive` / `target` / `debug_archive`, and `.data.order`
  makes the dependency order inspectable. We drive 20 published assets off it.
- **Digest reproducibility survives the workspace verb.** Packaging twice and
  diffing all three digests is stable across runs, which is what lets the
  release publish `SHA256SUMS` at all.
- **`plugin test --workspace` is a real gate.** It caught nothing this release
  because nothing was wrong, but it is the only thing enforcing the manifest
  edges; `WORKSPACE.md` §8 asked for it since Phase 1 and it is finally wired.

Nine PR cells (3 bundles × 3 OS) and three release cells, all green on 0.18.0.

### 1.1 `debug_archive` is always null

Every package in every cell reported `"debug_archive": null`, on all three OS.
The release therefore publishes no `*-debug` artifact — the same shape v0.1.0
shipped, so this is long-standing rather than new, and we have stopped
advertising debug assets in the notes. Flagging it only because `plugin package
--help` documents the lean/split behaviour as the default, which reads as though
a sibling `*-debug` package is the normal outcome. If splitting requires
something of the build profile, saying so at `package` time would be clearer
than a silent `null`.

## 2. P1 — staged `runtime_libs` cannot be activated outside `ost`

This is the new one, and it cost us a re-cut tag to find.

### 2.1 What the artifact contains

The shipped `usdVrmFileFormat` package stages its dependency libraries:

```text
<pkg>/lib                      # libUsdVrmFileFormat.dll   <- the plugin
<pkg>/runtime/libraries/lib    # libvrmSchema.dll
<pkg>/runtime/libraries/bin    # vrmContainer.dll
```

and declares them in the staged manifest:

```yaml
requires:
  runtime_libs:
  - runtime/libraries\bin
  - runtime/libraries\lib
```

Parsing the shipped DLL's PE import table confirms the dependency is real and
direct — `libUsdVrmFileFormat.dll` imports **`libvrmSchema.dll`** and
**`vrmContainer.dll`** by name.

### 2.2 The gap

Under `ost plugin run <extracted-pkg>`, this all works: ost reads the extracted
package's manifest, activates `requires.runtime_libs`, and the plugin loads. Our
clean-install smoke passes on exactly this path, opening a 309-prim avatar and
resolving embedded textures from three composed packages.

A consumer **not** running under `ost` — the documented "install from a GitHub
release into your own OpenUSD environment" path — gets no such help. They see:

```text
Failed to load plugin 'UsdVrmFileFormat': the specified module could not be found
 in '<pkg>/lib/libUsdVrmFileFormat.dll'
Cannot determine file format for @.../textures.vrm@
```

Note the second line: **the failure surfaces as "unknown file format".** USD
found `plugInfo.json`, tried to load the library, failed on a *transitive*
dependency, and reported it as an unregistered format. Nothing in the message
names `libvrmSchema` or `vrmContainer`, and nothing points at
`runtime/libraries/`. A consumer's reasonable first conclusion is that the
plugin is broken or the format was never registered.

### 2.3 Why `requires.runtime_libs` does not close it

The manifest field is a *description*, not an activation mechanism. To use it a
consumer must:

1. know to read the staged `openstrata.plugin.yaml` at all,
2. resolve `runtime/libraries\bin` — **which is not a portable path.** Report
   22 §11.5 and report 23 §5 filed the mixed separators as cosmetic. It is no
   longer cosmetic: this is the exact string a consumer must turn into a
   loader path, and splitting it on `/` yields `libraries\bin`.
3. know the platform-specific way to inject it, which on Windows is genuinely
   non-obvious: **Python 3.8+ removed `PATH` from the DLL search used for
   dynamically loaded modules**, so the "add it to `PATH`" instinct may not work
   from a Python host, and `os.add_dll_directory` applies to the CPython loader
   rather than to whatever `ArchLibraryOpen` does.

We could not establish the correct Windows recipe with confidence, so our
install guide now names the directories and the failure signature but
deliberately prescribes no mechanism, pointing users at the `ost` path instead.
That is an unsatisfying answer to ship, and it is the direct consequence of the
staging being an ost-internal convention.

### 2.4 Methodology note, offered as a warning

Three attempts to establish the mechanism (`PATH`, `os.add_dll_directory`, DLL
co-location) all failed — and all three were invalid. They ran
`ost plugin run plugins/usdVrmFileFormat` (the **source** bundle) while pointing
`PXR_PLUGINPATH_NAME` at the extracted packages, so ost read the source
manifest, staged nothing, and every experiment failed for that reason rather
than the one under test. The working harness passes the **extracted package
root** as the bundle argument.

That is an easy trap precisely because `--no-inject` sounds like it makes the
bundle argument inert. It does not: the bundle argument still selects whose
`requires` get staged. A warning when `--plugin-path` roots do not include the
bundle argument's own tree would have caught it immediately.

## 3. Asks for v0.19.0

Carried unchanged from report 23 §6: **(1) stage a `requires.bundles`
dependency's USD registration half**, (2) **`--from-package` composing
`--workspace`**, (3) **an aggregate product artifact**.

Added here:

4. **Give `requires.runtime_libs` a consumer-facing activation contract.** Any
   of: stage dependency libraries beside the plugin so the platform's default
   search finds them; emit a small per-platform activation snippet into the
   package; or document the layout as a stable, portable contract a consumer can
   implement. Today the layout is real, load-bearing, and undocumented outside
   ost's own runtime.
5. **Normalize staged paths to `/`** — re-filed a third time, and no longer
   cosmetic (§2.3). It is the field a consumer must parse to implement (4).
6. **Warn when `plugin run --no-inject` is given `--plugin-path` roots that
   exclude the bundle argument's tree** (§2.4) — a likely-wrong composition that
   currently fails as a confusing runtime error.
7. **Say something at `package` time when no debug package is produced**
   (§1.1), rather than reporting `debug_archive: null`.

Priority: (4) is P1 and, with report 23's (1), forms one theme — *the artifact
is closed under `ost` and open under everything else*. Both are cases where the
package contains the right bytes and the consumer has no supported way to reach
them. (5)–(7) are small.

## 4. Release outcome

v0.2.0 published 2026-07-18: three bundles × three targets, 20 assets, tagged
`1f5f71d`. The lane was tagged twice — the first tag was deleted unpublished
after the install-path gap in §2 was found end-to-end. Nothing in the packaging
itself had to change between the two cuts.
