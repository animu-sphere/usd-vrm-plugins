New, welcome additions:
- **`ost plugin run --with <bundle>`** now exists — the multi-plugin session
  surface. (Not exercised yet; we still ship a single bundle.)
- **L0 `bundle.runtime_libs`** check (SKIP here: "no runtime library dirs
  declared") — the `requires.runtime_libs` plumbing is in. usdVrm is the
  zero-runtime-libs case (cgltf is compiled into one TU), so it's a clean
  contrast partner for testing a plugin that *does* declare them.
- **L1 `runtime.cxx_abi`** is now enforced (see below).

## Fixed on our side, prompted by 0.3.0

- **`cxx_abi` was wrong in our manifest.** The scaffold default `libstdcxx` was
  never corrected; the runtime is `msvc143`. 0.3.0's now-active ABI check caught
  it (`plugin C++ ABI 'libstdcxx' != runtime 'msvc143'`). Set
  `cxx_abi: msvc143` → green. Good catch by the tool.
  - Minor ask: the manifest ABI is single-valued, but the correct value is
    platform-specific (`msvc143` on Windows, `libstdcxx` on a Linux runtime). A
    cross-platform bundle needs either per-variant ABI in the manifest or an
    "inherit from runtime" sentinel. Today you must edit it per target.

## Still open

- **Version mis-detection persists.** `ost runtime show` / `doctor` / `test`
  still report `openusd 25.05.01` for the v26.08 install (the `<27.0` range

  improved (`msvc143`, `py313`), so the version string looks like the last
  hard-coded/placeholder field — worth deriving from `pxr/pxr.h` `PXR_VERSION`.
