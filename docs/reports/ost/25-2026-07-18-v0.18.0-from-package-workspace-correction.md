# OST v0.18.0 correction: `--from-package` already composed `--workspace` ‚Äî we read the help text instead of running it

- Date: 2026-07-18
- Versions exercised: `ost 0.18.0` (local Windows, `cy2026-windows-x86_64-py313-usd`).
- Prior reports: `23-2026-07-18-v0.18.0-workspace-packaging-v0.19.0-asks.md`
  (¬ß3, the claim being corrected) and
  `24-2026-07-18-v0.18.0-first-workspace-release-v0.19.0-asks.md` (carried it forward).
- Context: upstream pointed out that report 23's re-filed ask ‚Äî "let
  `--from-package` compose `--workspace`" ‚Äî was already implemented when the
  report was written. It is. This report verifies that, retracts the conclusion
  report 23 drew from it, and corrects one attribution upstream made in the
  other direction.

## TL;DR

**Ask 2 was delivered in v0.18.0 itself.** `ost plugin test --workspace
--from-package` works, and on this repo it turns the composed configuration
from unverifiable into green. Report 23 ¬ß3 asserted the opposite and drew a
strong conclusion from it ‚Äî *"there is no ost verb that can verify what we
ship"* ‚Äî which is **wrong and is retracted here.**

We got it wrong the cheap way: we read the shipped `--help` and did not run the
command. The v0.18.0 binary's help text says `incompatible with --workspace`.
That text was stale at the moment it shipped.

**The P0 is unchanged.** Report 23 ¬ß6 ask 1 ‚Äî stage a `requires.bundles`
dependency's USD *registration* half ‚Äî still stands, and the two runs below are
a cleaner proof of it than the one report 23 published.

## 1. The correction, measured

Same packages, same target, same fixture. The only difference is `--workspace`:

```console
$ ost plugin test --workspace --from-package
== usdVrmFileFormat (packaged: ...usdVrmFileFormat-0.2.0-cy2026-windows-x86_64-py313-usd.tar.zst) ==
  [PASS] L2 plugin.discovery    USD registry resolves '.vrm' to a file format
  [PASS] L3 usdcat.read         usdcat read 'minimal.vrm' and emitted USDA
  [PASS] L4 python.stage_open   Usd.Stage.Open() opened the fixture
Result: OK (14 pass, 0 fail, 1 skip)
```

```console
$ ost plugin test plugins/usdVrmFileFormat --from-package
  [PASS] L2 plugin.discovery    USD registry resolves '.vrm' to a file format
  [FAIL] L3 usdcat.read         ... Exception thrown while opening layer: Used null prim
  [FAIL] L4 python.stage_open   Failed to open layer @.../minimal.vrm@
Result: FAILED (12 pass, 2 fail, 1 skip)
```

The implementation is `test_workspace_from_package()` in
`crates/ost-cli/src/commands/plugin.rs`, landed in `ad1ee11` and contained in
the `v0.18.0` tag. It extracts **every** workspace bundle's package up front,
then tests each bundle against its dependencies' *extracted* trees in
`graph.dependency_order()` ‚Äî with the comment stating the intent exactly: *"a
bundle's dependency must already exist as a shipped tree when its turn comes."*

So the composed, packaged configuration ‚Äî the one this repo actually ships ‚Äî
**does** have an ost-native gate, and has had one since v0.18.0.

## 2. Why this does not close the P0

The mechanism matters. `--workspace --from-package` passes because the
dependency's **own separate package** is extracted and put on the path. It does
not make `usdVrmFileFormat`'s package self-closed ‚Äî the second run above is the
same artifact failing alone, and it fails for exactly the reason report 23 ¬ß2
identified: the extracted package contains `libvrmSchema.dll` and the CMake
package (the link half) and no `plugInfo.json` / `generatedSchema.usda` (the
registration half).

Read together, the two runs isolate the defect better than report 23 managed:
**hold the artifact constant, add only the dependency's shipped tree, and L3/L4
flip from fail to pass.** The missing bytes are precisely the registration half.

Ask 1 therefore stands as P0, with its acceptance test now available rather than
hypothetical ‚Äî report 23 ¬ß6 called ask 2 "the acceptance test for (1)", and that
framing survives intact; only its availability changed.

Upstream's `508f862` ("v0.19.0 re-plan: composition and reach, plus the
packaging-closure P0") indicates this is already carried as a P0 upstream too.

## 3. One attribution correcting the other way

Upstream's read was that the stale help string cost this repo a hand-built test
harness ‚Äî `scripts/clean_install_smoke.py`. It did not.

That script is commit `f7a799d`, **2026-07-12**, written against `ost 0.12.0`
(report 15), six days before v0.18.0 existed and eleven days before report 23.
Its stated purpose is the near-term plan's ¬ß7.2 clean-install contract:
package ‚Üí import ‚Üí extract outside the repo ‚Üí run against the shipped layout.
Report 23 ¬ß3's "we gate on `scripts/clean_install_smoke.py` instead" describes
**reusing a script that already existed**, not building one in response to the
help text.

The real cost of the stale string was narrower and entirely ours to absorb: one
wrong paragraph in report 23 ¬ß3, carried into report 24, and an ask re-filed
against a feature that already shipped.

Whether the script is now redundant is a separate question and the answer is
mostly no. `--workspace --from-package` tests each bundle against `minimal.vrm`
inside `.strata/targets/`; the smoke script extracts outside the repo entirely
and drives textured avatars end to end (`textures.vrm`, `Seed-san.vrm`: open,
validate, resolve embedded texture bytes through the package resolver). The
overlap is real but partial, so the smoke script stays as the release gate and
the ost verb joins it rather than replacing it.

## 4. The one genuine ask this produced

**Shipped `--help` text is an interface, and this one was stale in the release
that changed the behavior it described.** The help was corrected upstream after
the tag ‚Äî HEAD now reads *"Composes with `--workspace`, which extracts every
bundle and tests each against its dependencies' extracted trees"*, which is both
accurate and exactly the sentence that would have prevented this report. But
every consumer running the `v0.18.0` binary reads the stale text, and a
downstream consumer's cheapest possible check on a capability question is
`--help`. Release notes are not a substitute: they are read once, help text is
read at the moment of the question.

No mechanism ask here beyond the obvious one ‚Äî treat a behavior change and its
help string as the same change.
