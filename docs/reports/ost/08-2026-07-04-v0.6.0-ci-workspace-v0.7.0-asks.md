## TL;DR

- **`usd-plugin-workspace` is the right repo shape.** This repo already behaves
  like that template: root project manifest plus self-contained plugin bundles
  under `plugins/`, each with its own `openstrata.plugin.yaml` and CMake entry.
- **0.6.0 closes most of the compiled co-located schema ask.** `ost plugin schema
  add` exists, `schema:` in the plugin manifest is recognized, and
  `ost plugin build` can emit an `OPENSTRATA_SCHEMA_SOURCES_FILE` CMake fragment
  with generated schema sources and the correct export define.
- **The immediate `usdVrm` migration should keep schema and file format in one
  bundle.** Splitting a separate `usd-schema` artifact looks elegant, but the
  file-format plugin currently calls typed `Vrm*API` C++ directly. A separate
  schema bundle would make standalone `ost plugin build plugins/usdVrm` and the
  0.6.0 CI artifact model harder than necessary.
- **`ost ci generate github` is not ready to commit as-is.** The generated
  workflow has a YAML indentation bug: `steps:` is emitted at the wrong level.
- **`strata.lock` has a stale extension version but still passes
  `ost lock --check`.** `runtime show` reports OpenUSD `26.08`; the lock records
  `25.05.01`; `ost lock --check --json` still says `up_to_date: true`.
- **`ost plugin package` records concrete ABI, but repeat packaging is not
  cleanly idempotent on this host.** The first package wrote
  `provenance.cxx_abi: "msvc143"`; a later rerun failed on the existing
  `.strata/.../package-stage` with Windows error 5.

## What 0.6.0 Improved

### 1. CI vocabulary exists

The new `ost ci` family is directionally right:

```sh
ost ci init
ost ci validate
ost ci generate github
```

The model is clear: `openstrata.ci.yaml` records explicit support cells, each
with a runtime artifact digest, plugin artifact digest, platform/profile, host
labels, and verification level.

This is a good fit for weekly or release support validation, especially for
heavy real-runtime cells that should not run on every PR.

### 2. Lockfile command exists

`ost lock` and `ost lock --check` are exactly the primitive CI needs before a
build/test step. The command can turn the resolved runtime into a machine-check
gate instead of relying on prose in README.

The repo currently ignores `strata.lock`; that made sense before the lockfile
was a CI primitive. For real CI we should choose one policy explicitly:

- commit `strata.lock` and run `ost lock --check`; or
- keep it generated and avoid presenting it as a reproducibility gate.

### 3. Co-located compiled schema hook exists

A 0.6.0 scaffolded file-format bundle with a schema now gets:

```cmake
if(DEFINED OPENSTRATA_SCHEMA_SOURCES_FILE AND EXISTS "${OPENSTRATA_SCHEMA_SOURCES_FILE}")
    include("${OPENSTRATA_SCHEMA_SOURCES_FILE}")
endif()
```

The generated fragment adds schema include dirs, generated `.cpp` files, and the
generated export define:

```cmake
target_sources(${PLUGIN_NAME} PRIVATE
    ".../schema-sources/metadataAPI.cpp"
    ".../schema-sources/tokens.cpp"
)
target_compile_definitions(${PLUGIN_NAME} PRIVATE SCHEMAHOST_EXPORTS)
```

That is the missing mechanism report #7 asked for.

## v0.7.0 Asks

### 1. Fix `ost ci generate github` indentation

Observed generated excerpt:

```yaml
jobs:
  cell:
    name: ${{ matrix.name }}
    runs-on: ${{ matrix.runs_on }}
    strategy:
      matrix:
        include:
          - name: example-linux-cy2026-usd
            runs_on: ["self-hosted", "linux", "x64"]
steps:
      - name: Check the pinned artifacts are in the local registry
```

`steps:` needs to be under `jobs.cell`, aligned with `strategy`, not at the
document root. Until this is fixed, downstream repos either cannot commit the
generated workflow or must patch it after generation.

### 2. Separate PR CI from support-matrix CI

The 0.6.0 generated workflow assumes both runtime and plugin artifacts are
already in the runner's local registry. That is useful for support validation,
but it is not a normal PR CI flow.

For v0.7.0, consider generating two workflows or two modes:

- **source CI:** checkout -> `ost lock --check` -> `ost plugin build` ->
  `ost plugin test --up-to 5` -> upload reports.
- **support CI:** consume pinned runtime/plugin artifacts from
  `openstrata.ci.yaml` -> `ost artifact verify` -> `ost plugin test`.

This distinction would make the feature easier to adopt without self-hosted
artifact seeding on day one.

### 3. Make placeholder digests harder to miss

`ost ci init` writes all-zero example digests. `ost ci validate` accepts them
without `--resolve`, which is technically fine as syntax validation, but it is
easy to mistake for a usable matrix.

Suggested v0.7.0 behavior:

- warn on `sha256:000...000` placeholders even without `--resolve`; or
- name the command `ost ci lint` for syntax-only checks and reserve `validate`
  for usable support cells; or
- have `ci generate github` refuse placeholders unless `--allow-placeholders`
  is passed.

### 4. Fix lock/runtime extension drift

Current runtime:

```json
"extensions": [
  { "id": "openusd", "version": "26.08", "features": ["core"] }
]
```

Current `strata.lock` after `ost lock` / `ost lock --check`:

```json
"extensions": [
  { "id": "openusd", "version": "25.05.01", "features": ["core"] }
]
```

`ost lock --check --json` still reports:

```json
{ "up_to_date": true }
```

That makes the lockfile unsafe as a CI source of truth. v0.7.0 should either
write the real runtime extension version (`26.08`) or omit extension versions
that cannot be derived from the same source of truth as `runtime show`.

### 5. Document the co-located schema migration path for existing bundles

0.6.0 has the key pieces, but an existing repo still needs clear instructions for
the non-scaffold case:

- add `schema: source: schema/schema.usda` to an existing file-format bundle;
- add each `usd-schema:<TypeName>` to `provides`;
- include `OPENSTRATA_SCHEMA_SOURCES_FILE` from CMake;
- decide whether generated sources are build-tree-only or committed fallback;
- merge generated schema `Types` into the existing `plugInfo.json` that also
  contains the `SdfFileFormat` type;
- ensure `generatedSchema.usda` is staged next to that same `plugInfo.json`.

This repo's likely v0.7-ready shape:

```yaml
plugin:
  name: usdVrm
  kind: usd-fileformat
provides:
  - usd-fileformat:vrm
  - usd-schema:VrmHumanoidAPI
  - usd-schema:VrmExpressionAPI
  - usd-schema:VrmLookAtAPI
  - usd-schema:VrmSpringBoneAPI
  - usd-schema:VrmColliderAPI
  - usd-schema:VrmConstraintAPI
schema:
  source: schema/schema.usda
```

### 6. Support workspace-level plugin testing

For a `usd-plugin-workspace`, it would be useful to have a root-level command
that discovers plugin bundles and runs their build/test gates:

```sh
ost plugin test --workspace --up-to 5
# or
ost workspace test --plugins --up-to 5
```

Today the repo root CMake can discover bundles, but `ost plugin test` remains
bundle-oriented. As soon as a repo has schema, resolver, and file-format bundles,
workspace-level orchestration becomes the natural CI surface.

### 7. Make `ost plugin package` reruns idempotent

After one successful package, a subsequent run failed:

```json
{
  "category": "io",
  "code": "IO_ERROR",
  "message": "i/o error at ...\\.strata\\targets\\...\\package-stage: access denied (os error 5)"
}
```

The first package is useful and records concrete ABI provenance correctly. The
rerun behavior should either clean/reuse `package-stage` safely or use a fresh
staging directory per invocation.

## Net

0.6.0 is a strong step toward making this repo a real `usd-plugin-workspace`.
For v0.7.0, the high-value fixes are: make generated GitHub Actions valid, make
lockfiles reflect the same runtime metadata as `runtime show`, document existing
bundle migration for co-located compiled schemas, and offer a source-based PR CI
path alongside artifact-based support validation.
