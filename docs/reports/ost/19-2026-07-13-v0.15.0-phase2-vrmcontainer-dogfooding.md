- Date: 2026-07-13
- `ost --version`: `ost 0.15.0`
- Context: workspace Phase 2 extracted the GLB/byte-range boundary into the
  plain shared CMake library `libs/vrmContainer`, then exercised v0.15.0's
  source dependency closure with the Phase 1 `usdVrm -> vrmSchema` edge.

  `18-2026-07-13-v0.14.0-phase1-vrmschema-split-dogfooding.md`.

## TL;DR

The top v0.15.0 ask is delivered: a plain `ost plugin build plugins/usdVrm`
now builds and installs `vrmSchema` first, configures the importer against the
workspace prefix, and `plugin test` composes the same dependency into the
session. Generated source CI consumes that closure too, so importer cells are
back at L5 and the repo-owned schema bootstrap is gone.

Phase 2 also confirms the remaining boundary precisely. v0.15.0 fail-closed
rejects the reserved `requires.libraries` key rather than silently ignoring it,
but does not yet build or discover a plain library. `usdVrm` therefore
bootstraps `vrmContainer` as an installed CMake package, stages its DLL/SO in a
declared `requires.runtime_libs` directory, and enforces the no-OpenUSD/no-plugin
boundary with repo tests.

One regression surfaced: Windows L5 still injects `\r` into multiline USDA
string values while capturing flattened output, even when both source and
golden are LF and OpenUSD reads both strings with `\n`. v0.15.0's new bounded
diff and targeted hint make the failure excellent to diagnose, but the suggested
checkout fix cannot fix this transport-created CR. The Windows schema source-CI
cell is temporarily L4; macOS/Linux schema and all importer cells remain L5.

## What v0.15.0 fixed in real use

### `requires.bundles` is executable now

`ost plugin build plugins/usdVrm --dry-run` renders this order:

1. configure/build/install `plugins/vrmSchema` into the target-specific
   workspace prefix;
2. configure/build `plugins/usdVrm` with that prefix on CMake package search.

The real build followed that order and the importer resolved
`find_package(vrmSchema CONFIG REQUIRED)` without repo glue. A selected
`plugin test plugins/usdVrm` automatically carried schema registration and
`libvrmSchema` into the session: 13 pass / 0 fail through L5. The workspace
gate reported 2 nodes / 1 edge / valid and passed both bundles through L4.

The generated three-OS importer cells now use `up_to: 5`. No cell-level
`with:` duplication was needed.

### Versioned `requires:` now fails closed

The silent-typo failure mode from v0.14.0 is closed. For plain libraries the
current contract is deliberately an explicit stop: `requires.libraries` is
reserved but rejected until identity, versioning, package discovery, and
runtime materialization can be made portable.

That is a materially better failure mode, but it leaves one repo-owned adapter:
the importer configures/builds/installs the sibling `vrmContainer` into a
private prefix and consumes only its exported CMake package. It never uses a
sibling `add_subdirectory` in standalone mode.

### The small Phase 1 asks shipped too

- The current `usd-schema-cpp` template now carries the consumer `NOMINMAX` and
  guarded pxr dependency fixes that Phase 1 had to port locally.
- A non-schema consumer dry-run no longer receives the stale generated-schema
  source fragment.
- L5 reports the first bounded line diff and a specific semantic-CRLF hint.

## Phase 2 result

`vrmContainer` now owns:

- strict GLB 2 magic/version/declared-length/chunk-table validation;
- immutable non-owning JSON, BIN, and buffer views with overflow-safe range
  checks;
- the existing content-addressed `images/<hash>.<ext>` naming contract shared
  by the importer and package resolver;
- a shared-library export and installed
  `find_package(vrmContainer CONFIG REQUIRED)` package.

It owns no `plugInfo.json`, plugin manifest, `pxr` include, registration macro,
or OpenUSD link dependency. The boundary check scans source/registration shape
and the built DLL/SO dependency table after every test-enabled repo build;
CTest also records it as a standalone test.

The importer and co-located resolver now validate every GLB before cgltf and use
the same checked buffer-view/resource-name API. Existing authored behavior is
unchanged: the Phase 0 baseline passed, including the plugin-only exported-symbol
union. The plain runtime is staged under `runtime/lib`, declared through
`requires.runtime_libs`, and included in the lean plugin archive.

## New evidence / next asks

1. **Fix Windows L5 flatten transport before semantic comparison.** Repro:
   source fixture and golden are `w/lf`; `Sdf.Layer.FindOrOpen(...).comment`
   returns the identical `\n` string for both; L5's actual flattened text gains
   `\r` first in the authored multiline comment and, after making that comment
   single-line, in the automatically generated multiline `doc`. Preserve real
   authored CR, but do not turn the Windows stdout/file transport newline into
   authored data.
2. **Finish the `requires.libraries` contract.** Phase 2 provides a concrete
   producer/consumer: CMake package id `vrmContainer`, version synchronized with
   the repo, build/install before consumers, prefix discovery, and one runtime
   directory that must reach source and package sessions.
3. **Product/package dependency closure.** `--from-package` on the importer alone
   cannot load `vrmSchema`; `--from-package --with plugins/vrmSchema` passes, but
   that is not a clean aggregate install. This remains Phase 5's digest-pinned
   product descriptor need.
4. Carry the generator-owned bootstrap/pull fragment, coherent
   `ost ci pin bootstrap --version`, and stale-PDB diagnostic follow-ups.
