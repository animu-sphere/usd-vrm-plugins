# OST dogfooding report #6 — templatization impressions: building a *compiled, co-located* schema by hand

- Date: 2026-06-30
- `ost --version`: **0.4.0**
- Same host; runtime USD root `C:/dev/build/usd2605usdview` (v26.x).
- Context: implemented the Phase 4 pilot — a typed **`VrmHumanoidAPI`** compiled into
  the existing `usdVrm` file-format plugin (branch `roadmap/p4-humanoid-schema`). This
  report is the requested "impressions on templatization": what `ost`'s templates did
  and didn't cover for the path we actually took, written from doing it.

## TL;DR

- We **did not use `ost plugin new usd-schema`**. 0.4.0 ships that template (report
  #5), but it produces a **codeless, standalone** bundle; our decision was a
  **compiled schema co-located** in the file-format plugin (so the authorer calls the
  generated `UsdVrmHumanoidAPI::Apply(prim)`). Those are different builds, so the
  template didn't apply.
- The whole pilot was therefore **hand-rolled**, and the steps we hand-rolled are
  exactly the "schema into an existing bundle" support `ost` doesn't own yet (report
  #4 ask #2). They're captured below as the concrete shape of that ask.
- Pleasant surprise: code generated by a **stale pip `pxr` (USD 23.05)** compiled and
  **loaded clean against the 26.x** runtime. Version skew didn't bite for a simple
  single-apply API schema.

## What the template gives vs. what "compiled + co-located" needs

`ost plugin new usd-schema` (codeless, standalone) is a good fit for *its* case — a
resource-only schema other tools can depend on. It does **not** help when the schema
must be **compiled into an existing plugin's library and `plugInfo`**, which is what a
producer (the importer) wants. The delta we filled by hand:

1. **`schema.usda` for codegen, not codeless.** `skipCodeGeneration = false`, and
   `libraryPrefix` tuned to `"Usd"` so class `VrmHumanoidAPI` becomes C++
   `UsdVrmHumanoidAPI` rather than `UsdVrm`+`VrmHumanoidAPI` = `UsdVrmVrmHumanoidAPI`
   (double "Vrm"), while the USD identifier/token stays `VrmHumanoidAPI`. The template
   has no knob or guidance for this name-composition footgun.
2. **Running `usdGenSchema` at all — a Python tool — was the biggest friction**, and
   it's environmental:
   - The `@usd/schema.usda@` sublayer (which defines `APISchemaBase`) didn't resolve
     until we set `PXR_AR_DEFAULT_SEARCH_PATH` to the install's
     `lib/usd/usd/resources`.
   - The interpreter's importable `pxr` was a **stale pip `usd-core` 23.05** shadowing
     the 26.x install (this host has **no** matching-version Python bindings —
     `USDROOT/lib/python` has no `pxr`; only `lib/site-packages/pxr` does, used by the
     tests). We had to pass `-t <install>/lib/usd/usd/resources/codegenTemplates` so
     the right templates were used regardless of which `pxr` ran the script.
   - This is the same "wrong pxr in the environment" family as the long-standing
     version mis-detection (report #3): nobody composed a correct USD-Python env, so
     the schema generator grabbed whatever `import pxr` found. **This is precisely the
     value `ost` would add** — it already composes the certified runtime env for
     `plugin build`/`test`; a schema step run through that env removes all of the
     above.
3. **Choosing which generated files to keep.** For a co-located plugin with no Python
   module we keep `api.h` / `tokens.{h,cpp}` / `vrmHumanoidAPI.{h,cpp}` and **drop**
   `wrap*.cpp`, `module.cpp`, `generatedSchema.module.h`, `generatedSchema.classes.txt`.
   `usdGenSchema` even warns "`module.cpp` is missing, run usdInitSchema" — noise for
   our case. A template/owned step should know the co-located-no-pymodule shape.
4. **Merging `Types` into the existing `plugInfo.json`** (the ask #2 core) beside the
   `SdfFileFormat` entry, plus staging `generatedSchema.usda` next to it. We scripted
   this in `tools/generate_schema.py`.
5. **`USDVRM_EXPORTS`** had to be defined on the library target or the generated
   `USDVRM_API` macro resolves to `ARCH_IMPORT` and the authorer can't link the typed
   API from the same DLL. A scaffold/CMake-snippet would prevent this.
6. **Test wiring is separate.** The plain-CMake CTest builds its own `plugInfo` from
   `tests/plugInfo.in.json` and points discovery at it — so the schema `Types` had to
   be added **there too**, and `generatedSchema.usda` copied next to that test
   `plugInfo`, or the schema simply doesn't register under test.

## What `tools/generate_schema.py` ended up being

It's the stand-in for the missing `ost` step: run `usdGenSchema` with the right env
(`PXR_AR_DEFAULT_SEARCH_PATH` + `-t templates`), copy the compiled outputs into
`src/schema/`, copy `generatedSchema.usda` into resources, and **merge** the generated
`Types` into the committed `plugInfo.json`. ~120 lines, most of it papering over the
environment problem `ost` already solves elsewhere.

## Concrete asks for `ost` (refines report #4 ask #2)

A first-class **"add a schema to an existing plugin"** flow — e.g.
`ost plugin schema add <bundle>` or schema-awareness in `ost plugin build` — that:

- runs `usdGenSchema` **in the composed runtime env** (matching Python + `pxr`,
  templates, sublayer search path) so none of §2 is the user's problem;
- defaults to the **compiled, co-located** shape (keep C++, drop the Python-module
  helpers), with `--codeless` for the standalone case the current template already
  covers;
- **merges** the generated `Types` into the bundle's existing `plugInfo.json` and
  stages `generatedSchema.usda`;
- emits the `USDVRM_EXPORTS`-style define / CMake snippet so the compiled API links;
- ideally nudges on the `libraryPrefix`/identifier name composition.

Until then, co-located schemas are a hand-roll — doable (this pilot proves it), but
exactly the kind of boilerplate `ost`'s file-format template removed for the importer.

## Addendum — the rest of the schemas, and an encoding footgun

After the pilot we extended `schema/schema.usda` with `VrmExpressionAPI`,
`VrmLookAtAPI`, and (secondary motion has two prim shapes) `VrmSpringBoneAPI` +
`VrmColliderAPI` — five applied schemas in one file, one `generate_schema.py` run,
one merged `plugInfo`. The multi-class path was mechanical, which is the point: once
the env/merge/export plumbing exists, adding a schema is just more `schema.usda`.

One concrete new finding: **`usdGenSchema` writes its generated files in the
process's locale encoding** (here Windows **cp932**), so an em-dash (`—`, U+2014) in
a schema `doc=` string aborted generation with `'cp932' codec can't encode`. ASCII-only
doc strings dodge it. An `ost`-owned schema step should force UTF-8 output (or warn on
non-encodable doc text) — easy to hit on a Japanese-locale Windows host, and the error
points at a codec, not at the offending doc string.
