- `ost --version`: **0.3.0** (same build as report
  [#3](03-2026-06-28-v0.3.0-followup.md))
- Same host; runtime USD v26.08 at `C:/dev/build/usd2605`.
- Trigger: Phase 4 of the roadmap is a **typed schema** (`VrmHumanoidAPI`,
  `VrmExpressionAPI`, `VrmLookAtAPI`, `VrmSecondaryMotionAPI`) that would
  supersede the namespaced-attribute conventions (`vrm:humanBones:*`,
  `vrm:expressionType`, `vrm:joints`, …) the file-format plugin authors today.
  The question this report answers: **should that schema be scaffolded with
  `ost plugin new usd-schema`?**

## TL;DR

Not yet — there is no template. `ost plugin new` advertises `usd-schema` as a
valid `<KIND>`, but invoking it errors out:

```
$ ost plugin new usd-schema vrmSchemaProbe --dir <tmp>
error: no template yet for kind 'usd-schema' (4a ships usd-fileformat; others follow)
```

So the KIND enum is plumbed through the CLI (help text + arg validation) ahead of
the generator.

A second, more open question sits upstream of the missing template: **does the
schema even want its own bundle, or should it live inside the existing file-format
plugin?** There are real arguments both ways (see "Separate bundle vs. co-located"
below); the current lean is co-location, but it's unsettled. Either way it reframes
the `ost` ask to include "let an existing plugin also provide a schema," not just
"scaffold a standalone `usd-schema` bundle." Phase 4 stays deferred regardless (it
was gated on the metadata shape stabilizing); this report just captures the options
and requirements so the choice is ready when we pick it up.

## Why the file-format template doesn't transfer

`ost plugin new usd-fileformat` is excellent (report #1), but a schema bundle is a
different build: there is usually **no hand-written `.cpp`** — the source of truth
is a `schema.usda`, and `usdGenSchema` generates the C++/Python (or, for a
**codeless** schema, nothing but an enriched `plugInfo.json`). The scaffolding
needs are correspondingly different:

- **`schema.usda`** — the authored class definitions (`class "VrmHumanoidAPI"
  (inherits = </APISchemaBase>) { ... }`), with `customData` for the library
  name / prefix / API-schema kind (single-vs-multiple-apply).
- **A generation step** — invoke `usdGenSchema schema.usda .` at build (or commit)
  time. This is the piece the file-format template has no analogue for, and the
  one most worth `ost` owning (it needs the runtime's `usdGenSchema` + Python on
  PATH — exactly the env `ost` already composes for `plugin test`).
- **`plugInfo.json` with a `Types` section** for the generated schema classes
  (codeless schemas live *entirely* in this file under
  `Plugins[].Info.Types` + `"isCodeless": true`), not the file-format plugin's
  single `SdfFileFormat` factory entry.
- **No shared library at all** for a codeless schema — so the L0
  `plugin.shared_library` / `bundle.plug_info.library_path` checks that report
  #1/#3 fought with must become **conditional on the schema being codeless**, or
  doctor will hard-fail a perfectly valid bundle.

## Separate bundle vs. co-located in the file-format plugin

This is a genuine design question with arguments both ways — recorded here as
open, not settled. The one hard fact is permissive: a USD schema registers through
a `plugInfo.json` `Types` section, and nothing in USD says one bundle can't provide
**both** an `SdfFileFormat` *and* schema types from the same `plugInfo` / library
(usdGeom itself ships geometry schemas next to its other types). So "schema = its
own bundle" is a convention, not a requirement, and co-location is at least on the
table.

Arguments **for co-locating** (in the file-format plugin):

- **The importer is the schema's only producer and reference consumer (today).**
  The authorer writes raw `prim.CreateAttribute("vrm:humanBones:hips", …)`. With a
  **compiled** schema in the same library it could call the generated
  `VrmHumanoidAPI::Apply(prim)` / typed setters, keeping the schema definition and
  what the importer writes closer to a single source of truth. A separate bundle
  pushes the importer toward either a cross-bundle build dependency or
  hand-authored attributes that only *happen* to match the schema.
- **One bundle, one build, one `plugInfo`** — `usdGenSchema` output compiles into
  the existing library; the `plugInfo.json` gains a `Types` block beside the
  `SdfFileFormat` entry. No second bundle or `--with` orchestration.

Arguments **for a separate bundle**:

- **Reuse without the importer.** An independent consumer (renderer, rig tool,
  another DCC) that interprets `VrmHumanoidAPI` on an already-imported stage
  shouldn't have to ship/keep cgltf and the `.vrm` reader nearby. No such consumer
  exists yet, but a published schema is the kind of thing other tools build on.
- **Cleaner separation of concerns / independent versioning** of the data contract
  vs. the importer implementation.

Two facts soften the "reuse" argument and keep the door open whichever way we
start:

- USD loads a plugin's **library lazily** — only when one of its types is actually
  instantiated. A consumer that merely needs the schema *registered* (for `HasAPI`,
  `Apply`, fallbacks) pays no importer-load cost from a co-located bundle; the
  file-format `.dll` is dlopened only when a `.vrm` is opened.
- A **codeless** schema (resources only, no generated C++) is the easiest to share
  standalone — but it gives the importer no typed C++ API, so it doesn't deliver
  the producer-side single-source-of-truth benefit. It only fits if we decide the
  importer should keep authoring raw attributes.

Splitting after the fact is mechanical (move the schema sources + the `Types`
block to a new `plugInfo`), so starting co-located doesn't paint us into a corner.
**Current lean:** co-locate a compiled schema, revisit if a real importer-free
consumer appears — but this is open for discussion, not a decision.

## Concrete asks for `ost`

The standalone `usd-schema` template is the obvious gap, but the co-location
option (if we take it) surfaces a second need worth supporting either way —
**adding a schema to an existing plugin**, not only scaffolding a new bundle:

1. A standalone `usd-schema` template: a starter `schema.usda` (one single-apply
   `*API` + the `customData` library block) and an owned `usdGenSchema` build step.
   Make the L0 library checks **codeless-aware** (skip `shared_library` /
   `library_path` and validate the `Types` / `"isCodeless"` block instead), or it
   hard-fails a valid resource-only schema.
2. A way to add a schema to an **existing** bundle: a `schema.usda` + a build step
   that runs `usdGenSchema` and **merges** the generated `Types` into the bundle's
   `plugInfo.json` (which already has the `SdfFileFormat` entry), reusing the
   `plugin build` env that resolves the runtime (it needs `usdGenSchema` + Python on
   PATH — exactly what `ost` already composes).
3. A schema `tests` contract (for either shape): assert the classes register
   (`Usd.SchemaRegistry`), an `*API` applies to a prim, and authored attributes
   round-trip — the schema analogue of the file-format L4 `stage_open`.

## Where this leaves Phase 4

- **Phase 4 stays deferred**, now with a blocker and a leaning (not a decision):
  when we take it up, the current preference is to **add a compiled schema to the
  existing `usdVrm` plugin** rather than a separate `ost plugin new usd-schema`
  bundle — but the bundle-vs-co-located question is open and worth revisiting with
  fresh eyes (and against whatever `ost` ships). Track the "schema in an existing
  plugin" ask against this report either way.
- No change to the shipping plugin meanwhile: the namespaced-attribute conventions
  remain the contract until the typed schema supersedes them.
