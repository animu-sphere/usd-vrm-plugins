- Date: 2026-07-04
- `ost --version`: **0.5.0**
- Host: Windows x86_64, MSVC 14.34, project `cy2026` / `usd`.
- Runtime USD root: `C:/dev/build/usd2605usdview` (**OpenUSD 26.08**).
- Trigger: re-check the findings from the 2026-06-30 reports after upgrading from
  `ost` 0.4.0 to 0.5.0, then align this repo with the new 0.5 shape.

## TL;DR

- **Version detection is effectively fixed for plugin gates.** `ost plugin doctor`
  and `ost plugin test` now read the real `pxr.h` version (`26.08`) instead of
  accepting the stale manifest value (`25.05.01`). `ost runtime validate` now also
  fails on version drift.
- **The local runtime had to be re-adopted.** Before re-pull, v0.5.0 correctly
  reported drift: manifest `25.05.01`, install `26.08`. Re-running
  `ost runtime pull cy2026 --profile usd --from-usd C:/dev/build/usd2605usdview --force`
  refreshed the manifest; `runtime validate`, `doctor`, and `runtime show` are now
  green and record `openusd 26.08`.
- **The workspace naming ask is closed.** `ost init --help` now advertises
  `usd-plugin-workspace`, and `ost plugin new` points to it in its "Next" output.
  The old `plugin-workspace` name still works as an alias.
- **The codeless schema path is healthier.** A fresh `ost plugin new usd-schema`
  bundle passes schema registration / apply-roundtrip tests, and its generated
  CMake runs `usdGenSchema` with `PYTHONUTF8=1` and `PYTHONIOENCODING=utf-8`. An
  em-dash in a schema doc string generated successfully on this Japanese-locale
  Windows host.
- **The compiled, co-located schema ask remains open.** There is still no
  `ost plugin schema add` (or equivalent) that updates an existing file-format
  plugin, runs `usdGenSchema`, merges `Types` into an existing `plugInfo.json`,
  stages `generatedSchema.usda`, and wires the generated C++ into the existing
  library.

## Repo Alignment Done

1. `plugins/usdVrm/openstrata.plugin.yaml` now uses:

   ```yaml
   runtime:
     openusd: ">=25.05,<27.0"
     cxx_abi: inherit
   ```

   This matches the 0.5.0 file-format scaffold guidance. It keeps the source
   bundle portable across Windows (`msvc143`), macOS (`libcxx`), and Linux
   (`libstdcxx`), while letting `ost` resolve the concrete target ABI.

2. `plugins/usdVrm/CMakeLists.txt` now defines `USDVRM_BUILD_TESTS` when the
   plugin is configured directly. This restores the standalone `ost plugin build`
   + `ctest --test-dir plugins/usdVrm/build/...` path that the macOS report
   expected.

3. README text now treats `ost 0.5+` as the baseline and describes the schema
   output as the full `Vrm*API` set rather than only `VrmHumanoidAPI`.

4. `docs/ROADMAP.md` now carries the v0.5.0 status instead of the old 0.4.0 open
   list: version detection, `usd-plugin-workspace`, UTF-8 codeless generation, and
   `cxx_abi: inherit` are no longer forward asks.

## Still Open for v0.6.0

### 1. Compiled schema into an existing bundle

This remains the highest-value `ost` ask for this repo. The current template is
good for standalone, codeless schemas. `usdVrm` needs the other shape:

- schema source lives in `plugins/usdVrm/schema/schema.usda`;
- generated C++ is compiled into `libUsdVrmFileFormat`;
- the existing `plugInfo.json` carries both `SdfFileFormat` and schema `Types`;
- `generatedSchema.usda` is staged beside that same `plugInfo.json`;
- generated Python wrapper files are not needed;
- the library target needs the generated export define (`USDVRM_EXPORTS` here).

Concrete v0.6.0 shape:

```sh
ost plugin schema add plugins/usdVrm schema/schema.usda
# or:
ost plugin build plugins/usdVrm  # schema-aware when manifest declares sources
```

The command should run `usdGenSchema` inside the composed runtime env, force UTF-8,
choose the co-located/no-Python-module generated-file set, merge `Types` into the
existing `plugInfo.json`, copy `generatedSchema.usda`, and emit the CMake/export
snippet needed by the target.

`ost plugin build` already passes an `OPENSTRATA_SCHEMA_SOURCES_FILE` CMake cache
variable, but this repo does not consume it, so CMake warns that it is unused. If
that is intended as the schema hook, it should be documented and made useful for
the compiled co-located path.

### 2. Adopted-runtime drift repair UX

v0.5.0 detects version drift correctly. The remaining UX problem is repair:

- `runtime validate` tells the user to re-pull;
- the exact source path is already known in the runtime manifest;
- repair still requires manually typing `ost runtime pull ... --from-usd <usd-root> --force`.

Suggested v0.6.0 affordance: `ost runtime validate --fix`, `ost runtime repair`, or
an exact copy-paste command that includes the recorded `USD root`.

### 3. `cxx_abi: inherit` artifact semantics

`inherit` is the right source-bundle shape. The next thing to verify is packaging:
a built/package manifest should record the concrete target ABI (`msvc143`,
`libcxx`, `libstdcxx`) so binary artifacts remain self-describing even when source
manifests defer to the runtime.

### 4. macOS OpenUSD build ergonomics

The macOS OpenUSD build report has not been re-run under 0.5.0. These are still
candidate v0.6.0 follow-ups until tested:

- oneTBB 2021.9.0 with newer CMake needing `-DCMAKE_POLICY_VERSION_MINIMUM=3.5`;
- Apple Silicon codesign fallback when only Command Line Tools are installed;
- usdview Python-side dependency hints (`PySide6`, `PyOpenGL`, `Jinja2`) and PATH
  setup for `pyside6-uic`.

## Net

The repo now matches the practical `ost 0.5.0` shape: portable ABI declaration,
green runtime metadata, green plugin build/test, and restored standalone CTest.
The next `ost` product step is not another codeless schema template; it is owning
the compiled schema update path that this repo currently carries as
`tools/generate_schema.py`.
