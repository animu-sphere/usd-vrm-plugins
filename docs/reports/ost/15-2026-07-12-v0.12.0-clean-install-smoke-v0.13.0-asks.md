## TL;DR

The packaging → extract → relocated-run path is **good enough to build a real
clean-install test on top of, with no bespoke env plumbing**. The single most
useful discovery: an **extracted package is a valid relocated bundle**, so
`ost plugin run <extract-dir>` puts *only* the extracted plugin on
`PXR_PLUGINPATH_NAME` — exactly the "don't reference the build tree" property the
clean-install test needs, for free.

Two things stand in the way of a reproducible **release** (not this PR, but PR3):
packaging is **not byte-reproducible**, and the release artifact **ships the
Windows PDB**.

## What worked (keep it)

1. **`ost artifact extract <DIGEST> <DEST>`** unpacks with the archive digest
   re-verified, into any fresh dir. Clean, scriptable, and it refuses a
   mismatched digest.
2. **Extracted package == relocated bundle.** `ost plugin inspect <extract-dir>`
   passes L0 and `session.plugin_path` reports the *extract dir*, and
   `ost plugin run <extract-dir> -- python …` activates the runtime with that
   bundle's plugin path. This is what made a true clean-install test trivial:
   - discovery: `Sdf.FileFormat.FindByExtension("vrm")` resolves, and
     `Plug.Registry()` serves `libUsdVrmFileFormat.dll` from under the extract
     dir (asserted — catches any build-tree/source leak).
   - open: `textures.vrm` (9 prims) + `Seed-san.vrm` (309 prims) from the package.
   - validate: the standalone validator reports `0F/0E/0W/0I` for both.
   - texture resolve: the package resolver serves the embedded PNG straight from
     `…/textures.vrm[images/<hash>.png]` (70 B) with **no temp extraction**.
3. **`--json` is stable and scripting-friendly:** `{ok, schema, warnings,
   data:{archive, archive_digest, archive_size, files, target, packaged}}`.
4. **`ost artifact import <dist-dir>`** is idempotent ("Already in the registry
   as …"), so a package→import→extract driver is safe to re-run.

Net: the whole driver is ~90 lines of Python shelling out to
`ost plugin build|package` + `ost artifact import|extract` + `ost plugin run`.

## Findings / v0.13.0 asks

1. **Packaging is not byte-reproducible.** Two `ost plugin package` runs over an
   *unchanged* build produced different archive digests
   (`sha256:1518387a…` → `sha256:92c7767b…`). This breaks the plan's
   reproducible-release + stable-checksum goal (§7.7/§8) before it starts. Please
   support deterministic packaging: honor `SOURCE_DATE_EPOCH`, normalize entry
   mtimes/ordering/permissions, and make the archive a pure function of the
   staged files. **Highest-value ask for the release lane (PR3).**
2. **The release artifact ships the PDB.** `lib/libUsdVrmFileFormat.pdb`
   (~3.9 MB of the ~15.6 MB archive) is included. For a distributable, debug
   symbols should be opt-out or split into a separate debug artifact
   (`--with-debug` / a sibling `*-debug` package), so the shipped plugin stays
   lean. (Related to #1: the PDB is also a likely non-determinism source.)
3. **No first-class "bare runtime" or external-plugin-path run.**
   `ost plugin run <bundle>` always injects *that* bundle's plugin path with no
   opt-out. Running the **extracted bundle** as the bundle is a clean workaround,
   but a documented primitive would be better for clean-install/discovery
   testing: either `ost runtime run` (USD env only, no plugin) or
   `ost plugin run --plugin-path <dir>` / `--no-inject`, so a harness can point a
   session at an arbitrary installed/extracted plugin tree.
4. **A native packaged-artifact smoke level.** Today L2 `plugin.discovery` runs
   against the *source* bundle only — it cannot catch a build-tree path baked
   into the shipped `plugInfo`/`LibraryPath`. An optional level or
   `ost plugin test --from-package` that extracts the just-built package and runs
   discovery/open against it would make this PR's hand-rolled check a first-class
   `ost` capability. (Complements the prior report's ask #4 about repo-specific
   smoke surviving CI regeneration.)
5. **CLI ergonomics: `artifact extract` positional vs `export`.**
   `ost artifact extract <DIGEST> <DEST>` is positional, but muscle memory from
   `ost artifact export` (and other tools) reached for `--into <DEST>` first,
   which errors. Consider accepting `--into` as an alias, or aligning the two
   verbs' signatures.

## For the repo (not blocking)

- The clean-install smoke lives as `scripts/clean_install_smoke.py` (driver) +
  `plugins/usdVrm/tests/clean_install_smoke.py` (in-session assertions). It is
  script + local-verify for now; wiring it as a CI job waits on PR3 (release
  workflow) and ideally on ask #4 above.

## Net

The consumer path (`package → import → extract → relocated run`) is solid and
already enabled a zero-build-tree clean-install test. The gaps are all on the
**release-quality** side — reproducible archives and lean, symbol-split
artifacts — which is precisely what PR3 will need from `ost 0.13.0`.
