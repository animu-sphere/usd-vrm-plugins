## #4 — the `usd-schema` template now exists (verified)

```
$ ost plugin new usd-schema vrmSchemaProbe --dir <tmp>
Created usd-schema plugin 'vrmSchemaProbe' in <tmp>/vrmSchemaProbe/
  openstrata.plugin.yaml
  CMakeLists.txt
  README.md
  .gitignore
  schema.usda
  plugin/resources/vrmSchemaProbe/plugInfo.json
  plugin/resources/vrmSchemaProbe/generatedSchema.usda
  tests/fixtures/basic.usda
```

What it generates, against report #4's three asks:

- **Ask #1 (standalone template + codeless-aware doctor) — met.** `schema.usda`
  carries the `customData` library block (`libraryName` / `libraryPrefix` /
  `tokensPrefix`) and `skipCodeGeneration = true`, plus one example single-apply
  `*API` class inheriting `</APISchemaBase>`. The manifest declares
  `schema: { codeless: true }`, and its comments confirm doctor validates the
  `Types` block instead of a shared library — exactly the codeless-aware L0 behavior
  #4 asked for (so a resource-only schema no longer hard-fails the `shared_library` /
  `library_path` checks).
- **Ask #3 (schema tests contract) — met.** A `tests/fixtures/basic.usda` is emitted
  with `tests.smoke` wired to it; the manifest comment states the execution levels
  "assert the classes register and the applied attributes round-trip" — the schema
  analogue of the file-format L4 `stage_open`.
- **Ask #2 (add a schema to an *existing* bundle) — not addressed.** The template
  only scaffolds a **new standalone bundle**. There is still no path that runs
  `usdGenSchema` and **merges** the generated `Types` into the existing `usdVrm`
  bundle's `plugInfo.json` (which already carries the `SdfFileFormat` entry). That
  was report #4's second ask and remains open.

### Gap vs. the repo's leaning: codeless default ≠ compiled + co-located

The template defaults to **codeless** (`skipCodeGeneration = true`, no C++/Python,
no shared library). Report #4's *current lean* for this repo is the opposite: a
**compiled** schema **co-located inside `usdVrm`**, so the authorer can call the
generated `VrmHumanoidAPI::Apply(prim)` / typed setters instead of writing raw
`vrm:humanBones:*` attributes (the producer-side single-source-of-truth argument).
The shipped template therefore does **not** match the shape we'd most likely adopt:

- it's a *separate bundle*, not a `Types` block merged into the file-format plugin
  (ask #2), and
- it's *codeless*, so it gives the importer no typed C++ API — the very benefit the
  co-location lean is chasing.

Neither is a defect — codeless-standalone is the reasonable default for a template,
and report #4 itself notes splitting/merging after the fact is mechanical. But it
means **adopting Phase 4 in this repo is no longer blocked on `ost` having *a*
template; it's gated on choosing compiled-co-located vs codeless-standalone**, and —
if we take the lean — on the still-missing ask #2 (merge `Types` into an existing
bundle) or on doing that merge by hand.

## #2 §1 — `plugin-workspace` now ships the dual-mode root

Report #2 §1 hand-rolled a dual-mode root `CMakeLists.txt` + `CMakePresets.json`
because `ost init --bare` + `ost plugin new` left the repo with no top-level CMake,
and suggested `ost init` grow a template that emits it. 0.4.0 did:

```
$ ost init --template plugin-workspace
  template: plugin-workspace
  files: CMakeLists.txt  CMakePresets.json  README.md  .gitignore
```

`ost init --help` describes it as "a dual-mode root for a repo of `ost plugin new`
bundles" — exactly report #2 §1's ask. What it generates vs. our committed root:

| | `plugin-workspace` template | our committed root |
| --- | --- | --- |
| Resolve OpenUSD once (`if(NOT pxr_FOUND)`) | yes | yes |
| Bundle discovery | **auto-glob** of root-level dirs that have `openstrata.plugin.yaml`+`CMakeLists.txt`, `CONFIGURE_DEPENDS` (new bundles picked up automatically) | **explicit** `add_subdirectory(plugins/usdVrm)` |
| Expected bundle location | **immediate child of repo root** | **`plugins/<name>`** (report #2 §1 "1 project → N bundles under `plugins/`") |
| Presets | one `default` Ninja + `CMAKE_BUILD_TYPE=Release` | per-platform Windows-MSVC / Linux-Ninja, `USD_INSTALL_ROOT` env |
| Release-only default | via preset cacheVar | via multi-config guard in CMakeLists |
| Tests / `enable_testing` / `install(FILES)` | none | yes |

So the template is the **official equivalent** of our hand-rolled root and validates
that design — but adopting it verbatim is **not** a drop-in: it auto-globs **root-level**
bundles, while our repo nests the bundle under `plugins/`, so the template's loop
would not find `plugins/usdVrm`. Our root is also richer (platform presets, tests,
install).

**Recommendation:** keep our committed root (don't regress the platform presets /
tests), but consider borrowing the template's one genuine improvement — replace the
hard-coded `add_subdirectory(plugins/usdVrm)` with a `CONFIGURE_DEPENDS` glob scoped
to **`plugins/*`**, so a future second bundle is picked up without editing the root.
Low-risk, and it future-proofs the multi-bundle direction the repo is named for.

### Naming friction worth reporting to `ost`

Finding the feature required guessing the template name — concrete naming friction:

- **"workspace" is a term that exists only in this template's name.** There is no
  `ost workspace` command (`ost workspace --help` → "unrecognized subcommand"), so
  the vocabulary isn't reinforced anywhere; you have to already know the string to
  reach it via `ost init --template plugin-workspace`.
- **The `init --template` choices aren't a parallel set.** `cpp-library` (a language
  kind), `usd-plugin` (a domain kind), and `plugin-workspace` (a repo *shape*) mix
  three different axes, so "I want a repo of USD plugin bundles" doesn't obviously map
  to `plugin-workspace` over `usd-plugin`.
- **"plugin" is overloaded.** It names both an `init --template` (`usd-plugin` /
  `plugin-workspace`) and the `ost plugin` subcommand that *populates* such a repo
  (`ost plugin new`). The relationship "use `init --template plugin-workspace`, then
  `ost plugin new` into it" is not discoverable from the names.
- **`plugin-workspace` drops the `usd-` prefix** that `usd-plugin` / `usd-fileformat`
  / `usd-schema` carry, even though it's specifically for a repo of USD plugin
  bundles — so it doesn't group with them alphabetically or conceptually.

Suggestion for `ost`: align the `init --template` names on one axis and surface the
shape in help text — e.g. `usd-plugin-workspace` (keeps the `usd-` family) with a
one-line "a root for multiple `ost plugin new` bundles", and a forward-pointer from
`ost plugin new` ("create the repo root first with `ost init --template …`").

## #3 — version mis-detection still open in 0.4.0

```
$ ost runtime show openstrata-cy2026-windows-x86_64-py313-usd
  ...
  Extensions:
    - openusd 25.05.01 [core]

$ ost plugin doctor plugins/usdVrm
  [PASS] L1 runtime.openusd.version  runtime OpenUSD 25.05.01 satisfies '>=25.05,<27.0'
  [PASS] L1 runtime.cxx_abi          C++ ABI msvc143 matches runtime
```

Unchanged from #1–#4: the reported `25.05.01` is the last placeholder/hard-coded
field (ABI/variant detection is correct — `msvc143`, `py313`). The range check
"passes" for the wrong reason. The bundle's `openusd` range is already widened to
`>=25.05,<27.0`, so this isn't currently masking a failure here, but the gate is
still not enforcing a real version. Worth deriving from `pxr/pxr.h` `PXR_VERSION`.
