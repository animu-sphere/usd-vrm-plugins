## Executive summary

v0.15.0 delivered the previous top ask: `requires.bundles` is now an
executable dependency closure for source build, test, run, and generated CI.
The real `usdVrm -> vrmSchema` edge builds in topological order and the
consumer passes L5 without repo-owned schema bootstrap code.

Phase 2 exposes the next three seams:

1. Windows L5 capture creates semantic `\r` inside multiline USDA strings.
2. `requires.libraries` correctly fails closed but cannot yet describe, build,
   discover, or materialize a plain CMake library such as `vrmContainer`.
3. Source dependency closure does not become package/product closure;
   `--from-package` still needs an explicit source `--with` for `vrmSchema`.

The first is a regression that currently weakens one hosted gate. The second
is the direct v0.16.0 feature ask. The third may remain product-workspace scope,
but v0.16.0 should at least make the intended ownership and diagnostic clear.

## P0 — stop Windows L5 transport from changing authored values

### Evidence

`ost plugin test plugins/vrmSchema --up-to 5` fails only at L5 on Windows.
Both files are LF-pinned:

```text
i/lf w/lf  plugins/vrmSchema/tests/fixtures/basic.usda
i/lf w/lf  plugins/vrmSchema/tests/fixtures/basic.usda.golden.usda
```

Opening the fixture and golden directly with OpenUSD returns identical layer
comments containing `\n`. The L5 actual produced by ost instead contains
`\r` inside the multiline string. Making that authored comment single-line
only moves the first difference to OpenUSD's generated multiline `doc`, which
rules out checkout EOL as the cause and points to flatten stdout/file capture.

The new v0.15.0 bounded diff and CRLF hint made the difference easy to find,
but the suggested checkout fix is false for this case: the checkout is already
LF and the CR is created later.

### v0.16.0 acceptance criteria

- The committed fixture and golden pass L5 on Windows, macOS, and Linux.
- Capturing `usdcat --flatten` does not translate newline bytes into authored
  USDA string content.
- A genuinely authored `\r` inside a string remains significant; the fix must
  be at process/file transport, not a blanket semantic normalization.
- The semantic-CRLF hint distinguishes a CR already present in the input from
  one introduced while capturing the flatten result.
- Regression coverage includes both an authored multiline layer comment and
  OpenUSD's generated multiline `doc` value.

Current repo mitigation: only the Windows `vrmSchema` source-CI cell is capped
at L4. All importer cells and the macOS/Linux schema cells remain L5.

## P1 — make `requires.libraries` executable

### Concrete producer/consumer supplied by Phase 2

`libs/vrmContainer` is a deliberately plain C++ shared library:

- no `openstrata.plugin.yaml`, `plugInfo.json`, pxr include, registration
  macro, or OpenUSD link dependency;
- installed package identity `vrmContainer` with exported target
  `vrmContainer::vrmContainer`;
- version synchronized with the repository;
- one DLL/SO/dylib that the consumer needs at runtime;
- unit tests plus source and binary boundary checks.

`plugins/usdVrm` needs the library at configure, link, source-session, and
package-session time. Under v0.15.0 the repo must configure/build/install the
sibling into a private prefix, call
`find_package(vrmContainer CONFIG REQUIRED)`, copy the runtime into
`runtime/lib`, and declare that loader directory separately through
`requires.runtime_libs`.

### Suggested manifest shape

The exact schema is ost-owned, but this is the dependency the repo needs to
express:

```yaml
requires:
  libraries:
    - id: vrmContainer
      version: ">=0.1,<0.2"
```

The plain library itself should not need to impersonate a USD plugin bundle.
A small library/package descriptor beside its CMake project is preferable to
adding plugin registration concepts to it.

### v0.16.0 acceptance criteria

- Manifest validation resolves a library identity and compatible version and
  rejects missing, duplicate, incompatible, and cyclic edges with named fixes.
- `ost plugin build plugins/usdVrm` builds/installs the library before the
  consumer and adds only its installed prefix to CMake package discovery.
- The consumer continues to use
  `find_package(vrmContainer CONFIG REQUIRED)`; ost does not require sibling
  `add_subdirectory()` coupling.
- Source `plugin test`/`plugin run` materializes the selected library's runtime
  directory in the loader environment on Windows, macOS, and Linux.
- Generated CI consumes the same closure without cell-local duplication.
- Package behavior is explicit: either include the declared library runtime or
  record a resolvable dependency. A successful source session followed by an
  unloadable package must not be the default.
- `ost plugin inspect --json` and the test report expose the resolved library
  id, version, prefix/artifact provenance, and runtime materialization.

Once these hold, the repo can delete its entire `vrmContainer` bootstrap and
runtime-copy adapter rather than merely hiding it behind new glue.

## P2 — define package/product dependency closure

### Evidence

The source consumer is now excellent:

```text
ost plugin test plugins/usdVrm --up-to 5
13 pass / 0 fail / 2 skip
```

The lean importer archive contains `runtime/lib/vrmContainer.dll`, and this
passes:

```text
ost plugin test plugins/usdVrm --from-package \
  --with plugins/vrmSchema --up-to 5
14 pass / 0 fail / 1 skip
```

Without `--with`, the packaged importer cannot load its declared bundle
dependency. The explicit operand is a source checkout, not a digest-pinned
installable product closure.

### v0.16.0 acceptance criteria or explicit deferral

- Prefer: package/test/publish records and resolves the transitive
  `requires.bundles` and `requires.libraries` closure by identity, compatible
  version, target, and digest.
- Acceptable deferral: fail before L2 with a diagnostic naming the missing
  packaged dependency and point to the product descriptor command/contract
  that owns aggregate installation.
- `--from-package` should never silently fall back to a mutable sibling source
  tree when claiming clean-install validation.

This aligns with workspace Phase 5's digest-pinned product descriptor; ost and
the repo should share one closure model rather than invent parallel metadata.

## Smaller carried asks

1. Generator-owned reusable bootstrap/runtime-pull workflow fragment.
2. A coherent `ost ci pin bootstrap --version <version>` update path for the
   CI source and generated workflows.
3. Keep stale-PDB cleanup failures diagnostic/non-fatal when the final plugin
   artifact itself is valid.

These are useful, but none should displace the Windows L5 regression or the
plain-library closure.

## Regression contract to preserve from v0.15.0

- `requires.bundles` build order and workspace-prefix discovery.
- Automatic dependency registration and loader paths in test/run sessions.
- Generated CI using the same dependency closure as the CLI.
- Strict rejection of reserved/unknown `requires` fields.
- The template fixes for MSVC `NOMINMAX`, guarded pxr discovery, and no stale
  schema source fragment on non-schema consumers.
- Bounded first-difference output at L5.
