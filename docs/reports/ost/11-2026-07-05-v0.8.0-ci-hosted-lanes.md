## v0.8.0 findings

- `ost runtime export` works but is heavy on adopted local runtimes: this
  runtime's external prefix (`C:/dev/build/usd2605usdview`) is a full USD build
  tree — **14.4 GB / 18,029 files** — and the export took **~52 minutes**
  wall-clock, CPU-bound single-threaded, with **zero progress output** the
  whole time (the staged `tar.zst` even reports 0 bytes until the handle
  closes, so it looks hung from the outside).
- The generated source-CI workflow is honest about its contract, and that
  contract exposes the next gap (below): steps assume `ost` on PATH and the
  pinned runtime artifact **already present in the runner's local registry**;
  it verifies + `ost runtime pull --from-artifact`, builds from checkout, runs
  the pyramid, packages but never publishes. Fork-PR safety notes are rendered
  into the file. Action pins are full SHAs. Good renderer hygiene overall.
- `ost ci generate github` refuses to overwrite an existing workflow without
  `--force` — correct default, clear message.

## The hosted bootstrap gap (main ask for v0.9.0)

The PR/source lane is now rendered but **cannot actually run on a GitHub-hosted
runner yet**, because the renderer leaves two acquisition problems to the
operator, and today neither has a supported answer:

1. **Installing `ost` on a hosted runner.** There is no rendered setup step, no
   `setup-ost` action, and no public release channel to fetch a pinned `ost`
   binary from.
2. **Seeding the 1.93 GiB runtime artifact into the runner's registry.** The
   registry is local-only; the workflow's "`ost artifact import` from a
   handoff" comment presumes the operator invents the transport.

Asks, in priority order:

- A rendered bootstrap step (or official action) that installs a pinned `ost`
  and imports the pinned runtime artifact from a URL/OCI reference recorded in
  `openstrata.ci.yaml` — e.g. `runtime_artifact_source: ghcr.io/...@sha256:...`.
  OCI registries are content-addressed, so digest verification composes
  naturally (our plan: GHCR via `oras`, `actions/cache` keyed by digest).
- A **slim/SDK-profile export**: full adopted build trees carry test/example/
  symbol payload a source build doesn't need; `include/lib/bin/plugin`-only
  export would cut both the 52-minute export and the per-PR download.
- Multithreaded zstd and/or a `--level` knob, plus progress output for
  `runtime export`.
- Clarify the golden-file naming in the L5 skip message
  (`<fixture-filename>.golden.usda`, extension included).

## Net

v0.7.0's blocker — "no real digests" — is gone: 0.8.0 produced both artifacts,
the matrix validates with `--resolve`, and both lanes render cleanly from
`openstrata.ci.yaml`. The support lane is runnable as soon as a matching
self-hosted runner is registered and seeded. The new blocker is narrower and
well-defined: hosted-lane bootstrap (ost install + artifact transport). Until
that lands in `ost` or we hand-roll a pre-step, the generated
`ost-source-ci.yml` will fail at "Check ost is available" on GitHub-hosted
runners and must stay branch-protection-optional.
