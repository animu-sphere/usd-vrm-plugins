## TL;DR

The top v0.16.0 ask is **delivered**: `requires.libraries` is executable.
A plain library carries an `openstrata.library.yaml` descriptor, the workspace
graph validates the `bundle -> library` edges, and `build`/`test`/`run`/`package`
compose the library into the prefix, session, and lean archive automatically.
The entire repo-owned `vrmContainer` bootstrap-and-runtime-copy adapter from the
0.15 era is gone (−74 lines): CMake is back to a plain
`find_package(vrmContainer CONFIG REQUIRED)`.

Phase 3 then split the resolver cleanly on top of that. Three bundles
(`vrmSchema`, `usdVrm`, `usdVrmPackageResolver`) + one library (`vrmContainer`)
now validate as one graph, each tests standalone through L5, the Phase 0
baseline stays byte-identical, and the composed clean-install resolves embedded
textures from three shipped packages.

One new seam surfaced, and it is the headline finding: **`ost plugin view` and
`ost plugin test-view` do not compose the `requires.bundles` closure** — only
`--with` bundles — so a bare `ost plugin view plugins/usdVrm <avatar>` opens
without `vrmSchema` and fails inside `Sdf.Layer.FindOrOpen`.

## What v0.16 delivered in real use: `requires.libraries`

### Descriptor + graph

`libs/vrmContainer/openstrata.library.yaml`:

```yaml
schema: openstrata.library/v1alpha1
library:
  id: vrmContainer
  version: 0.1.0
cmake:
  package: vrmContainer
  target: vrmContainer::vrmContainer
```

`ost plugin test --workspace --up-to 1` reports the new edges:

```text
nodes: usdVrm (usd-fileformat), usdVrmPackageResolver (usd-package-resolver),
       vrmSchema (usd-schema)
bundle edges:  usdVrm -> vrmSchema (contract 1)
library edges: usdVrm -> vrmContainer,  usdVrmPackageResolver -> vrmContainer
libraries:     vrmContainer   issues: []   passed: true
```

Before the descriptor existed, the same manifest edge failed closed exactly as
hoped: `WORKSPACE_LIBRARY_DEPENDENCY_MISSING: bundle 'usdVrm' requires missing
library 'vrmContainer'`.

### Build / session / package

`ost plugin build plugins/usdVrm --dry-run`:

```text
Workspace composition: 1 bundle dependency, 1 library -> .../workspace-prefix
== build library vrmContainer ==     (configure + build + install into prefix)
== build dependency vrmSchema ==     (configure + build + install into prefix)
== build primary usdVrm ==           (configures against the prefix)
```

A clean `ost plugin build` + `ost plugin test plugins/usdVrm --up-to 5` passes
L0-L5 with **no** repo bootstrap: `find_package(vrmContainer CONFIG REQUIRED)`
resolves against the ost-managed prefix, and the library's DLL directory is on
the session loader path automatically (the old repo POST_BUILD copy into
`runtime/lib` + `requires.runtime_libs` is deleted). `ost plugin package` stages
the closure under `runtime/libraries/{bin,lib}` with a `dependencies.json` that
records `id`, `version`, `cmake_package`, `cmake_target`, `provenance`, and the
runtime directories — the `inspect --json`/report provenance the v0.16 ask
requested.

### Acceptance-criteria scorecard (v0.16 P1)

- Manifest validation resolves library id + version, rejects missing/cyclic — **yes**
- build installs the library before consumers, prefix-only discovery — **yes**
- consumer keeps `find_package(... CONFIG REQUIRED)`, no sibling add_subdirectory — **yes**
- source session materializes the library loader dir (Windows verified) — **yes**
- package records/includes the runtime + a resolvable dependency record — **yes**
- descriptor/provenance exposed in `dependencies.json` — **yes**

Only gap remaining from that list: `--from-package` on the importer alone still

## Phase 3 result

`usdVrmPackageResolver` owns the `ArPackageResolver` registration and the
`avatar.vrm[images/<hash>.<ext>]` byte-serving path; it depends only on
`vrmContainer` + OpenUSD `Ar`. `tests/check_boundaries.py` asserts, on the built
DLL, that it imports `vrmContainer` and imports neither `UsdVrmFileFormat` nor
`vrmSchema`, and on the source that it never authors a stage. The importer keeps
authoring the same package paths but no longer registers or links the resolver
— it is a runtime bundle dependency, absent unless a session loads it.

## New findings for the next ost

### P0 — `plugin view` / `test-view` ignore the `requires.bundles` closure

`ost plugin view` and `ost plugin test-view` build their session from
`load_with_bundles(--with)` **only**; they never call the
`merge_composed_bundles(bundle, graph_dependencies, explicit)` path that
`build`/`test`/`run`/`package` use. So a consumer's declared dependencies are
silent in a view session.

Repro (real user hit this):

```text
ost plugin view plugins/usdVrm C:\...\AliciaSolid.vrm
  -> Sdf.Layer.FindOrOpen(...) raises Tf.ErrorException (schema apply with
     vrmSchema absent), then on a Japanese (cp932) Windows usdview masks it:
     UnicodeDecodeError: 'utf-8' codec can't decode byte 0x8e ...
```

The same avatar opens fine under `ost plugin run plugins/usdVrm -- python -c
"Usd.Stage.Open(...)"` (279 prims) because `run` **does** compose `vrmSchema`.
Workaround is explicit: `--with plugins/vrmSchema --with
plugins/usdVrmPackageResolver`.

Acceptance criteria:
- `plugin view` / `test-view` compose the validated `requires.bundles` (and
  `requires.libraries`) closure by default, exactly like `run`/`test`, with
  `--with` still adding extras.
- Failing to open a fixture surfaces the underlying `Tf` error, not a secondary
  `UnicodeDecodeError`: decode tool stderr/commentary as the OS locale (or
  lossily), never assume UTF-8. This is a general robustness fix — any non-UTF-8
  locale masks every usdview error today.

### P1 — carry the Windows L5 flatten-CR fix forward / confirm it

The repo keeps Windows PR cells at L4. The hosted-Windows `usdcat --flatten
--out` CR-into-multiline-string behavior from the Phase 2 report is not
reproducible on a local worktree (local L5 passes), so this split cannot verify
whether 0.16/0.17 fixed it. Ask: state explicitly in the release notes whether
the hosted-Windows capture path changed, so the repo can restore L5 with
evidence rather than guessing. (`normalize()` correctly preserves semantic CRLF,
so the fix must be in capture/transport, not comparison.)

### P2 — document the `openstrata.library.yaml` schema

The descriptor shape had to be reverse-engineered field-by-field from
`PARSE_FAILED` messages (`unknown field manifest`; `expected schema|library|
cmake|requires|runtime`; `library: expected id|version`; `missing field cmake`).
Ask: a `ost plugin new usd-library` (or `ost library new`) scaffold and/or a
documented schema, the way plugin kinds have templates.

### P3 (environmental) — release binary triggers Defender ML heuristic

The optimized-release `ost.exe` (0.16.0 and a fresh 0.17.0 release build) is
quarantined on Windows 11 by Defender as `Trojan:Win64/jscramSteal.DA!MTB`; a
`dev`-profile build of the identical source is not flagged. Not an ost bug, but
a shipped-binary reputation/signing concern worth noting for the release lane,
since it silently breaks `cargo install`/PATH usage without admin to restore.

## Regression contract preserved from v0.15/0.16

- `requires.bundles` build order + workspace-prefix discovery + session
  registration (now joined by `requires.libraries`).
- Generated CI uses the same closure as the CLI (no cell-local `with:`).
- Strict rejection of unknown `requires`/descriptor fields.
- Bounded first-difference + CRLF hint at L5.
- L5 golden uses the first **smoke** fixture (`smoke_fixture`), not `roundtrip`
  — so the resolver puts a host-independent `basic.usda` first and keeps the
  asset-path-bearing `embedded_texture.usda` second.
