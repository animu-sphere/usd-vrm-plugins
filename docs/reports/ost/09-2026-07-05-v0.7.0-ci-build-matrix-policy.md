# OST v0.7.0 CI build matrix policy notes

- Date: 2026-07-05
- Source policy read: the upstream OpenStrata CI build-matrix / GitHub Actions
  policy (an open-strata working document, not published in this repo)
- Context: `usd-vrm-plugins` PR #18 added a repo-local self-hosted source CI
  workflow. The GitHub Actions run queued because the repository has no matching
  self-hosted runner registered. This exposed the distinction the policy document
  makes: GitHub-hosted SDK builds and self-hosted real-host validation are
  different CI lanes, not interchangeable `runs-on` labels.

## TL;DR

For `ost` v0.7.0, CI should move from generated ad hoc GitHub Actions jobs toward
a portable OpenStrata CI contract:

```text
openstrata.ci.yaml + named runner profiles + digest-pinned artifacts + ost ci
```

GitHub Actions should be the first renderer/executor of that contract, not the
source of CI semantics. A plugin project should be able to run a normal SDK build
on a GitHub-hosted runner, then run real compatibility validation on selected
self-hosted runners.

## Answer to the Runner Question

There are GitHub-hosted Windows runners, but they are not `self-hosted` runners.
The current PR workflow requests:

```yaml
runs-on: [self-hosted, Windows, X64]
```

That means GitHub waits for a repository/org self-hosted runner with those labels.
The repository runner API returned zero repo-level runners, so the job remains
queued. A GitHub-hosted runner would be requested with a fixed image such as a
Windows hosted profile, not with `self-hosted`.

The v0.7.0 policy should make this explicit:

- PR/source build lane: use a `github-hosted` runner profile when possible.
- Real OpenUSD/DCC/GPU validation lane: use a `self-hosted` runner profile.
- Workflow authors should not write raw `runs-on` labels as the CI contract.

## Core Model

### 1. Support Cells, Not Cartesian Products

Each CI cell is an intentional support claim:

```yaml
cells:
  - name: plugin-static-windows
    lane: pull_request
    runner: windows-hosted
    runtime_artifact: sha256:...
    target: cy2026-windows-x86_64-py313-usd
    profile: usd
    verification_level: l3
    publish: never
```

`ost ci` should never infer a broad OS x Python x compiler x DCC product unless
the project author explicitly declares those cells.

### 2. Runner Profiles

Cells reference named profiles:

```yaml
runners:
  windows-hosted:
    kind: github-hosted
    image: windows-2022
    cost_class: metered
    billing:
      acknowledgement: required
    capabilities:
      - os:windows
      - arch:x86_64
      - compiler:msvc

  usd-windows-real:
    kind: self-hosted
    labels: [self-hosted, windows, x64, usd-26]
    cost_class: operator-managed
    trust: verified
    capabilities:
      - os:windows
      - arch:x86_64
      - usd:26
      - python:3.13
```

The GitHub renderer maps these profiles to `runs-on`:

- `github-hosted.image` -> `runs-on: windows-2022`
- `self-hosted.labels` -> `runs-on: [self-hosted, windows, x64, usd-26]`

User-provided arbitrary labels should not be accepted through
`workflow_dispatch`.

### 3. Digest-Pinned Inputs

Runtime and plugin artifacts must be identified by full SHA-256 digests.
Human-readable OpenUSD versions, platform names, and plugin versions are metadata,
not identity.

The workflow should verify every artifact before extracting or executing it.

### 4. CI Lanes

`ost ci` should model lanes directly:

- `pull_request`: safe source build, static validation, lower verification,
  package creation without publishing.
- `main`: all PR checks plus candidate package upload/attestation where policy
  permits.
- `scheduled`: digest-pinned real compatibility validation on selected
  self-hosted runners.
- `workflow_dispatch`: approved choices only; no arbitrary runner labels, shell
  commands, artifact URLs, or script paths.

## v0.7.0 Product Asks

### 1. Extend `openstrata.ci.yaml`

Add first-class `runners`, `cells`, and `lanes` concepts:

```yaml
version: 1

runners:
  windows-hosted:
    kind: github-hosted
    image: windows-2022
    billing:
      acknowledgement: required

cells:
  - name: usdvrm-pr-windows
    lane: pull_request
    runner: windows-hosted
    runtime_artifact: sha256:...
    target: cy2026-windows-x86_64-py313-usd
    profile: usd
    verification_level: l4
    publish: never
```

This turns the current hard-coded workflow into a renderer output, not a repo
policy.

### 2. Generate GitHub-Hosted SDK Build Jobs

The first high-value v0.7.0 workflow should support:

```text
checkout
install/restore ost
ost ci validate
ost artifact verify <runtime digest>
ost runtime pull ... --from-artifact <runtime digest>
ost plugin build
ost plugin test --up-to <level>
ost plugin package
upload reports/package
```

This lets normal plugin PRs run without a self-hosted runner, provided a
self-contained runtime SDK artifact is available.

### 3. Keep Self-Hosted for Real Compatibility

Self-hosted runners should remain for:

- real OpenUSD install loading when not represented by an SDK artifact;
- DCC hosts such as Maya, Houdini, Nuke, Katana;
- GPU/Hydra/render-delegate validation;
- proprietary licenses, private networks, or persistent large caches.

The scheduled lane should be the first place to use these privileged runners.

### 4. Add Hosted Runner Cost Visibility

`ost ci validate` should warn when a `github-hosted` profile is used:

```text
warning[OSTCI0201]: GitHub-hosted runner may incur billable usage
  runner: windows-hosted
  image: windows-2022
  next_action: Confirm billing acknowledgement and review GitHub Actions usage settings.
```

For publish-capable cells, missing acknowledgement should be an error.

Generated workflows should emit a GitHub Actions notice before hosted-runner work
starts.

### 5. Add `ost ci plan`

`ost ci plan --json` should expose cost/trust/execution facts without estimating
money:

```json
{
  "hosted_jobs": 1,
  "operator_managed_runner_classes": ["usd-windows-real"],
  "requires_billing_acknowledgement": true,
  "publish_capable_jobs": 0
}
```

This gives users a preflight view before generating workflows.

### 6. Enforce Fork PR Safety

Generated PR workflows must not:

- publish artifacts;
- promote artifacts;
- expose secrets;
- target privileged self-hosted runners;
- accept arbitrary user-supplied `runs-on`, command, script, or artifact URL
  inputs.

Fork PRs should be build-only unless a maintainer explicitly re-runs a protected
lane.

### 7. Fix the Existing GitHub Renderer Issues

Report #8 already found:

- `ost ci generate github` emits `steps:` at the wrong indentation level.
- starter placeholder digests pass syntax validation too quietly.
- artifact-matrix generation currently models support validation, not normal PR
  source build.

The runner-profile model should address these while keeping the generated YAML
small and deterministic.

### 8. Preserve CI Evidence

Reports should record:

- runner profile and resolved backend runner;
- runtime digest;
- plugin digest when present;
- target/profile;
- verification level;
- validation result;
- package provenance;
- whether the lane was PR/main/scheduled/manual.

## Implications for `usd-vrm-plugins`

The current `.github/workflows/ost-source-ci.yml` is a useful bridge, but it is
not the desired long-term v0.7.0 shape.

Near-term:

- keep PR #18 as a self-hosted source CI experiment;
- expect it to queue until a matching runner is registered or the `runs-on`
  labels are changed;
- keep the dogfooding report of the queue behavior as v0.7.0 input.

v0.7.0 target:

- declare `windows-hosted` and `usd-windows-real` runner profiles in
  `openstrata.ci.yaml`;
- run PR SDK builds on `windows-hosted` using a digest-pinned runtime SDK
  artifact;
- run scheduled real OpenUSD validation on `usd-windows-real`;
- generate the GitHub workflow from `ost ci generate github`;
- remove hand-written `runs-on` policy from the repo workflow.

## Acceptance Criteria for v0.7.0

The v0.7.0 CI work is ready when:

- a plugin PR can build on a GitHub-hosted runner using a digest-pinned runtime
  SDK artifact;
- `ost ci validate` catches missing hosted-runner billing acknowledgement;
- generated workflows emit hosted-runner billing notices;
- CI cells reference runner profiles, not raw labels;
- fork PRs cannot publish or reach privileged self-hosted runners;
- a scheduled self-hosted cell can revalidate a pinned runtime/plugin pair;
- generated reports preserve digests, runner profile, target, lane, and
  verification outcome;
- `ost ci generate github` output is valid YAML and deterministic.

## Net

The runner queue in PR #18 is not a failure of CI trigger wiring. It is evidence
that hard-coded self-hosted labels are too low-level for the OpenStrata CI
contract. v0.7.0 should make GitHub-hosted SDK builds first-class, keep
self-hosted runners for real compatibility, and let `openstrata.ci.yaml` express
that split portably.
