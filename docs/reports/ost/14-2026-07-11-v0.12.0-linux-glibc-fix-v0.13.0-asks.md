v0.10.0 report's headline blocker — a Linux runtime that could not load on
the github-hosted `ubuntu-24.04` runner — is **resolved**, and it was resolved
by two independent things landing at once:

1. **Producer discipline:** rebuild the `cy2026/usd` Linux runtime **inside a
   `ubuntu:24.04` container (glibc 2.39)** instead of on bleeding-edge
   WSL/Ubuntu-26.04 (glibc 2.43). The runtime's real glibc floor is now **2.38**.
2. **`ost 0.12.0` now measures and records the real glibc floor** from the
   exported ELF binaries and stamps the honest target
   `linux-x86_64-glibc238-py313` — no longer a fabricated `glibc228` label. This
   is exactly v0.11.0 ask #7.

Three v0.11.0 asks are confirmed fixed in 0.12.0: **#7 (glibc floor)**, **#2
(Linux symlink export)**, and **#1 (`ost artifact push` to GHCR)**. The Linux
runtime is published to GHCR and the hosted Linux PR lane is re-added
(`usdvrm-pr-linux`, PR #26).

## Build environment (portable Linux runtime recipe)

The Linux runtime must be built against an old-enough glibc. Rather than fight
the host, the whole toolchain is pinned in a `Dockerfile` based on
`ubuntu:24.04`, which matches the hosted PR runner exactly:

- `ubuntu:24.04` → glibc 2.39 (hosted `ubuntu-24.04` is also 2.39).
- Python 3.13 from the **deadsnakes PPA** (`python3.13`, `-dev`, `-venv`) — no
  source build needed this time; `python3.13-dev` provides
  `libpython3.13.so.1.0` for embedding. A venv at `/opt/py313` (prepended to
  PATH) makes `python`, `python3`, and `python3.13` all resolve to 3.13 with
  `Jinja2` + `PyOpenGL` + `PySide6`, so `ost`/`build_usd.py` pick the right
  interpreter regardless of which name they invoke.
- `libxt-dev` (and the usual X/GL dev libs) installed up front — the v0.10.0
  MaterialX `Xt was not found` failure did not recur.
- `oneTBB` 504 issue from the v0.10.0 report **did not recur** this run.
- `ost` musl static binary + `oras` installed into the image.

The runtime store (`/root/.ost`) lives in a **named docker volume** so each
`ost` step is a separate, robust `docker run` (a `sleep infinity` + `exec`
container died with exit 255 mid-provisioning; the image + volume pattern is
much more reliable in WSL).

## Recheck of v0.11.0 asks

### 1. `ost artifact push` against GHCR — FIXED

`OST_REGISTRY_USER=snkmcb` + `OST_REGISTRY_PASSWORD=<PAT>` now works
(`auth_mode: token-exchange-basic`). The v0.10.0 double-`sha256:sha256:` upload
bug is gone. Result:

```json
{ "status": "pushed",
  "artifact_digest": "sha256:f6a2047c…",
  "oci_digest": "sha256:d202c6d7…" }
```

Note: `ost artifact push` produces a **different OCI manifest digest** than
`oras push` of the same dist (`d202c6d7…` vs `390ed698…`) — different manifest
layering/annotations for the identical archive. Both round-trip through
`ost artifact pull` with all pins passing. We standardized the CI pin on the
**ost-native** `d202c6d7…` (it holds the tag, and the producer path is now
first-class). Minor doc-worthy asymmetry, not a bug.

### 2. Linux symlink export — FIXED

`ost runtime export --slim` exported the build-source Linux runtime **directly**,
with its 48 shared-library symlink chains intact. No `cp -aL` re-adopt
workaround was needed (unlike v0.10.0). Export result: `files: 3948`,
`archive_size: 124,167,020`, `glibc_floor: "2.38"`,
`target: linux-x86_64-glibc238-py313`.

### 3. Document Linux build prerequisites — ADDRESSED locally

Captured as the `Dockerfile` recipe above (deadsnakes py3.13, libxt-dev, X/GL
dev libs, venv with Jinja2/PyOpenGL/PySide6). Still worth folding into upstream
`ost` docs.

### 4. package-stage fallback — NOT EXERCISED

No `ost plugin package` in this run (runtime-only work).

### 5. Corpus CTest smoke in generated CI — STILL OPEN

Regenerating `ost-source-ci.yml` with 0.12.0 still does not carry a repo-specific
corpus smoke step; the generated lane runs the `ost plugin test` pyramid only.
Re-raised below.

### 6. Build-host dep preflight — NOT EXERCISED

The container's venv already has the build deps, so a pristine-Python preflight
could not be reproduced.

### 7. Measure and record the real glibc floor — FIXED (the blocker)

`ost runtime export` now emits `glibc_floor: "2.38"` and stamps
`target: linux-x86_64-glibc238-py313`. Independently confirmed with
`readelf -V`: the max referenced symbol across every `.so`/bin is `GLIBC_2.38`,
and the previously-failing `libusd_gf.so` maxes at `GLIBC_2.38`. `ost artifact
pull --require-target linux-x86_64-glibc238-py313` now means what it says.

## New v0.13.0 asks

1. **Make the portable-Linux build base a documented, first-class recipe.**
   The `glibc228`-vs-real-floor trap is easy to fall into. `ost` docs should say
   plainly: build Linux runtimes in a container whose glibc ≤ the lowest target
   runner (e.g. `ubuntu:24.04` for hosted `ubuntu-24.04`, or manylinux_2_28 for
   broader reach), and note the deadsnakes-py3.13 + venv + libxt-dev
   prerequisites. Consider shipping a reference Dockerfile.
2. **Reconcile `ost artifact push` vs `oras push` OCI manifests.** The same dist
   pushed by each tool yields different OCI digests. Document the canonical
   producer path (prefer `ost artifact push`) so CI pins don't drift, or make
   `ost artifact push` reproduce the `oras` manifest byte-for-byte.
3. **Warn when `--require-target` is weaker than the host.** Now that the floor
   is measured, `ost` could go one step further at pull time: if the artifact's
   `glibcNNN` floor exceeds the *current host's* glibc, fail (or loudly warn)
   even without `--require-target`, catching ABI mismatches before the first
   `dlopen`.
4. **Keep repo-specific smoke tests across regeneration (carried from #5).**
   `ost ci generate` still drops the corpus CTest smoke step. Support declarative
   post-build commands in `openstrata.ci.yaml`, or have `ost plugin test` cover
   the corpus assets.
5. **Optionally surface the measured floor in `runtime show`/`validate`.** The
   floor is computed at export; exposing it earlier (at build/validate) would let
   producers catch a too-high floor before publishing.

## Net

v0.12.0 turns the Linux producer path from "works but lies about its ABI" into
"works and tells the truth about its ABI." Building in a `ubuntu:24.04` container
plus the new measured-floor labeling closes the v0.11.0 blocker, and the direct
symlink export + working `ost artifact push` remove the two remaining producer
escape hatches. All three OpenStrata PR lanes (Windows, macOS, Linux) now consume
digest-pinned public GHCR runtimes with honest target labels.
