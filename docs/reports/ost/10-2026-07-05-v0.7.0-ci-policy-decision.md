## Decision

Treat `openstrata.ci.yaml` as the CI contract and generated GitHub Actions as
renderer output. Do not encode long-term CI policy directly as handwritten
`runs-on` labels.

The repo should use two explicit lanes:

1. **PR/source lane:** GitHub-hosted SDK build, from checkout, against a
   digest-pinned runtime SDK artifact.
2. **Scheduled/support lane:** self-hosted real compatibility validation, from
   pinned runtime/plugin artifacts.

This matches roadmap Phase A: prove compatibility continuously before expanding
schema/documentation work further.

## Repo Policy

- Keep the repo shape as a `usd-plugin-workspace`.
- Keep the VRM typed schema co-located in the `usdVrm` file-format bundle.
- Add a root `openstrata.ci.yaml` only after real artifact digests are available;
  placeholder digests should not be committed.
- Prefer runner profiles:
  - `windows-hosted`: `github-hosted`, `windows-2022`, PR/source lane.
  - `usd-windows-real`: `self-hosted`, labels like
    `[self-hosted, windows, x64, usd-26]`, scheduled/support lane.
- Keep `.github/workflows/ost-source-ci.yml` as a temporary bridge only. It
  should not be branch-protection-required while no matching self-hosted runner
  exists.
- Once `openstrata.ci.yaml` has real digests, generate workflows with
  `ost ci generate github` and stop hand-authoring CI semantics in workflow YAML.
- Do not commit `strata.lock` yet. The root lock is still local-runtime shaped
  and platform-specific; CI reproducibility should come from artifact digests in
  `openstrata.ci.yaml`. Use `strata.lock` as an ignored local sanity check.

## v0.7.0 Findings

Verified locally:

- `ost ci plan` exists and reports lanes, runner classes, hosted jobs, and output
  workflow names.
- `ost ci validate` warns on scaffold placeholder digests.
- `ost ci validate --resolve` fails unresolved placeholder digests.
- `ost ci generate github` refuses placeholder digests unless
  `--allow-placeholders` is passed.
- The GitHub renderer no longer emits the old root-level `steps:` indentation bug.
- A proposed two-cell matrix produced both:
  - `.github/workflows/ost-support-matrix.yml`
  - `.github/workflows/ost-source-ci.yml`
- `ost lock --check` now detects the stale `openusd 25.05.01` lock. Regenerating
  with `ost lock` writes `openusd 26.08`, matching `ost runtime show`, and then
  `ost lock --check` passes.
- `ost plugin test plugins/usdVrm --up-to 5 --json` passes:
  `13 pass`, `0 fail`, `4 skip`.

Still open:

- The local artifact registry is empty, so this repo cannot yet render a usable
  PR/source SDK workflow with real digests.
- `ost plugin package plugins/usdVrm --json` still fails on this Windows host
  when a previous `package-stage` exists:
  `access denied (os error 5)`.
- L5 still skips because there is no `<fixture>.golden.usda` roundtrip baseline.
