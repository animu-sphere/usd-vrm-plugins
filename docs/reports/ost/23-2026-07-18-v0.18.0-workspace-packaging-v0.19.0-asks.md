# OST v0.18.0: workspace packaging half-lands — `bundles` are recorded and linked, but not registered, so the packaged product still cannot open a file

- Date: 2026-07-18
- Versions exercised: `ost 0.18.0` (local Windows, `cy2026-windows-x86_64-py313-usd`).
- Prior report: `22-2026-07-17-v0.17.0-evidence-gate-v0.18.0-asks.md`.
- Context: adopting 0.18.0 to cut **v0.2.0**, this repo's first multi-bundle
  release. Report 22 §11 filed four workspace-packaging asks against exactly
  this version; this report verifies them and re-files what remains.

## TL;DR

Two of report 22's four §11 asks landed, and the evidence-gate knob from §2/§8
landed too. But the way ask 2 was implemented makes the resulting artifact
**worse than before in one specific respect**: `dependencies.json` now reports
the bundle closure as resolved, and the package now carries the dependency
bundle's *link-time* half, while the *USD registration* half is still absent.
So the artifact asserts a closure it does not have, and the failure still
surfaces where it always did — at `Usd.Stage.Open`.

| # | Ask from report 22 §11.4 | 0.18.0 |
| --- | --- | --- |
| 1 | `ost plugin package --workspace` | **delivered** |
| 2 | Record `bundles` in `dependencies.json` | **delivered, but incomplete** — see §2 |
| 3 | Aggregate product artifact | not delivered |
| 4 | `--from-package` composes `--workspace` | not delivered |
| §11.5 | Normalize mixed path separators | not delivered |
| §8 | Cache-hit test must use the gate's own predicate | **delivered** — see §4 |

## 1. What landed

`ost plugin package --workspace` exists and does what §11.4(1) asked —
packages every discovered bundle in dependency order off the same validated
graph `test --workspace` checks:

```console
$ ost plugin package --workspace --json
{"data":{"order":[...],"packages":[
  {"name":"vrmSchema","archive_digest":"sha256:3bb8e3f2f1e7...","version":"0.2.0",...},
  {"name":"usdVrmFileFormat","archive_digest":"sha256:cdde0bee0b08...","version":"0.2.0",...},
  {"name":"usdVrmPackageResolver","archive_digest":"sha256:9e625e72036d...","version":"0.2.0",...}],
  "workspace":true},"ok":true,"schema":1,"warnings":[]}
```

The `.data.packages[]` / `.data.order` shape is clean to consume; our release
lane drives asset staging straight off it. Three archives, not one — see §3.

## 2. Ask 2 landed as a *link* closure, not a *registration* closure

`dependencies.json` in the extracted `usdVrmFileFormat` package now has the
`bundles` key report 22 §11.2 asked for:

```json
{
  "bundles": [
    {"id": "vrmSchema", "version": "0.2.0", "contract": 1,
     "kind": "usd-schema", "descriptor": "openstrata.plugin.yaml",
     "provenance": "source-workspace"}
  ],
  "libraries": [ {"id": "vrmContainer", ...} ],
  "schema": "openstrata.dependencies/v1alpha1"
}
```

0.18.0 goes further than the ask and **stages** the bundle too — the package
grew from 43 to 49 files:

```console
runtime/libraries/lib/libvrmSchema.dll
runtime/libraries/lib/vrmSchema.lib
runtime/libraries/lib/cmake/vrmSchema/vrmSchema{Config,ConfigVersion,Targets,Targets-release}.cmake
```

That is the whole staged contribution. What is **not** there:

```console
$ find <extracted> -name plugInfo.json
<extracted>/plugin/resources/usdVrmFileFormat/plugInfo.json    # only its own
$ find <extracted> -name '*.usda'
$ # nothing -- no generatedSchema.usda
```

A `usd-schema` bundle has two halves. The link half (`libvrmSchema` + CMake
package) matters to someone *building against* it. The registration half
(`plugin/resources/vrmSchema/plugInfo.json` + `generatedSchema.usda`) is what
makes USD able to *apply the schemas at runtime*. Only the first is staged —
even though the bundle is declared `kind: usd-schema`, which is precisely the
kind whose value is the registration half.

### 2.1 The consequence, measured

`ost plugin test --from-package` on the packaged importer:

```text
[PASS] L2 plugin.discovery   USD registry resolves '.vrm' to a file format
[FAIL] L3 usdcat.read        usdcat could not read the fixture: Failed to open
                             "...\tests/fixtures/minimal.vrm" - Exception thrown
                             while opening layer: Used null prim
[FAIL] L4 python.stage_open  Usd.Stage.Open() failed: Failed to open layer
Result: FAILED (12 pass, 2 fail, 1 skip)
```

The same build tested from source: `OK (12 pass, 0 fail, 3 skip)`, L5 golden
roundtrip included. And the same *packages*, composed by hand with all three
bundles on the plugin path:

```text
[1/4] discovery        .vrm -> file format 'vrm'
[2/4] open             opened textures.vrm (9 prims); opened Seed-san.vrm (309 prims)
[3/4] validate         textures.vrm: VALID (0F/0E/0W/0I); Seed-san.vrm: VALID
[4/4] texture resolve  resolved 70B from container
clean-install smoke: PASS
```

So the package contents are correct and complete *except* for the registration
half, and that single omission is the difference between a working product and
one that registers a file format it cannot read.

This is the §3.2 / §11.2 pattern a third time: **the artifact reports success
while omitting the thing that makes it usable.** 0.17.0 at least omitted the
bundle honestly — `dependencies.json` had no `bundles` key at all, so a
consumer could detect the gap. 0.18.0 records the bundle as resolved *and*
ships a `libvrmSchema.dll` that makes the tree look complete. A consumer
inspecting the artifact now has more reason to believe a closure that still
isn't there.

### 2.2 What we expected instead

For a `requires.bundles` edge on a `usd-schema` bundle, staging should carry
the dependency's plugin resource directory into the package and add it to the
session plugin path — the same way `runtime_libs` directories are staged and
declared. Concretely, one more staged tree:

```text
runtime/bundles/vrmSchema/plugin/resources/vrmSchema/{plugInfo.json,generatedSchema.usda}
```

plus a `requires.runtime_plugin_paths` (or equivalent) entry so
`L0 session.plugin_path` includes it. Then `--from-package` would pass without
a workspace flag at all, because the package would genuinely be closed.

## 3. Asks 3 and 4 did not land, and 4 is what actually gates us

`--workspace` on `package` emits **three archives**, not an aggregate
`usdVrmPlugins-<version>-<target>.tar.zst` (WORKSPACE.md §5). That is fine for
now — publishing three assets is a documentation problem, not a correctness
one, and it is what v0.2.0 does.

`--from-package` remains documented "incompatible with `--workspace`", so the
composed configuration is still the one configuration ost cannot verify from
its packaged artifacts. Combined with §2, this means **there is no ost verb
that can verify what we ship.** Our release lane therefore cannot use
`plugin test --from-package` as its packaged-artifact gate at all — a bare
per-bundle run tests the one configuration that provably fails. We gate on
`scripts/clean_install_smoke.py` instead, which composes all three packaged
artifacts and opens, validates, and texture-resolves real models. It is a
stronger test than the ost verb it replaces, and it is still hand-maintained
repo code doing work the toolchain models internally.

## 4. Delivered from the report 22 evidence-gate findings

Worth recording, because both were red-lane issues:

- **§2 (the gate has no knob):** `ci generate` now renders `require_evidence`
  and `evidence_flags` as **matrix variables** per cell, rather than
  hardcoding `--require-sbom --require-provenance` into the run step. Cells can
  now be tuned individually while an artifact is republished. `ci generate`
  also warns per-cell when a pinned artifact cannot satisfy the rendered gate,
  naming the cell and digest — which is exactly the diagnosis report 22 had to
  produce by hand.
- **§8 (cache-hit test bypassed the gate's predicate):** the generated cache-hit
  check now applies the same `${{ matrix.evidence_flags }}` as the gate:

  ```diff
  - && ost artifact verify "${{ matrix.runtime_artifact }}" --json > ...
  + && ost artifact verify "${{ matrix.runtime_artifact }}" ${{ matrix.evidence_flags }} --json > ...
  ```

  So an evidence-less cached record now fails the predicate and falls through
  to the eviction + re-pull path, instead of wedging the lane permanently.

## 5. Smaller note: mixed path separators still ship

Unchanged from report 22 §11.5 — a Windows-produced package still writes
host-shaped separators into portable, digest-addressed data:

```json
"runtime_directories": ["runtime/libraries\\bin", "runtime/libraries\\lib"]
```

and the staged manifest carries the same `runtime/libraries\bin`. It works
because Windows accepts both, but a Windows-produced and a Linux-produced
package still differ in a field describing the same layout, and any consumer
splitting on `/` gets a wrong component.

## 6. Asks for v0.19.0

1. **Stage a `requires.bundles` dependency's USD registration half**, not just
   its link half — at minimum for `kind: usd-schema`, whose entire purpose is
   runtime registration. Today's staging makes the package look closed while
   leaving it functionally open (§2). **This is the P0.** Until it lands, no
   single published bundle from a split workspace is independently installable,
   and `dependencies.json` overstates what the artifact contains.
2. **Let `--from-package` compose `--workspace`** (report 22 ask 4, re-filed).
   With (1) this may become unnecessary for the single-bundle case, but it
   remains the only way to verify a multi-bundle product from its artifacts.
   It is the acceptance test for (1).
3. **An aggregate product artifact** (report 22 ask 3, re-filed, still P1). Our
   release now publishes three assets per target and documents the install
   order; an aggregate would collapse that back to one.
4. **Normalize staged paths to `/`** (report 22 §11.5, re-filed, still small).

Priority note: (1) is a correctness regression in the artifact *contract* — not
in the sense that anything worked before and broke, but that the artifact's
self-description became confidently wrong where it used to be honestly silent.
Everything else here is a feature gap with a working repo-side workaround.
