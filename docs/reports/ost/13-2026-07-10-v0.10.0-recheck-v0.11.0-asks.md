## TL;DR

The important v0.9.0 asks are fixed in v0.10.0:

1. **`runtime pull --build` now records the built OpenUSD version** (`26.05`),
   not the catalog default (`25.05.01`), and validates without re-adopting.
2. **Schema-generation Python deps are provisioned into the runtime**
   (`jinja2` + `MarkupSafe` in `lib/python`) and `runtime validate` now checks
   them.
3. **`runtime export --slim` keeps top-level `resources/`**, so the MaterialX
   CMake config no longer makes slim runtimes unusable.
4. **`ost artifact push` exists** as the producer verb the OCI transport was
   missing.

Full local dogfood also passed: from-source runtime build, runtime validation,
slim runtime export, artifact verify, plugin schema generation, configure,
compile/link, L5 verification, CTest corpus smoke, package, and local plugin
publish.

## Follow-up: hosted Linux PR lane fails on ubuntu-24.04 (glibc floor mislabel)

After wiring the new `usdvrm-pr-linux` hosted cell into CI (PR #23), the very
first hosted Linux PR run failed. This is the most important finding of this
report and turns Linux-runtime ABI targeting into a hard v0.11.0 blocker.

Failing run:
`https://github.com/animu-sphere/usd-vrm-plugins/actions/runs/29034868407`
(job `usdvrm-pr-linux`, GitHub-hosted `ubuntu-24.04`).

The consume path looked perfectly healthy: `ost artifact pull` resolved and
verified the pinned runtime, including `target_match` against
`linux-x86_64-glibc228-py313`, and `runtime materialize` succeeded. The job then
died in the very next step, `ost plugin build` → `schema-generate`:

```text
==> [schema-generate] .../bin/usdGenSchema ...
ImportError: /lib/x86_64-linux-gnu/libm.so.6: version `GLIBC_2.43' not found
             (required by .../lib/libusd_gf.so)
error[EXTERNAL_TOOL_FAILED] (phase: schema-generate): exit 1
##[error]Process completed with exit code 6.
```

Root cause: the Linux runtime was built in WSL on **Ubuntu 26.04, whose glibc is
2.43**. Its binaries (`libusd_gf.so`, etc.) therefore reference `GLIBC_2.43`
versioned symbols. But the runtime's recorded target ABI is
`linux-x86_64-glibc228-py313` — a nominal **glibc 2.28** label that does not
reflect the real symbol floor. GitHub-hosted `ubuntu-24.04` ships **glibc 2.39**,
so the dynamic loader rejects `GLIBC_2.43` and `usdGenSchema` cannot import
`pxr`.

Two problems compound here:

1. **The `glibc228` variant label is fabricated, not measured.** `ost` appears
   to stamp a catalog-default ABI tag rather than computing the maximum
   `GLIBC_x.y` symbol version actually referenced by the exported binaries (e.g.
   via `readelf -V` / dynamic version needs). A slim runtime built on glibc 2.43
   should be labeled `glibc243` (or higher of its real floor), never `glibc228`.
2. **`--require-target` gave false confidence.** Because the consumer only
   string-matches the recorded target label, the digest-pinned pull "passed" its
   ABI check while the runtime was in fact unusable on the target runner. The
   metadata pin is only as trustworthy as the label the producer wrote, and here
   the producer wrote a label the binaries contradict.

Net effect: a glibc-2.43 runtime can never run on the `ubuntu-24.04` hosted
runner. The fix is on the producer side (build the Linux runtime against an old
glibc — a manylinux_2_28 / older-Ubuntu / container base — so `glibc228` is
*true*, or detect and record the real floor), not on the CI YAML. Until the
Linux runtime is rebuilt on a compatible glibc, the hosted Linux PR lane cannot
be green and should not gate PRs.

## New v0.11.0 asks

1. **Fix `ost artifact push` against GHCR.** In this run, `OST_REGISTRY_TOKEN`
   returned 403, while `OST_REGISTRY_USER` + `OST_REGISTRY_PASSWORD` advanced to
   upload but generated an invalid `digest=sha256:sha256:<hex>` request and
   failed with HTTP 400. `oras` with the same token worked. The producer verb
   should match the already-solid pull path.
2. **Support Linux symlinks in runtime export.** Linux SDKs naturally contain
   shared-library symlink chains. `runtime export --slim` currently rejects them
   as unsafe package-staging inputs, which blocks direct export of a build-source
   Linux runtime. Preserve safe relative symlinks, dereference them intentionally,
   or expose an explicit policy switch.
3. **Document Linux build prerequisites.** For this WSL build, Ubuntu 26.04
   needed a source-built Python 3.13.14, `unzip`, `libxt-dev`, and the usual
   compiler/CMake/Ninja/X11/OpenGL dev packages. The oneTBB GitHub archive URL
   also returned 504 HTML; `codeload.github.com` worked as a cache seed.
4. **Make package-stage fallback quieter or more actionable.** `ost plugin package`
   succeeds, but repeated runs emitted `STAGE_FALLBACK`:
   the previous `package-stage` was "held open by another process", so `ost`
   staged into `package-stage-<id>` instead. The fallback is much better than
   the old hard failure, but v0.11.0 should either identify the locking process,
   sweep stale fallback stages deterministically, or expose a `--clean-stage`
   operator escape hatch.
5. **Give generated source CI a first-class way to keep repo-specific smoke
   tests.** Regenerating with v0.10.0 dropped the previous corpus CTest smoke
   step. Either support declarative post-build commands in `openstrata.ci.yaml`
   or make `ost plugin test` cover the corpus assets directly.
6. **Optionally re-test build-host dependency preflight on a clean Python.**
   Runtime-side `jinja2` is fixed. The remaining unknown is whether a pristine
   Python without `Jinja2`/`PyOpenGL`/`PySide6` now gets a clear `ost` preflight
   before `build_usd.py`.
7. **Measure and record the real glibc floor for Linux runtimes (BLOCKER).**
   The WSL/Ubuntu-26.04 Linux runtime was exported with the target ABI label
   `linux-x86_64-glibc228-py313`, but its binaries reference `GLIBC_2.43`, so it
   fails to load on the GitHub-hosted `ubuntu-24.04` runner (glibc 2.39) during
   `usdGenSchema` (`version GLIBC_2.43 not found`, exit 6). `ost runtime export`
   should compute the maximum referenced `GLIBC_x.y` symbol version from the
   exported ELF binaries and stamp that as the ABI variant, and `ost artifact
   pull --require-target` should treat a runtime whose real floor exceeds the
   host as a failed ABI match — not silently pass a fabricated `glibc228` label.
   Correspondingly, document that portable Linux runtimes must be built against
   an old glibc base (manylinux_2_28 / older-Ubuntu / container) rather than on a
   bleeding-edge host, so the recorded floor is genuinely low.

## Net

v0.10.0 closes the production blockers from the v0.9.0 report. The natural arc
now works locally: build OpenUSD from source through `ost`, validate the runtime,
export a slim-but-consumable runtime artifact, build/test/package the plugin, and
publish the plugin into the local artifact registry. The follow-up GHCR work now
also proves public Windows and Linux runtime consumption by digest. The remaining
v0.11.0 work is concentrated on the producer side: make `artifact push` reliable,
make Linux symlinked SDK exports first-class, and document the build prerequisites
well enough that this path is boring on a fresh machine.
