# OST v0.9.0: hosted-bootstrap gap closed, first OCI runtime publish, and two v1.0.0-rc1 asks

- Date: 2026-07-09
- `ost --version`: `ost 0.9.0`
- Inputs:
  - `docs/report/ost/11-2026-07-05-v0.8.0-ci-hosted-lanes.md` (the asks this report re-verifies)
  - Task: rebuild OpenUSD from source through `ost`, then publish the runtime to
    GHCR as the `runtime_remote` image the new CI schema references.

## TL;DR

Every v0.8.0 "main ask for v0.9.0" landed, and the hosted-bootstrap gap is not
just closed but **enforced**: the old matrix now fails `ost ci validate`. The
generated source lane renders a full pinned-`ost` bootstrap + `oci://` runtime
pull. Slim export, multithreaded/tunable zstd, and export progress output all
shipped. Two new frictions surfaced while doing a real end-to-end publish:

1. **`runtime pull --build` records the wrong OpenUSD version** (catalog default,
   not the built `pxr.h`), and the drift validator's own suggested fix is a no-op.
2. **There is still no `ost artifact push`** — the *consume* side of OCI transport
   is first-class, but *producing* the image the schema demands is left to `oras`.

## v0.8.0 asks — resolution status (all FIXED)

### 1. Hosted bootstrap — FIXED and ENFORCED
The v0.8.0 blocker was "the PR lane renders but cannot run on a hosted runner:
no way to install `ost`, no way to seed the 1.93 GiB runtime." v0.9.0 closes both:

- The matrix schema gained a top-level `bootstrap.ost` block (pinned `version`,
  release-asset `repository`, optional per-triple `sha256` pins) and a per-cell
  `runtime_remote: { uri: oci://…@sha256:<digest>, expected_oci_digest }`.
- **Enforcement:** the existing 0.8.0 `openstrata.ci.yaml` (hosted source cell,
  no `runtime_remote`) now **fails** `ost ci validate` / `--resolve` with exit 3:
  > `cell 'usdvrm-pr-windows': source cells on GitHub-hosted runners require a
  > runtime_remote reference (uri: oci://…@sha256:<digest>) — nothing else can
  > seed the runner's registry`
- The rendered source lane now contains, for hosted cells:
  - a **Bootstrap step** that downloads a pinned `ost` release asset
    (`https://github.com/animu-sphere/open-strata/releases/download/v0.9.0/ost-cli-<triple>.<ext>`),
    fetches its `.sha256`, verifies the hash (and any CI-pinned sha256), unpacks,
    prepends to `GITHUB_PATH`, and asserts `ost --version == ost 0.9.0`;
  - a **remote pull step**:
    `ost artifact pull <runtime_remote> --expect-artifact <runtime_artifact> --require-kind runtime`,
    fronted by an `actions/cache` keyed by the artifact digest that is explicitly
    documented as *speed only, never a correctness precondition* (cache miss just
    re-pulls). Self-hosted cells keep air-gapped local `ost artifact import`.
- Supporting CLI surface is all present: `ost artifact resolve` (tag → immutable
  digest; refuses mutable refs), `ost artifact pull` (`oci://…@sha256` or
  `file://<dist>`, with `--expect-artifact`, `--require-kind`, `--require-target`,
  `--plain-http`).

Renderer hygiene remains good: full action SHAs, fork-PR safety notes, CI
evidence (`.ost-ci/*.json`) uploaded, `--force`/`--stdout`/`--allow-placeholders`
flags on `ci generate github`.

### 2. Slim / SDK-profile export — FIXED
`ost runtime export --slim` keeps only `include/lib/bin/plugin/cmake` + config and
drops the adopted build/source tree. Concrete result here: the cy2026/usd runtime
exported to a **27.3 MiB** `tar.zst` (vs the v0.8.0 full export's ~1.93 GiB).

### 3. Multithreaded / tunable zstd — FIXED
`ost runtime export` gained `--jobs` (defaults to host parallelism; `--jobs 1`
forces single-thread) and `--level 1–22` (default 19).

### 4. Export progress output — FIXED
`runtime export` now streams per-file progress
(`3678/3678 files, 162.2 MiB in 14s`), fixing the v0.8.0 "looks hung for 52 min"
complaint. (The 52 min figure was dominated by a 14.4 GB adopted build tree;
`--slim` sidesteps that entirely.)

### 5. L5 golden skip-message wording — NOT re-verified
The fixture `plugins/usdVrm/tests/fixtures/minimal.vrm.golden.usda` now exists, so
the level no longer skips; the message-clarity nit was not re-exercised. Carried
forward as a minor ask.

## Dogfood: rebuild OpenUSD from source through `ost`

Goal: a clean, latest-OpenUSD, Python 3.13 runtime, built by `ost` via
`build_scripts/build_usd.py`, then published to GHCR.

- Source: `PixarAnimationStudios/OpenUSD` checked out at **v26.05** (latest
  release; `PXR_MINOR_VERSION 26 / PATCH 5`), in `C:\dev\OpenUSD`.
- Command:
  `ost runtime pull cy2026 --profile usd --build C:\dev\OpenUSD --jobs 32 --force`
  with Python 3.13.14 first on PATH (usdview + schemagen enabled).
- **Prerequisite friction (build_usd.py, not ost):** a clean 3.13 lacks the
  packages build_usd.py needs, and build_usd.py aborts (exit 1) listing them:
  `Jinja2` (else usdGenSchema/usdgenschemafromsdr/usdInitSchema are silently
  omitted), `PyOpenGL` + `PySide6` (usdview). After
  `pip install Jinja2 PyOpenGL PySide6` the build completed in **~10 min** on 32
  cores, and re-running produced an **identical runtime digest** (deterministic).
  *Nice-to-have:* `ost` could surface/preflight these interpreter deps (it knows
  the profile wants usdview + schema tooling) rather than letting build_usd.py
  fail deep in the invocation.

### Finding A — `--build` records the catalog-default OpenUSD version, not the built one
`runtime pull --build` stamped the manifest's `openusd` extension as **25.05.01**
(a catalog default) even though the freshly built `include/pxr/pxr.h` reports
**26.05**. `ost runtime validate` *correctly* catches this:

> `[FAIL] openusd-version-drift — manifest records OpenUSD 25.05.01, but the
> install's pxr.h reports 26.05`

…but the two documented recoveries do not work for a `build`-source runtime:

- The validator's **own suggested fix** — `ost runtime pull … --build <src> --force`
  — is a **no-op**: re-running reproduced the exact same digest and the same
  wrong `25.05.01`.
- `ost runtime repair` **refuses**: `REPAIR_UNSUPPORTED_SOURCE` (repair re-adopts
  a `local` runtime only).

**Root cause (confirmed by ost's own note):** the version is taken from the
extension **catalog default**, and only the *adopt* path detects the real value.
Re-adopting the built install as `local` fixed it immediately:

```
# copy the clean SDK (exclude build/ + src/) to a stable root, then:
ost runtime pull cy2026 --profile usd --from-usd C:\dev\build\openusd-26.05-cy2026 --force
#   note: adopted OpenUSD reports version 26.05 (catalog default was 25.05.01)
ost runtime validate cy2026 --profile usd     # -> passed, openusd 26.05
```

This is a real blocker in the natural path: **export refuses an unvalidated
runtime** (`EXPORT_VALIDATION_REQUIRED`), so a from-source build cannot be
exported at all until the version is corrected — and the only correction is to
throw away the `build` provenance and re-adopt as `local`.

*Ask (v1.0.0-rc1):* the `--build` path should read the real version from the
built `pxr.h` (as the adopt path already does) and stamp that. Failing that,
make `runtime repair` (or a `runtime pull --redetect`) work for `build` source so
the validator's advice actually recovers the runtime, and fix the validator's
suggested-fix text so it isn't a no-op.

## Dogfood: publish the runtime to GHCR (animu-sphere, public)

Target image (multi-OS aware; OS/arch in the tag, repo stays OS-neutral):
`ghcr.io/animu-sphere/openstrata-runtime-cy2026-usd:26.05-windows-x86_64`
(Linux later as `:26.05-linux-x86_64`).

Exported artifact (slim, after the local re-adopt fix):
- OST artifact digest: `sha256:ac249bd764511edc00ea5c18692ddab74db36e82c373815edcfb1368ba353597`
- `openusd 26.05`, archive 28,663,030 bytes (~27.3 MiB), + `manifest.json` + `SHA256SUMS`.

### Finding B — OCI transport is consume-only; there is no `ost artifact push`
v0.9.0 makes *pulling* an image first-class (`artifact resolve`/`pull`, the
`runtime_remote` schema, the rendered CI pull step), but there is **no producer
verb**: `ost artifact` has `import/list/show/verify/export/extract/resolve/pull`
and no `push`. So the very image the `runtime_remote` contract requires must be
produced out-of-band with `oras` (plus a GHCR `write:packages` credential and a
manual visibility flip to Public). The operator is left to invent the transport
in exactly the spot the schema says "pull this."

Publish path actually used here (this exact `oras` recipe round-trips cleanly
through `ost artifact pull`):
```
oras push ghcr.io/animu-sphere/openstrata-runtime-cy2026-usd:26.05-windows-x86_64 \
  --artifact-type application/vnd.openstrata.runtime.v1 \
  openstrata-cy2026-windows-x86_64-py313-usd.tar.zst manifest.json SHA256SUMS
# -> OCI digest sha256:59488b57faff1c907947ea5f4ad63caa5ce7913cbf712f3b1de2154a49f5618c
ost artifact pull \
  oci://ghcr.io/animu-sphere/openstrata-runtime-cy2026-usd@sha256:59488b57… \
  --expect-artifact sha256:ac249bd7… --require-kind runtime
```
- Pushed OCI digest: `sha256:59488b57faff1c907947ea5f4ad63caa5ce7913cbf712f3b1de2154a49f5618c`
- **Round-trip: PASS.** `ost artifact resolve` returned the pinned digest and
  `ost artifact pull` passed every gate — `oci_digest`, `manifest_schema`,
  `archive_digest`, `pinned_artifact_digest`, `archive_safety`, `file_digests`,
  `kind_match`, `trust_policy`, `local_import` (default `oras push` layer
  titles = filenames are exactly what `ost` reads).
- Two smaller frictions on the produce side:
  - **`ost` does not use the `oras`/docker credential store.** Even after
    `oras login`, `ost artifact resolve/pull` on the (private) package returned
    `ARTIFACT_AUTH_DENIED (HTTP 401)` until `OST_REGISTRY_USER` +
    `OST_REGISTRY_PASSWORD` were set. Fine once documented, but surprising right
    after a successful `oras login`.
  - **GHCR visibility is not settable via API.** `PATCH
    /orgs/{org}/packages/container/{name}` → 404; making the package Public is a
    WebUI-only step — and here it was additionally gated by an org/enterprise
    "Package creation" policy (the pushing user was an org `member`, not owner) so
    the Public radio was greyed until the policy was lifted. Worth calling out in
    whatever `ost` docs describe the publish flow. **Once Public, the fully
    anonymous CI path works:** `ost artifact resolve`/`pull` (no credentials)
    passed every gate against `sha256:59488b57…`.

*Ask (v1.0.0-rc1):* add a first-class producer — `ost artifact push oci://…` (or
`ost runtime publish --to oci://…`) that emits the exact OCI layout/media-types
`ost artifact pull` expects, prints the resulting OCI digest for pasting straight
into `runtime_remote.expected_oci_digest`, and optionally sets package
visibility. Until then, document the canonical `oras` recipe + media types so
publishers don't guess.

### Finding D — `usdGenSchema` needs `jinja2`, but a from-source runtime doesn't ship it
The single failure was in `ost plugin build` → the **schema-generate** phase:
```
==> [schema-generate] py -3.13 …/bin/usdGenSchema …/schema.usda …
ModuleNotFoundError: No module named 'jinja2'
error[EXTERNAL_TOOL_FAILED] (phase: schema-generate): exit 1
```
`usdGenSchema` is a Python script that imports `jinja2` at runtime, but **`jinja2`
is not part of OpenUSD** — `build_scripts/build_usd.py` only needs it on the
*building* host and never installs it into the USD install tree. So the exported /
published runtime image (bin/ ships `usdGenSchema`, lib/python ships `pxr`, but
**not** `jinja2`) is silently incomplete for schema generation. It only worked
locally because the build host's Python happened to have `jinja2`; the clean CI
runner's `py -3.13` did not.

Workaround applied (makes the image self-contained): `pip install --target
<runtime>/lib/python jinja2` (pulls `MarkupSafe` too), then re-export slim and
re-push. `lib/python` is exactly what `ost` puts on `PYTHONPATH`, so any Python
that runs the bundled `usdGenSchema` now resolves `jinja2`. Republished image:
artifact `sha256:8c51aa15…`, OCI `sha256:b5b2d322…` (tag `:26.05-windows-x86_64`),
anonymous `ost artifact pull` PASS.

*Ask (v1.0.0-rc1):* a runtime that ships schema tooling (`usdGenSchema` in bin/)
should ship its Python deps (`jinja2` + `MarkupSafe`) in `lib/python`. Provision
them at `runtime pull --build` time (or vendor them during `runtime export`) so
every published runtime is complete for `ost plugin build`'s schema-generate
phase — otherwise the omission only surfaces on a clean runner, deep inside a
plugin build, as a bare `ModuleNotFoundError`.

### Finding E — `--slim` drops the top-level `resources/` that MaterialX's CMake config requires
With `jinja2` bundled, the next CI run cleared schema-generate and failed one
phase later, in `configure`:
```
CMake Error at …/lib/cmake/MaterialX/MaterialXConfig.cmake:19 (message):
  … resources referenced by variable MATERIALX_RESOURCES_DIR does not exist !
  (find_package(pxr) -> find_dependency(MaterialX) -> set_and_check)
```
`--slim` keeps `include/lib/bin/plugin/cmake/libraries`, but drops the SDK's
**top-level `resources/`** (≈132 MB, mostly MaterialX data). Yet `pxrConfig.cmake`
chains into `MaterialXConfig.cmake`, which `set_and_check`s
`MATERIALX_RESOURCES_DIR = <prefix>/resources` — a hard existence check at
`find_package(pxr)` time. So a slim runtime is **not** consumable by any plugin
that does `find_package(pxr)` when the runtime was built with MaterialX (the
default). Fixed by exporting **full** (no `--slim`); since the adopted root has no
build/src tree, the archive is still compact (~104 MB). Final published runtime:
artifact `sha256:28c2c00e…`, OCI `sha256:eda1b94d…`.

*Ask (v1.0.0-rc1):* `--slim` should retain `resources/` (or at least the MaterialX
resources the shipped `cmake/MaterialX` config `set_and_check`s), otherwise the
slim profile it advertises for CI is unusable for the common
`find_package(pxr)` + MaterialX plugin build.

### Outcome — hosted lane GREEN
After the two republishes (jinja2 into `lib/python`, then full export for
`resources/`), the hosted `windows-2022` PR job goes **green end-to-end in ~1m51s**
at the final digests above: bootstrap pinned `ost` → anonymous GHCR runtime pull →
schema-generate → configure → compile/link → verification pyramid → package. The
v0.8.0 hosted-bootstrap blocker is closed and *demonstrated* on real CI.

## v1.0.0-rc1 asks, in priority order

1. **Fix `--build` OpenUSD version detection** (stamp the built `pxr.h` version,
   not the catalog default); and make the drift validator's suggested recovery
   actually recover a `build`-source runtime (`repair`/`--redetect`). Today the
   only fix is discarding build provenance via `--from-usd` re-adopt, and export
   is hard-blocked until then.
2. **Ship schema-gen Python deps inside the runtime** (`jinja2` + `MarkupSafe` in
   `lib/python`) whenever `usdGenSchema` is bundled — this is what actually broke
   the first real hosted CI run (Finding D). Highest-impact single fix for green
   CI; today every published runtime image is silently incomplete for
   schema generation.
3. **`--slim` must keep `resources/`** (Finding E) — a slim runtime built with
   MaterialX is currently unusable by any `find_package(pxr)` plugin because
   `MaterialXConfig.cmake` `set_and_check`s `<prefix>/resources`. The slim profile
   is advertised for CI; it should stay consumable.
4. **Add a producer for OCI transport** (`ost artifact push` / `ost runtime
   publish --to oci://…`): emit the pull-compatible layout, print the OCI digest,
   optionally set visibility. Closes the loop the `runtime_remote` schema opened.
4. **Optional build-dep preflight**: warn on missing `Jinja2`/`PyOpenGL`/`PySide6`
   before invoking build_usd.py when the profile implies usdview + schema tooling.
   (The bootstrap release assets in Finding C are confirmed present on real CI, so
   that earlier ask is closed.)
5. **Clarify the L5 golden skip message** (`<fixture-filename>.golden.usda`,
   extension included) — carried over from v0.8.0.

## Net

The hosted-CI story that was "well-defined but blocked" in v0.8.0 is now
mechanically complete on the *consume* side and enforced by validation. The two
remaining rough edges are both on the *produce* side: a from-source runtime can't
be exported without working around a version-stamp bug, and the OCI image the CI
contract depends on still has to be pushed by hand. Both are squarely in scope
for a 1.0 that wants "build it, publish it, pull it in CI" to be a single
supported arc.
