# OST dogfooding report #1 — bootstrapping the usdVrm file-format plugin

- Date: 2026-06-28
- `ost` version: (see `ost --version`; built from `C:\dev\open-strata`)
- Host: Windows 11, MSVC 14.34 (VS 2022 Community), CMake 4.4, Ninja (winget)
- Runtime: adopted local OpenUSD at `C:\dev\build\usd2605` (actually **USD v26.08**, `PXR_VERSION 2608`)
- Context: standing up `usd-vrm-plugins` from an empty repo, driving everything through `ost`.

This is report #1 of an ongoing series. It covers project/runtime/bundle setup
(`ost init`, `ost runtime pull --from-usd`, `ost plugin new|doctor|build|test|run`).
The phases the team cares most about — **multi-plugin sessions
(`ost plugin run/view --with <bundle>`)** and **bundle-declared
`requires.runtime_libs`** — are not exercised yet (single plugin, no 3rd-party
runtime libs linked yet). Those get their own report once cgltf/VRM.h are vendored
and a second bundle or a non-USD runtime dep enters the picture.

---

## TL;DR — what blocked a green `ost plugin test`, in order

A freshly scaffolded bundle does **not** pass `ost plugin test` on Windows even
after a clean build. Three independent issues had to be worked around:

1. **`ost plugin build` passes a *relative* `CMAKE_TOOLCHAIN_FILE`** → CMake
   resolves it against the build dir and can't find it. (P1 — build is broken
   out of the box.)
2. **Scaffolded `plugInfo.json` has `LibraryPath: "libX.so"`** — wrong suffix on
   Windows, and more fundamentally points at a file that never exists next to
   `plugInfo.json` (the built lib lands in `lib/`, not the resources dir). (P1 —
   discovery can never succeed as scaffolded, on any OS.)
3. **`ost plugin test|run|doctor` compose `PXR_PLUGINPATH_NAME` (and `PATH`)
   with the *relative* bundle path you passed on the CLI.** USD resolves relative
   plugin search paths against its own lib dir, not the cwd, so discovery
   silently fails. Passing an **absolute** bundle path makes all of L2–L4 pass.
   (P1 — the headline command is path-sensitive in a non-obvious way.)

With (1) worked around (build via vcvars+Ninja with an absolute toolchain path),
(2) fixed (`LibraryPath: "../../../lib/libUsdVrmFileFormat.dll"`), and (3) worked
around (absolute bundle path), `ost plugin test C:/dev/usd-vrm-plugins/plugins/usdVrm`
reports **OK (10 pass, 0 fail, 3 skip)**.

## What went well

- **`ost plugin new usd-fileformat usdVrm --extension vrm` is excellent.** One
  command yields a self-describing bundle (`openstrata.plugin.yaml`), a working
  `SdfFileFormat` skeleton, `plugInfo.json`, fixtures, and a `.gitignore`. The
  manifest's `provides` / `requires` / `tests` sections are exactly the right
  contract surface.
- **`ost runtime pull cy2026 --profile usd --from-usd <path>`** to adopt an
  existing local USD build is the killer feature for dogfooding — no waiting on a
  real published runtime. `ost runtime validate cy2026 --profile usd` then passed
  all checks.
- **The staged verification pyramid (L0–L6) in `ost plugin doctor|test`** is
  genuinely good UX. The level labels + remediation hints (`↳ ...`) told me
  exactly what to look at each time, and the JSON/`summary.txt` reports under
  `.strata/reports/...` are a nice audit trail.
- **`ost plugin doctor` degrades gracefully without a runtime** (everything L1+
  becomes SKIP with a clear "needs a real runtime" reason) and lights up once one
  is adopted. The "Session env preview" block at the bottom of doctor output is
  very helpful — it's how I spotted the relative-path problem (#3).
- **Project→bundle defaulting works**: with `openstrata.toml` at the repo root
  (`ost init --bare`), `ost plugin <cmd> plugins/usdVrm` inherits
  `platform=cy2026, profile=usd` with no flags.

## Issues, ranked

### P1 — `ost plugin build` passes a relative `CMAKE_TOOLCHAIN_FILE`
`ost plugin build plugins/usdVrm` runs:
```
cmake -S plugins/usdVrm -B plugins/usdVrm/build/<target> \
  -DCMAKE_TOOLCHAIN_FILE=plugins/usdVrm/.strata/targets/<target>/toolchain.cmake ...
```
CMake resolves `CMAKE_TOOLCHAIN_FILE` **relative to the build directory**, not the
cwd, so it errors with *"Could not find toolchain file"* even though the file was
just generated at that (cwd-relative) path. The `--dry-run` output shows the same
relative path, so this is in the command composer.
**Fix:** absolutize `CMAKE_TOOLCHAIN_FILE` (and ideally `-S`/`-B`) before invoking
CMake.
**Workaround used:** invoke CMake myself inside a `vcvars64` shell with an
**absolute** `-DCMAKE_TOOLCHAIN_FILE=...`.

### P1 — relative `PXR_PLUGINPATH_NAME` in the session env breaks discovery
`ost plugin test plugins/usdVrm` (relative bundle path) emits
`PXR_PLUGINPATH_NAME += plugins/usdVrm/plugin/resources/usdVrm`. With
`TF_DEBUG=PLUG_INFO_SEARCH` I can see USD then looks under
`C:/dev/build/usd2605/lib/plugins/usdVrm/plugin/resources/usdVrm` — i.e. it
anchored the relative path at the **USD lib dir**, not the cwd. USD has always
treated relative plugin search paths this way; the session composer must therefore
hand USD **absolute** paths.
- `ost plugin test plugins/usdVrm` → L2/L3/L4 FAIL.
- `ost plugin test C:/dev/usd-vrm-plugins/plugins/usdVrm` → L2/L3/L4 PASS.

**Fix:** canonicalize the bundle path to absolute when composing
`PXR_PLUGINPATH_NAME`/`PATH`/`PYTHONPATH` for `run`/`test`/`view`/`doctor`. This
one matters a lot for the upcoming **multi-plugin `--with <bundle>`** work: every
`--with` bundle's plugInfo root and `lib/` will need to be absolutized too, or the
exact same silent-discovery-failure will hit each added plugin.

### P1 — scaffolded `plugInfo.json` can never load its own library
The template emits:
```json
"Root": ".", "LibraryPath": "libUsdVrmFileFormat.so", "ResourcePath": "."
```
Two problems:
1. `.so` suffix on Windows (should be `.dll` / `.dylib`).
2. The built library is staged in the bundle's `lib/` (per the scaffold
   `CMakeLists.txt`), but `LibraryPath` is resolved by USD relative to the
   plugInfo's `Root` (the resources dir) and USD **absolutizes it and dlopens that
   absolute path — there is no PATH/LD_LIBRARY_PATH fallback for the plugin lib
   itself.** A bare leaf name therefore fails even though `lib/` is on `PATH`.
   Verified: `LibraryPath: "libUsdVrmFileFormat.dll"` → L2 FAIL;
   `LibraryPath: "../../../lib/libUsdVrmFileFormat.dll"` → L2 PASS.

**Suggested fix:** the scaffold should generate `plugInfo.json` with (a) the
platform-correct suffix and (b) a `LibraryPath` that actually points at the lib
output dir — ideally via a `plugInfo.json.in` + CMake `configure_file` using
`${CMAKE_SHARED_LIBRARY_PREFIX}`/`${CMAKE_SHARED_LIBRARY_SUFFIX}` (this is what the
sibling LumaGraph plugin does). Note this collides with `ost plugin doctor`'s L0
`bundle.plug_info` check, which reads the *static* file — so either doctor should
accept a `.in` template, or `build` should regenerate the committed file.

### P2 — runtime version is mis-detected/mis-labelled
The adopted USD at `C:\dev\build\usd2605` is **v26.08** (`PXR_VERSION 2608`,
confirmed by the `pxrInternal_v0_26_8` symbols in an L4 error), yet:
- `ost runtime pull --from-usd` reported `extensions: openusd 25.05.01`.
- `ost plugin doctor` reported *"runtime OpenUSD 25.05.01 satisfies
  '>=25.05,<26.0'"*.
- The runtime id is `...-py313-usd`, but the install actually links **Python
  3.10** (`python310.lib`).

So the version gate is checking a hard-coded/placeholder `25.05.01` rather than the
real install's version. A true 26.08 runtime should *fail* `>=25.05,<26.0`, but it
"passes" because the detected version is wrong. This will mask real ABI/range
mismatches. **Fix:** parse `pxr/pxr.h` (`PXR_VERSION`) and the actual Python ABI
from the adopted install.

### P2 — `ost runtime show` / `validate` accept different identifiers
`ost runtime list` prints the full id
(`openstrata-cy2026-windows-x86_64-py313-usd`), and `runtime pull` echoes it, but
`ost runtime show <that-id>` and `ost runtime validate <that-id>` fail with
`PLATFORM_NOT_FOUND`; they want `cy2026 --profile usd`. The accepted identifier
form should be consistent across `list`/`show`/`validate` (accept either the full
id or the `<platform> --profile <profile>` pair).

### P3 — `--dry-run` shows the right intent but `build` diverges
`ost plugin build --dry-run` prints *"would generate ... toolchain.cmake then:
cmake ..."*. The real run does generate the toolchain but then fails on the
relative path (P1). The dry-run is a faithful preview of the *commands*, which is
good — it just shares the underlying relative-path bug.

### P3 — no `cl` on PATH; `host` compiler policy assumes a dev shell
The generated `toolchain.cmake` uses the `host` compiler policy + Ninja, which
needs `cl.exe` on `PATH`. From a plain shell that isn't true, and `ost plugin
build` doesn't wrap the invocation in `vcvars`. On Windows, `ost` could either
(a) auto-discover VS via `vswhere` and inject the dev environment, or (b) document
that `ost plugin build` must run from a Developer prompt / `ost devshell`. (I also
saw a stray `'vswhere.exe' is not recognized` line during CMake's own VS probe,
which is just MSYS PATH noise but adds to the confusion.)

### P3 — `tbb12_debug.lib` link failure when `CMAKE_BUILD_TYPE` is unset
With Ninja single-config and no `CMAKE_BUILD_TYPE`, USD's imported targets resolve
to the Debug config and the link pulls `tbb12_debug.lib`, which doesn't ship in a
Release-only USD install → `LNK1104`. Setting `-DCMAKE_BUILD_TYPE=Release` fixes
it. Since `ost` knows the runtime is Release-only, the generated toolchain/preset
should default `CMAKE_BUILD_TYPE`/the Ninja config to Release (or warn).

## Concrete asks for the upcoming ost work

For **multi-plugin sessions** and **`requires.runtime_libs`**, the above means:
- Absolutize **every** path that ends up in `PXR_PLUGINPATH_NAME` / `PATH` /
  `PYTHONPATH`, including each `--with <bundle>` root, its `lib/`, and any
  `requires.runtime_libs` dir. (P1 #3 will otherwise recur per added bundle.)
- Decide and document the canonical mapping from "built lib in `lib/`" →
  `plugInfo.json` `LibraryPath`. Multi-plugin sessions multiply the #2 problem.
- `requires.runtime_libs` should be prepended to the dynamic-loader path
  (`PATH`/`LD_LIBRARY_PATH`/`DYLD_LIBRARY_PATH`) of the session — this is exactly
  what replaces the hand-rolled usdview launch `.bat` files. A plugin that vendors
  e.g. its own `zlib.dll` (LumaGraph) or, here, statically links cgltf/VRM.h (so
  *no* extra runtime lib — a useful contrast case) should both be expressible.
