# usd-vrm-plugins roadmap

Living status + prioritized work for `usdVrm`, organized to match the project
**design & development policy** ([`DESIGN_POLICY.md`](DESIGN_POLICY.md)). The importer
feature build-out (the original importer Phase 1–4 plan) is **done**; the work ahead
is the policy's strategic pivot — *not* more features, but **doc/impl sync, schema
contract, `v0.1.0` release, Windows runtime CI, and the OpenExec runtime layer** — and
it is organized against the policy's **P0–P6** roadmap (policy §17).

> Phase numbering follows the design policy's **P0–P6** (target milestone `v0.1.0`).
> An earlier internal draft used "Phase A–E"; those items are folded into P0–P6 below.
> Importer phases 1–4, the Phase-1 correctness gaps, and the front-direction bake are
> all shipped — detail lives in git history and the "Shipped" summary here.

## Shipped — importer + typed schemas + reliability tooling

The pieces below are landed and continuously tested; they satisfy large parts of
**P0/P1/P3** and all of the semantic-contract and portability goals.

### Importer + typed schemas (importer Phases 1–4)

- **Read + canonicalize:** `.vrm` GLB read, VRM 0.x/1.0 detection, all version
  differences absorbed into `VrmCanonicalDocument` before any USD is authored.
- **Geometry / materials:** meshes (points/normals/UV/indices), non-skinned node
  transforms preserved, `UsdPreviewSurface` materials with the full texture set
  (base color, metallic-roughness, normal, emissive, occlusion; wrap modes,
  `KHR_texture_transform`) and MToon metadata (`vrm:mtoon:raw`, shader tagged MToon).
- **Skeleton / skinning:** one `UsdSkelSkeleton` unified across all glTF skins,
  topologically ordered, bind from **inverse bind matrices**; `UsdSkelBindingAPI`.
- **Animation:** glTF skeletal animation → `UsdSkelAnimation` (joint TRS clips;
  morph-weight animation is not yet authored).
- **Front-direction bake:** VRM 0.x −Z front baked into the data (mesh/normals/
  blendshapes/skeleton/clips), so every avatar shares one canonical +Z rest pose and a
  single shared clip library drives all of them. Provenance kept in `customData`
  (`vrm:sourceFrontAxis`, `vrm:frontAxisNormalized`). *(policy §7)*
- **Typed schemas — every `/Asset/rig/*` control prim is typed** (compiled, co-located
  in `usdVrm`; one `plugInfo` carries the `SdfFileFormat` + schema `Types`):
  `VrmHumanoidAPI`, `VrmExpressionAPI`, `VrmLookAtAPI`, `VrmSpringBoneAPI`,
  `VrmColliderAPI`, `VrmConstraintAPI`. Source in `schema/schema.usda`, regenerated by
  `tools/generate_schema.py`. Raw VRM blocks still preserved in `customData` as the
  lossless fallback. *(policy §5)*
- **Lossless preservation + diagnostics:** `vrm:meta`/`specVersion`/`vrm:rawExtension`
  and import warnings (`/Asset.customData.vrm:warnings`).
- **Tests:** synthetic fixture matrix + `usdvrm_python_smoke` stage-contract test +
  `usdvrm_transform_util` unit test, green via plain CMake and `ost plugin test`.

### Reliability tooling (satisfies much of P0/P3, policy §11–§12)

- **Standalone stage validator** — `tools/validate_vrm.py`: a validation contract over
  the imported stage (default prim, `/Asset`, skinned-mesh→skeleton binding,
  parent-before-child joint order, `JOINTS_0` range, material/texture/humanoid/
  expression/spring targets, raw↔typed source mapping). Runs over an already-imported
  stage, never re-reads the `.vrm` — separate from the importer (policy §10, §12). Emits
  typed diagnostics; non-zero exit on ERROR/FATAL. Tested in `test_usdvrm_validate.py`.
- **Error/warning code taxonomy** (Fatal/Error/Warning/Info) replacing free-text
  warnings. Stable `VRMxxx` codes: `src/model/VrmDiagnostics.h`, canonical severity
  catalog in `tools/vrm_diagnostics.py`, reference table in
  [`../plugins/usdVrm/docs/DIAGNOSTICS.md`](../plugins/usdVrm/docs/DIAGNOSTICS.md).
- **Compatibility report** (`tools/vrm_report.py`, human-readable + machine JSON:
  summary / diagnostics / asset-inventory / compatibility). Merges coded import-time
  diagnostics with the validator's findings, plus a feature-presence matrix. Verified on
  both corpus avatars.

### Asset portability (all of the policy §8 goal)

*Hand an import to another machine and textures still resolve; no temp-dir dependency.*

- `ArPackageResolver` for `.vrm` containers: embedded PNG/JPEG textures are authored as
  `avatar.vrm[images/<hash>.<ext>]` and served directly from the source container,
  removing the OS-temp extraction dependency for normal imports.
- Texture **export bundle** + portable asset-path policy + a `package` command
  (`tools/package_vrm.py`: package-local `textures/` + relative USD asset paths).
- **Asset inventory** output (`package_report.json`).

### Semantic-contract stabilization (schema contract v1)

- `Vrm*API` **schema specification** doc + **versioning** policy
  ([`../plugins/usdVrm/docs/SCHEMA_CONTRACT.md`](../plugins/usdVrm/docs/SCHEMA_CONTRACT.md);
  stages/report carry `schemaContractVersion = 1`).
- Raw-extension ↔ typed-API **correspondence table**; schema-compatibility tests;
  public validator-rule docs.
- **Two policy divergences reconciled** (contract v1 freezes the shipped shapes):
  spring bones stay under `/Asset/rig/SecondaryMotion/*` (not policy §4's
  `/Asset/physics/*`); humanoid stays per-bone `vrm:humanBones:<bone>` token attributes
  (not policy §5's `token[] vrm:humanBoneNames` + relationship sketch).

contract → Windows runtime CI → OpenExec runtime → MToon realization → round-trip.

### P3 — Windows CI & runtime verification  (medium)
*Goal: Windows build + open continuously verified; textured real models resolve; schema
registration succeeds.* (policy §14, §17-P3)

- [x] **OS axis:** `ost-source-ci.yml` PR lane runs `windows-2022`, `macos-15`,

### P4 — OpenExec runtime plugin (`execVrm`)  (medium)
*Goal: build a runtime graph from a static USD stage; drive the humanoid from Mocopi or
animation input; evaluate look-at / expression / spring bone as independent nodes;
swap the runtime without changing the importer.* (policy §10, §17-P4)

- [ ] `execVrm` module (separate plugin/module, policy §18): schema-discovery node,
      humanoid-retarget node, expression / look-at evaluators, spring-bone solver,
      constraint evaluator, graph ordering, **Mocopi adapter** prototype.

### P5 — MToon realization  (medium)
*Goal: source parameters preserved; a portable fallback exists; at least one renderer
reproduces the main MToon look; an image regression test exists.* (policy §9, §17-P5)

- [ ] `VrmMToonAPI` + source-semantics contract; MaterialX / custom-render-context
      approximation; renderer adapter; outline; conformance images; transparent-sorting
      behavior. Today MToon is `UsdPreviewSurface` fallback + `vrm:mtoon:raw` only.

### P6 — Round-trip / exporter research  (low)
*Goal: round-trippable fields explicit; export loss reportable; a limited VRM 1.0 export
path verifiable.* (policy §17-P6)

- [ ] USD→canonical reverse mapping; source-fidelity + loss report; detection of
      unsupported USD edits; exporter prototype. (A full VRM exporter remains a non-goal;
      this is research toward feasibility.)

## Acceptance criteria (production-oriented importer — policy §16)

Tracked here so "done" is unambiguous:

1. VRM 0.x/1.0 corpus continuously verified in CI. *(P3 / corpus expansion)*
2. Skinned mesh / skeleton / humanoid / expression / spring-bone inspectable on stage.
   ✅ (typed schemas + stage-contract tests)
3. Textures exportable as a portable package. ✅
4. MToon fallback vs fidelity responsibilities clear. *(P5)*
5. Import warnings / fidelity loss retrievable as a report. ✅ (coded taxonomy +
   `validate_vrm.py` + `vrm_report.py`)
6. Schema contract documented + versioned. ✅ (`SCHEMA_CONTRACT.md`, contract v1)
7. External pipelines (OpenExec) can run the importer as a structured task. *(P4)*

---

## Non-goals (policy §15, §19)

Out of scope for this plugin (handle via schema/adapter/OpenExec task/other plugin):
full VRM runtime physics execution (→ `execVrm`), pixel-perfect MToon across all
renderers, auto-repair of arbitrary broken glTF, a full VRM exporter (P6 is research
only), DCC-specific UI.

## Cross-cutting / tooling

- **Corpus expansion:** redistributable avatars across the policy axes (VRM 0.x, 1.0,
  VRoid, MToon-heavy, multi-skin, animated, non-ASCII names, texture compression,
  spring/collider complexity). Selection policy + provenance manifest in
  `../plugins/usdVrm/tests/corpus/CORPUS.md`. Vendored: **Seed-san** (VirtualCast) and
  **VRM1 Constraint Twist** (pixiv), both VRM 1.0 with `allowRedistribution: true`.
  Remaining axes: VRM 0.x, VRoid, animation clips, KTX2, multi-skin.
- **`ost` dogfooding** continues in `docs/report/ost` (see the numbered reports; `/docs`
  is git-ignored except this roadmap, so those reports stay local). Carried `ost` asks
  live in the latest report's forward-note.
- **Multi-plugin session dogfooding:** the eventual `execVrm` bundle (P4) exercises
  `ost plugin run/view --with` + `requires.runtime_libs`. The repo root already globs
  `plugins/*`, so a second bundle drops in without edits.
- **Coordinate/precision unit tests** locking `util/TransformUtil` conversions. ✅
